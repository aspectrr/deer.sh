## The Series

This is the first post in a 10-part series about building a hypervisor from scratch in Rust on Apple Silicon. By the end, we will boot a Linux kernel and get an interactive busybox shell running inside a virtual machine that we wrote every line of.

No third-party hypervisor crates. No QEMU. No KVM. Just raw FFI bindings to Apple's Hypervisor.framework, safe Rust abstractions on top, and a whole lot of bare-metal ARM64 systems programming. All the code runs on M1/M2/M3 Macs - if you have one, you can follow along and build it yourself.

Before we write any code, we need to understand the three foundational concepts that every hypervisor is built on: exception levels, two-stage address translation, and the VM exit loop. If you already know these, skim ahead. If not, this post will give you the mental model you need for everything that follows.

## What is a Hypervisor?

A hypervisor is software that creates and runs virtual machines. That is the one-sentence definition. The details of _how_ it does that vary enormously depending on architecture, but the job description is always the same: make one physical computer look like multiple independent computers, each with its own CPU, memory, and devices.

Hypervisors come in two flavors:

**Type 1 (bare metal)** runs directly on the hardware with no host OS underneath. Xen and VMware ESXi are Type 1 hypervisors. They _are_ the OS, in a sense - they boot first and then launch guest operating systems on top of themselves.

**Type 2 (hosted)** runs as a process on top of a conventional operating system. VirtualBox, QEMU with KVM, and Parallels are Type 2 hypervisors. They rely on the host OS for things like scheduling, memory management, and device drivers, but they still use hardware virtualization features to run guest code at near-native speed.

Apple's Hypervisor.framework is firmly Type 2. It is a macOS framework - a shared library that ships with the operating system - that exposes the CPU's hardware virtualization capabilities to userspace processes. You write a normal macOS application, link against Hypervisor.framework, call a handful of C functions, and suddenly you have a hardware-accelerated virtual CPU that can run unmodified guest code at full speed.

The framework handles the privileged operations: configuring the hardware virtualization extensions, managing the EL2 state, and mediating access between your process and the CPU's hypervisor mode. Your job is everything else: allocating guest memory, loading a kernel, handling VM exits, emulating devices. That "everything else" is what we will build in this series.

## ARM Exception Levels

ARM processors organize privilege using **exception levels**, numbered EL0 through EL3. Higher numbers mean more privilege. This is conceptually similar to x86's ring 0 through ring 3, but with some important differences.

<ExceptionLevelDiagram />

**EL0 - User Applications.** This is where your programs run. A web browser, a text editor, a database server - all EL0. Code at this level cannot access hardware directly, cannot modify page tables, cannot disable interrupts. If it tries, the CPU generates an exception and drops into EL1.

**EL1 - Operating System Kernel.** Linux, macOS, Windows - the kernel runs here. EL1 code can configure the MMU, manage page tables, handle interrupts, and access most system registers. When your program makes a syscall, the CPU transitions from EL0 to EL1 via the `SVC` (supervisor call) instruction.

**EL2 - Hypervisor.** This is where virtualization happens. EL2 has its own set of system registers that control the virtual machine's view of the hardware. It can intercept any operation the guest kernel performs at EL1 - memory accesses, system register reads, interrupt handling. The `HVC` (hypervisor call) instruction transitions from EL1 to EL2.

**EL3 - Secure Monitor.** The most privileged level. EL3 manages the boundary between the secure world (TrustZone) and the normal world. On Apple Silicon, EL3 is Apple's proprietary firmware - you cannot touch it and you do not need to. The `SMC` (secure monitor call) instruction transitions from EL2 to EL3.

The transition mechanism between levels is worth understanding. When the CPU moves from a lower exception level to a higher one, it saves the return address and processor state automatically. The `ERET` (exception return) instruction reverses this - it restores the saved state and drops back to the lower exception level. This is exactly how VM entry and VM exit work under the hood: the hypervisor at EL2 uses `ERET` to enter the guest at EL1, and the guest's traps bring execution back to EL2.

On Apple Silicon specifically, the picture looks like this:

- EL3: Apple firmware (we never interact with this)
- EL2: Managed by Hypervisor.framework's kernel extension
- EL1: The guest OS kernel inside our VM
- EL0: User processes inside the guest

When we call `hv_vcpu_run()` in our userspace process, Hypervisor.framework talks to its kernel extension, which configures the EL2 state and performs an `ERET` to enter the guest at EL1. The guest runs until something causes a trap back to EL2, at which point control returns through the kernel extension back to our userspace process. All of this happens transparently - we call one function and get back a struct describing why the guest stopped.

## Two-Stage Address Translation

On a normal (non-virtualized) machine, the CPU translates virtual addresses to physical addresses using a single set of page tables. Process A has its own page tables, process B has its own, the kernel switches between them on context switch. One stage of translation: VA to PA.

Virtualization adds a problem. The guest OS thinks it is managing physical memory. It builds page tables that map guest virtual addresses to what it believes are physical addresses. But those "physical" addresses are not real - they are just numbers the hypervisor made up. The hypervisor needs a second layer of translation to map those fake physical addresses to actual host physical addresses.

ARM calls these **Intermediate Physical Addresses (IPAs)**. The two stages are:

1. **Stage 1 (guest-controlled):** Guest VA to IPA. The guest kernel manages these page tables. It thinks IPAs are real physical addresses.
2. **Stage 2 (hypervisor-controlled):** IPA to PA. The hypervisor manages these page tables. They define which real physical memory the guest can access.

<AddressTranslation />

Both stages are walked by hardware. When the guest accesses memory, the CPU walks the stage-1 tables (using the guest's TTBR0_EL1 or TTBR1_EL1), gets an IPA, then walks the stage-2 tables (using the hypervisor's VTTBR_EL2) to get the real PA. If the TLB has a cached translation, all of this is skipped.

This is directly relevant to our hypervisor. When we call `hv_vm_map(host_ptr, ipa, size, flags)`, Hypervisor.framework populates the stage-2 page tables on our behalf. We are saying: "guest physical address `ipa` through `ipa + size` should map to the host memory at `host_ptr`." The guest kernel will build its own stage-1 tables on top of this, but it can never access host memory outside what we have mapped.

The performance cost is real. A TLB miss on bare metal walks one set of page tables - 4 memory accesses for a 4-level table. A TLB miss with two-stage translation walks both sets, nested. Each step of the stage-1 walk itself requires a stage-2 walk to translate the page table entry's IPA to a PA. For 4-level tables on both stages, that is up to 4 x 4 = 16 stage-2 lookups plus 4 stage-1 reads plus the final data access: 24 memory accesses total versus 4 on bare metal. TLB hit rates are generally high enough that this does not matter in practice, but it is worth knowing where the overhead comes from.

The more interesting consequence for us is what happens when the guest accesses an IPA that has _no_ stage-2 mapping. The hardware generates a **data abort** exception, which causes a VM exit. The hypervisor gets control, reads the faulting IPA from the exit information, and decides what to do. If the IPA is in a range we designated for a memory-mapped device, we emulate the device access. If it is truly unmapped, we inject a fault back into the guest.

This is the foundation of MMIO (Memory-Mapped I/O) emulation. We deliberately leave certain IPA ranges unmapped. When the guest reads or writes to those addresses - because it thinks there is a UART or an interrupt controller there - the hardware traps to us and we emulate the device in software. We will use this mechanism extensively when we build our virtual UART and GIC (interrupt controller) later in the series.

## The VM Exit/Entry Loop

The core of every virtual machine monitor is a loop: enter the guest, wait for it to stop, handle the reason it stopped, repeat. This is the **VM exit/entry loop**, and every hypervisor ever written - from Xen to KVM to our toy - has one.

<VmExitCycle />

The cycle has four phases:

### VM Entry

The hypervisor restores the guest's CPU state - general-purpose registers, system registers, program counter, stack pointer, processor flags - and transitions into the guest context. On ARM, this ultimately involves an `ERET` instruction that drops from EL2 to EL1 and starts executing guest code.

In our hypervisor, this happens when we call `hv_vcpu_run()`. Before the call, we may have modified guest registers (to inject return values, for example). The framework handles the actual context switch.

### Guest Execution

The guest runs natively on real hardware at full speed. There is no interpretation, no binary translation, no emulation layer. The CPU is executing guest instructions directly in EL1. The only thing different from bare-metal execution is the stage-2 page tables and a few trapped system registers.

This is why hardware virtualization is fast. The guest is not "simulated" - it runs on the actual CPU, at the actual clock speed, with the actual caches. The overhead comes only from VM exits.

### VM Exit

Something happens that the guest cannot handle on its own, or that the hypervisor has configured to trap. The CPU saves the guest's state and transitions back to EL2. Common exit reasons include:

- **HVC instruction:** The guest deliberately called into the hypervisor (similar to a syscall but one level up).
- **MMIO access:** The guest accessed an IPA with no stage-2 mapping. The hardware generated a data abort.
- **System register access:** The guest tried to read or write a system register that the hypervisor has trapped (like the timer control registers).
- **Timer expiration:** The virtual timer fired, and the hypervisor configured it to cause an exit.
- **Interrupt:** An external interrupt arrived that needs hypervisor attention.
- **WFI/WFE:** The guest executed a wait-for-interrupt or wait-for-event instruction, indicating it has nothing to do.

In our hypervisor, `hv_vcpu_run()` returns when a VM exit occurs. We read the `hv_vcpu_exit_t` struct to find out why. The `reason` field tells us the category (exception, vtimer, canceled), and for exceptions, the `syndrome` field contains the ARM **Exception Syndrome Register (ESR)** value, which encodes the exact cause - the exception class (EC), the instruction length (IL), and an instruction-specific syndrome (ISS).

### VMM Handling

This is where we do the actual work. Based on the exit reason, we:

- **HVC:** Read the guest's X0-X7 to determine what it wants, perform the action, write the result back to X0, advance PC past the HVC instruction.
- **MMIO:** Decode the faulting instruction from the syndrome, determine if it was a read or write, which register, what size. Route to the appropriate virtual device. If it was a read, write the result into the destination register. Advance PC.
- **System register:** The syndrome tells us which register and whether it was a read (MRS) or write (MSR). Emulate the access and advance PC.
- **Timer:** Set up the timer for the next tick, possibly inject a virtual interrupt into the guest.
- **WFI:** Either block the thread until an interrupt is pending, or just continue if one is already pending.

After handling, we loop back to VM entry. The guest continues from where it left off, unaware that it was ever interrupted (unless we deliberately modified its state).

The critical insight is that VM exits are expensive. Each one involves saving and restoring the full CPU context, switching between EL2 and EL1, and running our handling logic. A single VM exit can cost thousands of CPU cycles. The goal of a good VMM is to minimize exits. This means: map as much memory as possible up front (avoiding unnecessary data aborts), trap only the system registers you must emulate, and batch interrupt injection when you can.

### What This Looks Like in Code

In pseudocode, our entire hypervisor will look roughly like this:

```rust
let vm = Vm::new()?;
let memory = GuestMemory::new(GUEST_RAM_SIZE)?;
vm.map(&memory, GUEST_BASE_IPA)?;

load_kernel(&memory, kernel_path)?;

let vcpu = Vcpu::new()?;
vcpu.set_reg(PC, KERNEL_ENTRY)?;
vcpu.set_reg(CPSR, 0x3c4)?; // EL1h, DAIF masked

loop {
    let exit = vcpu.run()?;
    match exit {
        Exit::Hvc { .. } => handle_hvc(&vcpu)?,
        Exit::DataAbort { ipa, .. } => handle_mmio(&vcpu, ipa)?,
        Exit::SysReg { reg, .. } => handle_sysreg(&vcpu, reg)?,
        Exit::VTimer => handle_timer(&vcpu)?,
        Exit::Unknown => break,
    }
}
```

That is the skeleton of the entire project. Every post in this series fills in one piece: FFI bindings, safe wrappers, memory management, exit handling, device emulation, kernel loading. But the loop is always the same.
