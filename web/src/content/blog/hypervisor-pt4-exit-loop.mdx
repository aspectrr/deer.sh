## The Run Loop

Every virtual machine monitor is, at its core, a loop. You start the guest vCPU. It runs native code at full speed on the physical CPU. At some point the guest does something that it cannot handle on its own - a privileged instruction, an access to unmapped memory, a request for hypervisor services - and the CPU traps back to the host. The VMM inspects what happened, emulates the expected behavior, and resumes the guest. Over and over, millions of times per second.

On Apple Silicon, this loop centers on a single function: `hv_vcpu_run()`. When you call it, the current thread enters EL1 guest mode and starts executing whatever the guest's program counter points at. The call blocks until the guest triggers a VM exit. When it returns, the exit reason is waiting for you in the vCPU's exit information structure.

The simplicity is deceptive. That exit reason field is where all the complexity lives. The guest might have executed a hypercall instruction. It might have tried to access a system register that the hypervisor has trapped. It might have touched a memory address that doesn't have a stage-2 mapping. It might have hit a breakpoint. Each of these requires different handling, and getting any of them wrong means a crashed or hung VM.

Let's build the exit loop from scratch.

## Decoding the Syndrome

When `hv_vcpu_run()` returns and the exit reason is `HV_EXIT_REASON_EXCEPTION`, the real information lives in the syndrome field. This is the value of ESR_EL2 - the Exception Syndrome Register at EL2 - captured at the moment the guest trapped. ARM specifies a precise encoding for this register, and understanding it is essential for writing any VMM on the architecture.

The syndrome is a 64-bit value (though only the lower 32 bits are architecturally defined). It breaks down into three fields:

**Exception Class (EC) - bits [31:26]**: A 6-bit field that identifies what kind of exception occurred. This is your primary dispatch key. Was it a data abort? An HVC instruction? A system register access? The EC tells you.

**Instruction Length (IL) - bit [25]**: Indicates whether the trapped instruction was 16 bits (Thumb/compact) or 32 bits (standard AArch64). In AArch64 state this is always 1 (32-bit), but you will see it matter if you ever support AArch32 guests.

**Instruction Specific Syndrome (ISS) - bits [24:0]**: A 25-bit field whose interpretation depends entirely on the EC. For a data abort, the ISS tells you the access size, target register, and whether it was a read or write. For an HVC, it contains the immediate value from the instruction. Each exception class defines its own ISS format.

Here is the syndrome decoding module - small, but you will reach for these helpers constantly:

```rust
// src/arch/syndrome.rs

/// Extract the Exception Class (EC) from an ESR_EL2 value.
/// EC is bits [31:26] - identifies the type of exception.
pub fn ec(syndrome: u64) -> u32 {
    ((syndrome >> 26) & 0x3f) as u32
}

/// Extract the Instruction Specific Syndrome (ISS) from an ESR_EL2 value.
/// ISS is bits [24:0] - interpretation depends on EC.
pub fn iss(syndrome: u64) -> u32 {
    (syndrome & 0x1ffffff) as u32
}

/// Extract the Instruction Length (IL) bit from an ESR_EL2 value.
/// Returns true if the trapped instruction was 32 bits (always true in AArch64).
pub fn il(syndrome: u64) -> bool {
    (syndrome >> 25) & 1 == 1
}
```

And the EC constants we care about right now:

```rust
/// WFI or WFE instruction trapped
pub const EC_WFI: u32 = 0x01;

/// HVC instruction executed from AArch64
pub const EC_HVC64: u32 = 0x16;

/// SMC instruction executed from AArch64
pub const EC_SMC64: u32 = 0x17;

/// MSR, MRS, or system register access trapped
pub const EC_SYSREG: u32 = 0x18;

/// Instruction abort from a lower Exception Level
pub const EC_IABT_LOWER: u32 = 0x20;

/// Data abort from a lower Exception Level (MMIO lives here)
pub const EC_DABT_LOWER: u32 = 0x24;

/// BRK instruction from AArch64
pub const EC_BRK64: u32 = 0x3c;
```

That list is not exhaustive - the ARM Architecture Reference Manual defines dozens of EC values - but these are the ones that matter for a minimal hypervisor. The data abort class (0x24) is especially important because it is how MMIO works, but that deserves its own post. For now, we will handle the simpler traps.

## Building the Run Loop

With the syndrome decoding in place, the run loop itself is straightforward. The structure follows the universal VMM pattern: run, decode, dispatch, repeat.

```rust
use crate::arch::syndrome::*;
use crate::vcpu::Vcpu;
use applevisor::*;

pub fn run_vm(vcpu: &mut Vcpu) -> Result<(), Box<dyn std::error::Error>> {
    loop {
        // Enter the guest. This blocks until a VM exit occurs.
        vcpu.run()?;

        let exit = vcpu.exit_info();

        match exit.reason {
            HV_EXIT_REASON_EXCEPTION => {
                let syndrome = exit.exception.syndrome;

                match ec(syndrome) {
                    EC_HVC64 => {
                        handle_hvc(vcpu, iss(syndrome))?;
                        vcpu.advance_pc(4)?;
                    }
                    EC_SMC64 => {
                        handle_smc(vcpu)?;
                        vcpu.advance_pc(4)?;
                    }
                    EC_WFI => {
                        handle_wfi(vcpu)?;
                        vcpu.advance_pc(4)?;
                    }
                    EC_BRK64 => {
                        println!("BRK #{} - halting VM", iss(syndrome));
                        break;
                    }
                    ec_val => {
                        let pc = vcpu.get_reg(HV_REG_PC)?;
                        panic!(
                            "unhandled exception class: EC=0x{ec_val:02x} \
                             at PC=0x{pc:016x}, syndrome=0x{syndrome:08x}"
                        );
                    }
                }
            }
            HV_EXIT_REASON_VTIMER_ACTIVATED => {
                // Guest's virtual timer fired. We will handle this properly
                // when we implement interrupts. For now, just mask it and continue.
                vcpu.set_vtimer_mask(true)?;
            }
            reason => {
                panic!("unknown exit reason: {reason:?}");
            }
        }
    }

    Ok(())
}
```

A few things to note about this structure:

First, the outer match on `exit.reason` separates exception-based exits from other exit types. On Hypervisor.framework, the two main exit reasons are `HV_EXIT_REASON_EXCEPTION` (the guest did something that trapped) and `HV_EXIT_REASON_VTIMER_ACTIVATED` (the guest's virtual timer expired). Other hypervisors have more exit reasons - KVM has dozens - but Apple's framework keeps it simple.

Second, the inner match on the exception class is where all the interesting work happens. Each EC value gets its own handler. The pattern of "handle, then advance PC" repeats for most trap types, with BRK being the notable exception (we will get to why shortly).

Third, the panic on unhandled ECs is intentional for development. In a production hypervisor you would want graceful error handling, but during development a panic with the full syndrome and PC value gives you everything you need to diagnose what the guest did wrong.

## PC Advancement

This is a detail that trips up every first-time VMM author at least once: after handling most traps, you must manually advance the guest's program counter past the trapped instruction. The CPU does not do this for you.

In AArch64 state, all instructions are 32 bits wide (4 bytes). So advancing the PC means:

```rust
impl Vcpu {
    pub fn advance_pc(&mut self, bytes: u64) -> Result<(), Box<dyn std::error::Error>> {
        let pc = self.get_reg(HV_REG_PC)?;
        self.set_reg(HV_REG_PC, pc + bytes)?;
        Ok(())
    }
}
```

Why doesn't the CPU advance the PC automatically? Because the hypervisor needs to decide what happens next. Consider a data abort: the guest tried to access an unmapped address. Maybe the VMM wants to emulate the access and let the guest continue past the instruction. Maybe the VMM wants to inject a fault back into the guest, in which case the guest's exception handler will change the PC. Maybe the VMM wants to map the page and let the guest retry the same instruction. The CPU can't know which behavior the hypervisor wants, so it leaves the PC pointing at the trapped instruction and lets the VMM decide.

For synchronous traps like HVC, SMC, and WFI, we always advance. The guest issued the instruction deliberately, we handled it, and the guest should continue with the next instruction.

For BRK, we do **not** advance the PC. BRK is a debug breakpoint. If we advanced past it, the guest would skip the instruction following the breakpoint, which defeats the purpose of having a breakpoint in the first place. In our case we use BRK as a halt mechanism - the loop breaks and the VM stops - so the PC value doesn't matter afterward. But if you were building a debugger, you would want the PC to stay on the BRK so the user can inspect state and then choose to step or continue.

For data and instruction aborts, it depends. If you are emulating the access (MMIO), you advance. If you are mapping a page and want the guest to retry, you don't. We will deal with this in the MMIO post.

## The Guest Test Program

To exercise the exit loop, we need a guest program that deliberately triggers multiple different trap types. Here is an ARM64 program encoded as raw instruction words:

```rust
let code: &[u32] = &[
    0xD2800020, // mov x0, #1       -- set x0 = 1
    0xD2800041, // mov x1, #2       -- set x1 = 2
    0xD4000002, // hvc #0           -- hypercall: trap to hypervisor
    0xD2800062, // mov x2, #3       -- set x2 = 3
    0xD4000003, // smc #0           -- secure monitor call: trap
    0xD503207F, // wfi              -- wait for interrupt: trap
    0xD2800083, // mov x3, #4       -- set x3 = 4
    0xD4200000, // brk #0           -- breakpoint: halt the VM
];
```

This program interleaves register assignments with trapping instructions. The execution flow should be:

1. `mov x0, #1` - runs natively, no exit
2. `mov x1, #2` - runs natively, no exit
3. `hvc #0` - VM exit, EC = 0x16 (HVC64). Handler runs. PC advances.
4. `mov x2, #3` - runs natively after re-entry
5. `smc #0` - VM exit, EC = 0x17 (SMC64). Handler runs. PC advances.
6. `wfi` - VM exit, EC = 0x01 (WFI). Handler runs. PC advances.
7. `mov x3, #4` - runs natively after re-entry
8. `brk #0` - VM exit, EC = 0x3c (BRK64). Loop breaks.

If the run loop is correct, we will see exactly four VM exits in order: HVC, SMC, WFI, BRK. And after execution, the registers will hold x0=1, x1=2, x2=3, x3=4 - proving that execution continued correctly after each trap.

## Handling Each Trap Type

Let's look at each handler in detail.

### HVC - Hypercalls

The HVC (Hypervisor Call) instruction is how a guest deliberately requests service from the hypervisor. It is the virtual equivalent of a system call. The ISS field of the syndrome contains the 16-bit immediate value from the instruction - `hvc #0` gives ISS=0, `hvc #1` gives ISS=1, and so on.

The ARM calling convention for hypercalls uses registers x0 through x3 for arguments and x0 for the return value, mirroring the standard function call convention. This is defined by the ARM SMC Calling Convention (SMCCC) specification.

```rust
fn handle_hvc(vcpu: &mut Vcpu, imm: u32) -> Result<(), Box<dyn std::error::Error>> {
    let x0 = vcpu.get_reg(HV_REG_X0)?;
    let x1 = vcpu.get_reg(HV_REG_X1)?;

    println!("HVC #{imm}: x0={x0:#x}, x1={x1:#x}");

    // Example: hypercall #0 could be "print register"
    // hypercall #1 could be "get VMM version"
    match imm {
        0 => {
            // No-op for now. In a real hypervisor, this is where you
            // would dispatch to paravirtualized device drivers, memory
            // sharing, or performance counters.
        }
        _ => {
            // Unknown hypercall. Return -1 (not supported).
            vcpu.set_reg(HV_REG_X0, u64::MAX)?;
        }
    }

    Ok(())
}
```

In a real hypervisor, hypercalls are the foundation of paravirtualization. Linux's KVM uses them for things like clock synchronization, TLB flush requests, and paravirtualized spinlocks. Xen uses them extensively for memory management and event channels. For our minimal hypervisor, we just log and continue, but this is where you would hook in your own guest-host communication protocol.

### SMC - Secure Monitor Calls

The SMC instruction is meant to invoke ARM TrustZone's secure monitor. In a VM, there is no secure monitor - the hypervisor is the highest privilege level the guest can reach. The standard behavior is to return "not supported" and move on.

```rust
fn handle_smc(vcpu: &mut Vcpu) -> Result<(), Box<dyn std::error::Error>> {
    println!("SMC trapped - returning NOT_SUPPORTED");

    // SMCCC defines -1 (0xFFFFFFFF for 32-bit, 0xFFFFFFFFFFFFFFFF for 64-bit)
    // as the "not supported" return code.
    vcpu.set_reg(HV_REG_X0, u64::MAX)?;

    Ok(())
}
```

This is important for Linux boot. The kernel probes for PSCI (Power State Coordination Interface) support by issuing SMC calls early in boot. If the hypervisor doesn't handle SMC traps at all, the VM crashes before the kernel even prints its first message. By returning -1, we tell the kernel "PSCI over SMC is not available" and it falls back to other mechanisms or continues without it.

Some hypervisors do implement PSCI over SMC - this is how you would support multi-core VMs, since the guest uses PSCI `CPU_ON` calls to bring up secondary CPUs. But for a single-core VM, returning "not supported" is sufficient.

### WFI - Wait For Interrupt

WFI (Wait For Interrupt) is an idle hint. The guest is telling the CPU "I have nothing to do until an interrupt arrives." On bare metal, this puts the core into a low-power state. In a VM, it traps to the hypervisor.

```rust
fn handle_wfi(vcpu: &mut Vcpu) -> Result<(), Box<dyn std::error::Error>> {
    // The guest is idle. In a production hypervisor, you would:
    // 1. Check if any interrupts are pending for this vCPU
    // 2. If yes, return immediately (the guest should handle the interrupt)
    // 3. If no, yield the host thread or sleep until an interrupt arrives
    //
    // For now, we just continue immediately.
    let _ = vcpu;
    Ok(())
}
```

The simple approach - just advancing past the WFI and continuing - works but wastes CPU. The guest will immediately execute another WFI in its idle loop, trapping again, spinning the host CPU at 100%. A production hypervisor would block the host thread here until a virtual interrupt is ready to be delivered. We will revisit this when we implement the interrupt controller.

### BRK - Breakpoint

BRK is a software breakpoint instruction. The ISS contains the 16-bit immediate value. We use it as our halt mechanism:

```rust
// In the run loop:
EC_BRK64 => {
    let imm = iss(syndrome);
    let pc = vcpu.get_reg(HV_REG_PC)?;
    println!("BRK #{imm} at PC=0x{pc:016x} - halting VM");
    break;
}
```

No PC advancement. No register manipulation. Just break out of the loop. In a debugger scenario, you would want to inspect registers, memory, and let the user decide what to do next. In GDB remote stub implementations, BRK traps are how software breakpoints work - the debugger patches an instruction with `BRK #imm`, the VM exits, and the debugger reports the stop to the user.

## Putting It All Together

The complete flow for our test program looks like this:

1. Load the instruction bytes into guest memory at the entry point
2. Set the vCPU's PC to the entry point
3. Enter the run loop
4. See four exits in sequence: HVC, SMC, WFI, BRK
5. After the loop exits, read back x0 through x3

```rust
fn main() -> Result<(), Box<dyn std::error::Error>> {
    let vm = Vm::new()?;
    let mut vcpu = vm.create_vcpu()?;

    // Map guest memory and load the test program
    let mem = vm.map_memory(GUEST_MEM_BASE, GUEST_MEM_SIZE)?;
    let code: &[u32] = &[
        0xD2800020, // mov x0, #1
        0xD2800041, // mov x1, #2
        0xD4000002, // hvc #0
        0xD2800062, // mov x2, #3
        0xD4000003, // smc #0
        0xD503207F, // wfi
        0xD2800083, // mov x3, #4
        0xD4200000, // brk #0
    ];
    mem.write_u32_slice(0, code);

    // Set entry point
    vcpu.set_reg(HV_REG_PC, GUEST_MEM_BASE)?;

    // Run the VM
    run_vm(&mut vcpu)?;

    // Verify final state
    assert_eq!(vcpu.get_reg(HV_REG_X0)?, 1);
    assert_eq!(vcpu.get_reg(HV_REG_X1)?, 2);
    assert_eq!(vcpu.get_reg(HV_REG_X2)?, 3);
    assert_eq!(vcpu.get_reg(HV_REG_X3)?, 4);
    println!("All register values correct.");

    Ok(())
}
```

## Testing

The syndrome decoding functions are pure bit manipulation, which makes them ideal for unit testing:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_ec_extraction() {
        // EC = 0x16 (HVC64), IL = 1, ISS = 0
        let syndrome = 0x16 << 26 | 1 << 25;
        assert_eq!(ec(syndrome as u64), EC_HVC64);
        assert!(il(syndrome as u64));
        assert_eq!(iss(syndrome as u64), 0);
    }

    #[test]
    fn test_ec_brk() {
        // EC = 0x3c (BRK64), ISS = 42
        let syndrome = (0x3c << 26) | (1 << 25) | 42;
        assert_eq!(ec(syndrome as u64), EC_BRK64);
        assert_eq!(iss(syndrome as u64), 42);
    }

    #[test]
    fn test_ec_data_abort() {
        // EC = 0x24 (DABT_LOWER)
        let syndrome = 0x24 << 26;
        assert_eq!(ec(syndrome as u64), EC_DABT_LOWER);
    }

    #[test]
    fn test_iss_max_value() {
        // ISS is 25 bits, max value = 0x1FFFFFF
        let syndrome: u64 = 0x1ffffff;
        assert_eq!(iss(syndrome), 0x1ffffff);
    }
}
```

The integration test runs the full multi-trap guest program and verifies both the exit sequence and the final register state. If any exit is mishandled - if PC advancement is wrong, if a handler clobbers a register, if the exit dispatch is incorrect - the assertions will catch it.

## What We Have Now

After this post, the hypervisor has a complete exit loop that can:

- Decode ARM64 exception syndromes into exception class, ISS, and instruction length
- Dispatch to handlers based on the exception class
- Handle HVC (hypercalls), SMC (secure monitor calls), WFI (idle), and BRK (breakpoint/halt)
- Correctly advance the program counter after each handled trap
- Halt cleanly on BRK

This is the skeleton that everything else hangs on. Every feature we add from here - MMIO, interrupts, timers, device emulation - is just a new case in the EC match statement. The loop structure doesn't change. What changes is how much work happens inside each handler.

Next up: data aborts. When the guest accesses an address that doesn't have a stage-2 mapping, we get a data abort exit with EC=0x24. By not mapping certain address ranges and then handling the resulting aborts, we can emulate memory-mapped hardware that doesn't physically exist. That is MMIO, and it is how we will build every virtual device from the serial console to the interrupt controller.
