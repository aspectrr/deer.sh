## Why UART?

Up to this point, every interaction with our VM has been indirect. We load instructions, run the guest, read registers afterward, and check if the values are correct. The VM has no voice - it cannot print a message, display an error, or tell us what it is doing. It runs in silence and we inspect the aftermath.

The UART changes that. A Universal Asynchronous Receiver/Transmitter is the simplest possible I/O device: you write a byte to it and that byte appears on a serial terminal. Every ARM system has one. The Linux kernel uses it for console output from the earliest moments of boot, before the framebuffer is initialized, before the network stack is up, before any storage drivers are loaded. The kernel's first `printk` messages go to the UART.

The PL011 is ARM's standard UART design. It is what QEMU's "virt" machine type uses, which means every ARM64 Linux kernel already has a driver for it. When we eventually boot Linux in this hypervisor, the kernel will expect a PL011 at a specific address and will start writing boot messages to it. If our PL011 emulation is correct, those messages will flow through our MMIO handler and appear in the host terminal. A silent VM becomes a talking VM.

But before Linux, we need to verify the emulation works with a bare-metal guest program that writes characters to the UART one at a time. This is what we will build in this post.

## PL011 Register Map

The PL011 specification (ARM DDI 0183) defines over a dozen registers. A full-fidelity emulation would implement all of them: baud rate divisors, line control, DMA control, interrupt masks, modem control. We don't need any of that. For basic serial output - which is all we need to see boot messages - two registers are sufficient.

The base address is `0x0900_0000`. This is the address that QEMU's virt platform uses, and it is what the standard ARM64 device tree specifies. Using the same address means we can reuse existing device trees without modification.

### DR - Data Register (offset 0x000)

The data register serves dual purpose. Writing a byte to it transmits that byte on the serial line. Reading from it receives a byte (if one is available in the RX FIFO). For TX-only emulation, we only care about writes. When the guest writes to offset 0x000, we take the low 8 bits and output them as a character.

The real PL011 data register includes error flags in bits [11:8] (framing error, parity error, break error, overrun error) when reading. We don't need to emulate these - our virtual serial line doesn't have electrical errors.

### FR - Flags Register (offset 0x018)

The flags register reports the status of the TX and RX FIFOs:

- Bit 7: TXFE - TX FIFO empty (1 = empty)
- Bit 6: RXFF - RX FIFO full (1 = full)
- Bit 5: TXFF - TX FIFO full (1 = full)
- Bit 4: RXFE - RX FIFO empty (1 = empty)
- Bit 3: BUSY - UART busy transmitting (1 = busy)

For our purposes, we always return 0 for this register, which means: TX FIFO is not full (you can always write), RX FIFO is empty (nothing to read), UART is not busy. This is a simplification - a real UART's TX FIFO can fill up if you write faster than the baud rate can transmit. But our virtual UART has infinite bandwidth; bytes are consumed instantly by the host's `print!` macro.

### Registers We Ignore

The PL011 has many other registers that guests may probe or configure:

| Offset | Name  | Purpose                                       |
| ------ | ----- | --------------------------------------------- |
| 0x024  | IBRD  | Integer Baud Rate Divisor                     |
| 0x028  | FBRD  | Fractional Baud Rate Divisor                  |
| 0x02C  | LCR_H | Line Control (word length, parity, stop bits) |
| 0x030  | CR    | Control Register (enable TX, RX, UART)        |
| 0x034  | IFLS  | Interrupt FIFO Level Select                   |
| 0x038  | IMSC  | Interrupt Mask                                |
| 0x03C  | RIS   | Raw Interrupt Status                          |
| 0x040  | MIS   | Masked Interrupt Status                       |
| 0x044  | ICR   | Interrupt Clear                               |

Linux's PL011 driver writes to IBRD, FBRD, LCR_H, and CR during initialization to configure the baud rate and enable the UART. We accept these writes and silently discard them. The UART is always "on" and always at "infinite baud rate" regardless of what the guest configures. Reads to unimplemented registers return 0.

This approach - implement only what you need, stub everything else - is the standard strategy for device emulation. You can always add register-accurate emulation later if a guest depends on it. Start with the minimum that produces correct output.

## Implementing PL011

Here is the complete implementation:

```rust
// src/devices/pl011.rs

use crate::devices::MmioDevice;

/// PL011 UART register offsets
const DR: u64 = 0x000;    // Data Register
const FR: u64 = 0x018;    // Flags Register

/// Minimal PL011 UART emulation.
///
/// Supports TX only. Bytes written to the Data Register are passed
/// to a callback function. The Flags Register always reports TX ready.
pub struct Pl011 {
    tx_callback: Box<dyn FnMut(u8)>,
}

impl Pl011 {
    pub fn new(tx_callback: impl FnMut(u8) + 'static) -> Self {
        Pl011 {
            tx_callback: Box::new(tx_callback),
        }
    }
}

impl MmioDevice for Pl011 {
    fn read(&mut self, offset: u64, _size: u32) -> u64 {
        match offset {
            FR => {
                // Flags: TX FIFO not full (bit 5 = 0),
                // RX FIFO empty (bit 4 = 1).
                // Return 0x10 to indicate RXFE=1, everything else clear.
                0x10
            }
            _ => 0,
        }
    }

    fn write(&mut self, offset: u64, _size: u32, value: u64) {
        match offset {
            DR => {
                // Data Register: transmit the low byte.
                let byte = (value & 0xff) as u8;
                (self.tx_callback)(byte);
            }
            _ => {
                // Silently accept writes to other registers.
                // The guest may configure baud rate, line control, etc.
                // We ignore it all.
            }
        }
    }
}
```

A few design choices worth noting:

The `tx_callback` is a boxed closure rather than a fixed output destination. This makes the UART testable - in tests, the callback can append bytes to a Vec instead of printing to stdout. It also means you could route UART output to a file, a network socket, or a pty without changing the device implementation.

The FR register returns `0x10` (RXFE = 1) rather than plain 0. This tells the guest that the receive FIFO is empty, which prevents drivers from spinning in a read loop waiting for input that will never arrive. The TX bits remain 0 (TXFF = 0, meaning "not full"), so the guest can always write.

Writes to unknown offsets are silently accepted. This is defensive programming for device emulation - if you panic or return an error on every unexpected access, you will discover that guests touch registers you didn't expect, in orders you didn't anticipate. Silent acceptance of unknown writes is safer during development. Once you have a working system, you can add logging to see what the guest is actually writing to those registers.

## Registering with the MMIO Bus

Wiring the PL011 into the VM takes three lines:

```rust
use std::io::Write;

let uart = Pl011::new(|byte| {
    let stdout = std::io::stdout();
    let mut handle = stdout.lock();
    handle.write_all(&[byte]).unwrap();
    handle.flush().unwrap();
});

mmio_bus.register(0x0900_0000, 0x1000, Box::new(uart));
```

We map the UART at `0x0900_0000` with a 4KB region (one page). The TX callback writes each byte to stdout and flushes immediately. The flush is important - without it, output would be buffered and you wouldn't see characters appear until the buffer fills or the VM exits. For a serial console, you want every character to appear immediately.

The base address `0x0900_0000` is not arbitrary. This is the address that QEMU's virt machine type uses for its first PL011 UART, and it is the address that standard ARM64 device trees specify. When we boot Linux later, the device tree will tell the kernel "there is a PL011 at 0x09000000" and the kernel will start writing to it. Our MMIO handler will route those accesses to this exact device.

## The Guest Program

Now for the fun part: a bare-metal ARM64 program that writes "Hello from the VM!" to the UART, character by character. Each character requires two instructions: load the byte into a register, then store it to the UART data register address.

```rust
fn build_uart_hello() -> Vec<u32> {
    let mut code = Vec::new();

    // x1 = UART base address (0x09000000)
    // movz x1, #0x0900, lsl #16
    code.push(0xD2B12001);

    let message = b"Hello from the VM!\n";

    for &byte in message {
        // movz x0, #<byte>
        code.push(0xD2800000 | ((byte as u32) << 5));
        // str w0, [x1]  (32-bit store to UART DR)
        code.push(0xB9000020);
    }

    // hvc #0 - halt
    code.push(0xD4000002);

    code
}
```

Let's break down the instruction encoding:

**`movz x1, #0x0900, lsl #16`** loads the UART base address. The `movz` instruction with a 16-bit left shift places 0x0900 in bits [31:16] of x1, giving us 0x09000000. The encoding is `0xD2B12001`: D2 = movz 64-bit, B = shift 16, 12001 encodes the immediate and destination register.

**`movz x0, #byte`** loads a single ASCII byte into x0. The immediate is in bits [20:5] of the instruction word, so we shift the byte left by 5 and OR it into the base encoding `0xD2800000`. For the letter 'H' (0x48), this produces `0xD2800900`.

**`str w0, [x1]`** stores the 32-bit value in w0 (the low 32 bits of x0) to the address in x1. This triggers a data abort because 0x09000000 is not mapped in the stage-2 tables. The MMIO bus catches the abort, routes it to the PL011, and the PL011 calls our TX callback with the byte. The character appears on the terminal.

For a 20-character message (including the newline), this generates 41 instructions: 1 for the address setup, 2 per character (load + store), and 1 for the halt. Each store instruction causes a VM exit, so we exit and re-enter the guest 20 times. This is comically inefficient compared to real I/O, but it is exactly how every serial console works at the hardware level - one byte per store.

## Capturing Output for Testing

The closure-based TX callback makes testing straightforward. Instead of printing to stdout, capture the bytes:

```rust
use std::sync::{Arc, Mutex};

let output = Arc::new(Mutex::new(Vec::new()));
let output_clone = Arc::clone(&output);

let uart = Pl011::new(move |byte| {
    output_clone.lock().unwrap().push(byte);
});

mmio_bus.register(0x0900_0000, 0x1000, Box::new(uart));

// ... run the VM ...

let bytes = output.lock().unwrap();
let message = String::from_utf8_lossy(&bytes);
assert_eq!(message, "Hello from the VM!\n");
```

The `Arc<Mutex<Vec<u8>>>` is needed because the closure outlives the function that creates the VM and the MMIO bus needs to own it. In a single-threaded VMM you could use `Rc<RefCell<>>` instead, but the Arc/Mutex version works in both threaded and single-threaded contexts.

## How This Becomes Meaningful

Right now, the UART address `0x09000000` is hardcoded in our guest program. We build ARM64 instructions that reference this specific address. This is fine for testing, but it is not how a real operating system discovers hardware.

When we boot Linux, the chain works differently:

1. We load a device tree blob (DTB) into guest memory alongside the kernel
2. The DTB contains a node describing the PL011: its address (0x09000000), its size (0x1000), its compatible string ("arm,pl011"), and its interrupt number
3. The kernel's PL011 driver matches on the compatible string and reads the address from the DTB
4. The driver memory-maps the address and starts writing to the data register
5. Our MMIO handler catches every write, routes it to the PL011 device, and we see boot messages on the terminal

The same MMIO mechanism. The same PL011 implementation. The same TX callback printing to stdout. The only difference is the size of the guest - a 20-byte hello-world program versus a multi-megabyte kernel. The device emulation doesn't care. It sees stores to 0x09000000 and outputs bytes.

This is the power of the MMIO abstraction we built in the previous post. Each device is a self-contained struct behind a trait. The bus does routing. The run loop does dispatch. Adding a new device to support a more complex guest doesn't require changing any of the plumbing - just implement the trait and register it at the right address.

## Testing

### Unit Tests

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use std::cell::RefCell;
    use std::rc::Rc;

    #[test]
    fn test_dr_write_calls_callback() {
        let received = Rc::new(RefCell::new(Vec::new()));
        let received_clone = Rc::clone(&received);

        let mut uart = Pl011::new(move |byte| {
            received_clone.borrow_mut().push(byte);
        });

        // Write 'A' to the Data Register
        uart.write(0x000, 4, 0x41);

        let bytes = received.borrow();
        assert_eq!(&*bytes, &[0x41]);
    }

    #[test]
    fn test_dr_write_masks_to_byte() {
        let received = Rc::new(RefCell::new(Vec::new()));
        let received_clone = Rc::clone(&received);

        let mut uart = Pl011::new(move |byte| {
            received_clone.borrow_mut().push(byte);
        });

        // Write a 64-bit value - only the low byte should be transmitted
        uart.write(0x000, 8, 0xDEADBEEF_00000048);

        let bytes = received.borrow();
        assert_eq!(&*bytes, &[0x48]); // 'H'
    }

    #[test]
    fn test_fr_read_returns_rx_empty() {
        let mut uart = Pl011::new(|_| {});

        // Flags register should indicate RX FIFO empty
        let flags = uart.read(0x018, 4);
        assert_eq!(flags & (1 << 4), 0x10); // RXFE bit set
        assert_eq!(flags & (1 << 5), 0);     // TXFF bit clear
    }

    #[test]
    fn test_unknown_register_read_returns_zero() {
        let mut uart = Pl011::new(|_| {});

        assert_eq!(uart.read(0x024, 4), 0); // IBRD
        assert_eq!(uart.read(0x028, 4), 0); // FBRD
        assert_eq!(uart.read(0x100, 4), 0); // non-existent
    }

    #[test]
    fn test_unknown_register_write_no_panic() {
        let mut uart = Pl011::new(|_| {});

        // These should silently succeed
        uart.write(0x024, 4, 0x1234); // IBRD
        uart.write(0x028, 4, 0x5678); // FBRD
        uart.write(0x02C, 4, 0x60);   // LCR_H
        uart.write(0x030, 4, 0x301);  // CR
    }

    #[test]
    fn test_full_message_output() {
        let received = Rc::new(RefCell::new(Vec::new()));
        let received_clone = Rc::clone(&received);

        let mut uart = Pl011::new(move |byte| {
            received_clone.borrow_mut().push(byte);
        });

        for &byte in b"Hello from the VM!\n" {
            uart.write(0x000, 4, byte as u64);
        }

        let bytes = received.borrow();
        let message = String::from_utf8_lossy(&bytes);
        assert_eq!(message, "Hello from the VM!\n");
    }
}
```

### Integration Test

The full integration test loads the guest program, runs the VM, and checks that the captured output matches the expected string:

```rust
#[test]
fn test_uart_hello_world() {
    let output = Arc::new(Mutex::new(Vec::new()));
    let output_clone = Arc::clone(&output);

    let vm = Vm::new().unwrap();
    let mut vcpu = vm.create_vcpu().unwrap();

    let mut mmio_bus = MmioBus::new();
    let uart = Pl011::new(move |byte| {
        output_clone.lock().unwrap().push(byte);
    });
    mmio_bus.register(0x0900_0000, 0x1000, Box::new(uart));

    // Load guest code
    let code = build_uart_hello();
    let mem = vm.map_memory(GUEST_MEM_BASE, GUEST_MEM_SIZE).unwrap();
    mem.write_u32_slice(0, &code);

    vcpu.set_reg(HV_REG_PC, GUEST_MEM_BASE).unwrap();

    // Run until HVC halt
    run_vm(&mut vcpu, &mut mmio_bus).unwrap();

    let bytes = output.lock().unwrap();
    let message = String::from_utf8_lossy(&bytes);
    assert_eq!(message, "Hello from the VM!\n");
}
```

This test exercises the entire stack: vCPU execution, MMIO data abort handling, syndrome decoding, bus routing, PL011 register emulation, and callback invocation. If any layer is broken, the output will be wrong.

## What's Next

We have a working serial console. The VM can print messages to the host terminal through a standard ARM UART interface. This is the first device that makes the hypervisor feel like a real system rather than an instruction-execution testbench.

But our guest programs are still raw instruction arrays. Hand-encoding ARM64 instructions as `u32` hex values is tedious, error-prone, and doesn't scale. A 20-character print takes 41 instructions. Imagine hand-encoding a program with loops, function calls, and stack management.

Next up: loading real ELF binaries. We will parse ELF headers, map segments into guest memory at the right addresses, and set the entry point. This lets us write guest programs in Rust or C, compile them with a bare-metal toolchain, and load the resulting binary into the VM. The instruction-encoding era ends, and the "write normal code that runs in a VM" era begins.
