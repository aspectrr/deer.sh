## How MMIO Works

On a real ARM system, hardware devices expose their control interfaces through memory-mapped registers. The UART lives at one address, the interrupt controller at another, the timer at a third. Software interacts with these devices by reading and writing to their addresses, the same way it reads and writes normal RAM. The difference is that behind those addresses there is no memory - there is silicon that interprets the access and does something.

Write `0x41` to the UART's data register, and the letter 'A' appears on the serial console. Read from the timer's counter register, and you get the current tick count. Write to the interrupt controller's enable register, and IRQ line 33 starts firing. From the CPU's perspective, these look identical to memory accesses. The magic happens in the memory bus routing: accesses to certain address ranges get directed to device controllers instead of RAM.

In a virtual machine, we fake this entire mechanism. The guest thinks it has hardware at certain addresses. The hypervisor deliberately leaves those address ranges unmapped in the stage-2 page tables. When the guest accesses an unmapped address, the CPU raises a stage-2 page fault, which surfaces as a VM exit with the data abort exception class. The VMM then decodes what the guest tried to do - which address, what size, read or write, which register - and emulates the device's behavior in software.

This is how every virtual device in every hypervisor works: QEMU, Firecracker, Cloud Hypervisor, crosvm. The UART, the virtio devices, the interrupt controller, the RTC - all of them are backed by MMIO trap handling. The only alternative is PIO (port I/O), which is x86-specific and doesn't exist on ARM.

Let's build the MMIO subsystem.

## Data Abort Syndrome Decoding

When the guest accesses an unmapped IPA (Intermediate Physical Address), we get an exception exit with EC = 0x24 (data abort from a lower exception level). The ISS field for data aborts is rich with information - the CPU has already decoded the trapped instruction for us.

Here are the ISS fields we need:

**ISV (bit 24) - Instruction Syndrome Valid**: If this bit is 1, the CPU was able to decode the trapped instruction and the rest of the ISS fields are valid. If it is 0, we are out of luck - the CPU couldn't decode the access (this happens with certain load/store pair instructions and other complex addressing modes). For those cases you would need to fetch and decode the instruction yourself, but for most MMIO accesses ISV will be 1.

**SAS (bits [23:22]) - Syndrome Access Size**: Encodes the size of the access. 0b00 = 1 byte, 0b01 = 2 bytes (halfword), 0b10 = 4 bytes (word), 0b11 = 8 bytes (doubleword). This tells us how wide the guest's load or store was.

**SRT (bits [20:16]) - Syndrome Register Transfer**: The index of the general-purpose register involved in the access. For a store, this register contains the value being written. For a load, this is the register that should receive the read result.

**WnR (bit 6) - Write-not-Read**: 1 means the guest was writing to the address. 0 means it was reading.

In addition to the syndrome, the exit information structure provides the faulting physical address via `exit.exception.physical_address`. This is the guest physical address (IPA) that caused the abort - the address of the device register the guest was trying to access.

Here are the extraction helpers:

```rust
// src/arch/syndrome.rs (continued)

/// Check if the Instruction Syndrome is Valid for a data abort.
/// When false, SAS/SRT/WnR fields cannot be trusted.
pub fn dabt_isv(iss: u32) -> bool {
    (iss >> 24) & 1 == 1
}

/// Extract Write-not-Read bit. true = guest was writing, false = reading.
pub fn dabt_wnr(iss: u32) -> bool {
    (iss >> 6) & 1 == 1
}

/// Extract Syndrome Access Size. 0=byte, 1=halfword, 2=word, 3=doubleword.
pub fn dabt_sas(iss: u32) -> u32 {
    (iss >> 22) & 0x3
}

/// Extract Syndrome Register Transfer - which general-purpose register
/// is the source (for writes) or destination (for reads).
pub fn dabt_srt(iss: u32) -> u32 {
    (iss >> 16) & 0x1f
}

/// Convert SAS encoding to byte count: 0->1, 1->2, 2->4, 3->8.
pub fn dabt_access_size_bytes(sas: u32) -> u32 {
    1 << sas
}
```

Together, these five pieces of information tell us everything about the trapped access: the address, the size, the direction, and which register to read from or write to. The CPU has done the hard work of instruction decoding for us - we just need to parse the syndrome bits.

## The MmioDevice Trait

Before we can handle MMIO exits, we need a way to represent virtual devices. Each device maps to a region of the guest's physical address space and responds to reads and writes at offsets within that region.

The trait is minimal:

```rust
// src/devices/mod.rs

pub trait MmioDevice {
    /// Handle a read from the device.
    /// `offset` is relative to the device's base address.
    /// `size` is the access width in bytes (1, 2, 4, or 8).
    /// Returns the value the guest should see.
    fn read(&mut self, offset: u64, size: u32) -> u64;

    /// Handle a write to the device.
    /// `offset` is relative to the device's base address.
    /// `size` is the access width in bytes (1, 2, 4, or 8).
    /// `value` is what the guest wrote (masked to `size` bytes).
    fn write(&mut self, offset: u64, size: u32, value: u64);
}
```

The offset parameter is relative to the device's base address, not the absolute guest physical address. If a UART is mapped at `0x0900_0000` and the guest writes to `0x0900_0018`, the device sees offset `0x18`. This keeps devices portable - they don't need to know where they are mapped in the guest's address space.

The size parameter matters because real hardware registers have specific access width requirements. The PL011 UART's data register is 8 bits wide; writing a 32-bit value to it should only use the low 8 bits. The GICv2 distributor has 32-bit registers; a 64-bit access would span two registers. Devices need to know the access size to behave correctly.

## The MMIO Bus

With individual devices defined, we need a bus that routes accesses to the correct device based on address:

```rust
// src/mmio/bus.rs

use crate::arch::syndrome::*;
use crate::devices::MmioDevice;
use crate::vcpu::Vcpu;

struct MmioRegion {
    base: u64,
    size: u64,
    device: Box<dyn MmioDevice>,
}

pub struct MmioBus {
    regions: Vec<MmioRegion>,
}

impl MmioBus {
    pub fn new() -> Self {
        MmioBus {
            regions: Vec::new(),
        }
    }

    /// Register a device at the given base address with the given size.
    /// The device will handle accesses to [base, base + size).
    pub fn register(
        &mut self,
        base: u64,
        size: u64,
        device: Box<dyn MmioDevice>,
    ) {
        self.regions.push(MmioRegion { base, size, device });
    }

    /// Handle a data abort by routing it to the appropriate device.
    /// Returns an error if no device is registered at the faulting address.
    pub fn handle_data_abort(
        &mut self,
        vcpu: &mut Vcpu,
        iss_val: u32,
        addr: u64,
    ) -> Result<(), Box<dyn std::error::Error>> {
        // Validate that the syndrome is usable
        if !dabt_isv(iss_val) {
            return Err(format!(
                "data abort at 0x{addr:08x} with ISV=0 (cannot decode access)"
            ).into());
        }

        let is_write = dabt_wnr(iss_val);
        let sas = dabt_sas(iss_val);
        let size = dabt_access_size_bytes(sas);
        let srt = dabt_srt(iss_val);

        // Find the device that owns this address
        let region = self.regions.iter_mut().find(|r| {
            addr >= r.base && addr < r.base + r.size
        }).ok_or_else(|| {
            format!("no MMIO device registered at address 0x{addr:08x}")
        })?;

        let offset = addr - region.base;

        if is_write {
            // Guest is writing: read the value from the source register
            let reg = hv_reg_for_index(srt);
            let value = vcpu.get_reg(reg)?;
            // Mask the value to the access size
            let mask = match size {
                1 => 0xff,
                2 => 0xffff,
                4 => 0xffff_ffff,
                8 => u64::MAX,
                _ => unreachable!(),
            };
            region.device.write(offset, size, value & mask);
        } else {
            // Guest is reading: call the device and write result to
            // the destination register
            let value = region.device.read(offset, size);
            let reg = hv_reg_for_index(srt);
            vcpu.set_reg(reg, value)?;
        }

        Ok(())
    }
}

/// Map a register index (0-30) to the corresponding HV_REG constant.
/// Index 31 is the zero register (XZR) - writes are discarded, reads return 0.
fn hv_reg_for_index(index: u32) -> hv_reg_t {
    match index {
        0 => HV_REG_X0,
        1 => HV_REG_X1,
        2 => HV_REG_X2,
        3 => HV_REG_X3,
        4 => HV_REG_X4,
        5 => HV_REG_X5,
        // ... x6 through x29 ...
        30 => HV_REG_X30,
        _ => panic!("invalid register index: {index}"),
    }
}
```

The routing logic is simple: linear scan through registered regions, check if the address falls within [base, base+size), and dispatch to the device. For a hypervisor with hundreds of devices you would want a more efficient lookup (a sorted array with binary search, or an interval tree), but a linear scan is fine for the handful of devices we will have.

The `hv_reg_for_index` function bridges the gap between the SRT field (a 5-bit register index) and Hypervisor.framework's register constants. This mapping is straightforward but necessary - the syndrome gives us a number, the API wants an enum value.

## Integrating MMIO into the Run Loop

Adding MMIO handling to the exit loop is a single new match arm:

```rust
EC_DABT_LOWER => {
    let iss_val = iss(syndrome);
    let addr = exit.exception.physical_address;
    mmio_bus.handle_data_abort(&mut vcpu, iss_val, addr)?;
    vcpu.advance_pc(4)?;
}
```

That's it. The data abort fires, we extract the ISS and faulting address, hand them to the MMIO bus, and advance past the trapped instruction. All the complexity lives in the bus and device implementations, not the run loop. This is the right separation of concerns - the run loop does dispatch, the bus does routing, the device does emulation.

The updated run loop now looks like:

```rust
pub fn run_vm(
    vcpu: &mut Vcpu,
    mmio_bus: &mut MmioBus,
) -> Result<(), Box<dyn std::error::Error>> {
    loop {
        vcpu.run()?;
        let exit = vcpu.exit_info();

        match exit.reason {
            HV_EXIT_REASON_EXCEPTION => {
                let syndrome = exit.exception.syndrome;

                match ec(syndrome) {
                    EC_HVC64 => {
                        handle_hvc(vcpu, iss(syndrome))?;
                        vcpu.advance_pc(4)?;
                    }
                    EC_SMC64 => {
                        handle_smc(vcpu)?;
                        vcpu.advance_pc(4)?;
                    }
                    EC_WFI => {
                        vcpu.advance_pc(4)?;
                    }
                    EC_DABT_LOWER => {
                        let iss_val = iss(syndrome);
                        let addr = exit.exception.physical_address;
                        mmio_bus.handle_data_abort(vcpu, iss_val, addr)?;
                        vcpu.advance_pc(4)?;
                    }
                    EC_BRK64 => {
                        println!("BRK #{} - halting", iss(syndrome));
                        break;
                    }
                    ec_val => {
                        panic!("unhandled EC: 0x{ec_val:02x}");
                    }
                }
            }
            HV_EXIT_REASON_VTIMER_ACTIVATED => {
                vcpu.set_vtimer_mask(true)?;
            }
            reason => panic!("unknown exit: {reason:?}"),
        }
    }
    Ok(())
}
```

## Demo Devices

Before we build real hardware emulation, let's prove the MMIO subsystem works with two trivial devices.

### Echo Device

Write a value, read it back. The simplest possible stateful device:

```rust
// src/devices/echo.rs

use crate::devices::MmioDevice;

pub struct EchoDevice {
    last_value: u64,
}

impl EchoDevice {
    pub fn new() -> Self {
        EchoDevice { last_value: 0 }
    }
}

impl MmioDevice for EchoDevice {
    fn read(&mut self, _offset: u64, _size: u32) -> u64 {
        self.last_value
    }

    fn write(&mut self, _offset: u64, _size: u32, value: u64) {
        self.last_value = value;
    }
}
```

The echo device ignores the offset and size parameters entirely. It has one piece of state - the last written value - and every read returns it. This is not how real hardware works (real devices have multiple registers at different offsets), but it is perfect for validating that the MMIO plumbing is correct: writes go in, reads come back.

### Counter Device

A read-only device that increments on each read:

```rust
// src/devices/counter.rs

use crate::devices::MmioDevice;

pub struct CounterDevice {
    count: u64,
}

impl CounterDevice {
    pub fn new() -> Self {
        CounterDevice { count: 0 }
    }
}

impl MmioDevice for CounterDevice {
    fn read(&mut self, _offset: u64, _size: u32) -> u64 {
        self.count += 1;
        self.count
    }

    fn write(&mut self, _offset: u64, _size: u32, _value: u64) {
        // Writes are ignored - this is a read-only counter.
    }
}
```

This device demonstrates something important about MMIO: reads can have side effects. On real hardware, reading a status register might clear a pending interrupt flag. Reading a FIFO register consumes data from the queue. The counter device is a toy version of this pattern - each read mutates the device's state.

### Registering the Devices

```rust
let mut mmio_bus = MmioBus::new();

// Echo device at 0x1000_0000, 4KB region
mmio_bus.register(
    0x1000_0000,
    0x1000,
    Box::new(EchoDevice::new()),
);

// Counter device at 0x1000_1000, 4KB region
mmio_bus.register(
    0x1000_1000,
    0x1000,
    Box::new(CounterDevice::new()),
);
```

We give each device a 4KB region (one page), which is the minimum granularity for stage-2 page table mappings. The addresses are arbitrary - we just need to make sure they don't overlap with guest RAM or each other.

## The Guest Test Program

Here is an ARM64 program that exercises both MMIO devices. Each memory access to the device addresses will cause a data abort, which the MMIO bus will route to the appropriate device.

```rust
let echo_base: u64 = 0x1000_0000;
let counter_base: u64 = 0x1000_1000;

let code: &[u32] = &[
    // --- Load device base addresses ---
    // x10 = echo device base (0x10000000)
    0xD2A2000A,     // mov x10, #0x1000, lsl #16
    // x11 = counter device base (0x10001000)
    0xD2A2002B,     // mov x11, #0x1001, lsl #16

    // --- Echo device: write 0x42, then read it back ---
    0xD2800840,     // mov x0, #0x42
    0xF900014A,     // str x0, [x10]         -- write 0x42 to echo
    0xF940014A,     // ldr x1, [x10]         -- read from echo -> x1

    // --- Counter device: read twice ---
    0xF940016B,     // ldr x2, [x11]         -- first read -> x2 (expect 1)
    0xF940016C,     // ldr x3, [x11]         -- second read -> x3 (expect 2)

    // --- Halt ---
    0xD4200000,     // brk #0
];
```

The expected execution trace:

1. `mov x10, ...` and `mov x11, ...` - load base addresses, no MMIO exits
2. `mov x0, #0x42` - set the value to write, no MMIO exit
3. `str x0, [x10]` - **data abort**: write 0x42 to echo device at offset 0
4. `ldr x1, [x10]` - **data abort**: read from echo device, x1 receives 0x42
5. `ldr x2, [x11]` - **data abort**: read from counter device, x2 receives 1
6. `ldr x3, [x11]` - **data abort**: read from counter device, x3 receives 2
7. `brk #0` - halt

After execution: x0=0x42, x1=0x42, x2=1, x3=2.

The fact that x1 equals x0 proves the echo device round-trip works. The fact that x2 and x3 are consecutive proves the counter device maintains state across reads. Four MMIO traps, four correct emulations.

## The Full MMIO Data Flow

Let's trace a single MMIO access end to end to make the mechanism completely clear. Take the `str x0, [x10]` instruction, where x0=0x42 and x10=0x10000000:

1. The guest CPU executes `str x0, [x10]`. This is a store of x0 to the address in x10.
2. The CPU translates the guest virtual address through the guest's page tables (stage-1) to get a guest physical address: 0x10000000.
3. The CPU walks the stage-2 page tables to translate the guest physical address to a host physical address. There is no mapping for 0x10000000 - we deliberately left this range unmapped.
4. The CPU raises a stage-2 data abort. Execution transfers to EL2 (the hypervisor).
5. `hv_vcpu_run()` returns. The exit reason is `HV_EXIT_REASON_EXCEPTION`.
6. We read the syndrome: EC=0x24 (data abort from lower EL).
7. We extract ISS fields: ISV=1, WnR=1 (write), SAS=0b11 (8 bytes), SRT=0 (x0).
8. We read the faulting address: 0x10000000.
9. The MMIO bus finds the echo device (base=0x10000000, size=0x1000).
10. Offset = 0x10000000 - 0x10000000 = 0.
11. It's a write, so we read x0 (value = 0x42) and call `echo_device.write(0, 8, 0x42)`.
12. The echo device stores 0x42 in `self.last_value`.
13. We advance PC by 4.
14. We call `hv_vcpu_run()` again. The guest continues with the next instruction.

Thirteen steps for one memory access. It sounds expensive, and it is - each MMIO access costs thousands of cycles for the VM exit and re-entry. This is why high-performance virtual devices use virtio with shared memory rings instead of MMIO for data transfer. MMIO is used for control operations (configuration, status checks, doorbell notifications) where the frequency is low enough that the overhead doesn't matter.

## Testing

The syndrome extraction helpers are straightforward to unit test:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_dabt_wnr() {
        // Bit 6 = 1: write
        assert!(dabt_wnr(1 << 6));
        // Bit 6 = 0: read
        assert!(!dabt_wnr(0));
    }

    #[test]
    fn test_dabt_sas() {
        // SAS = 0b11 (doubleword) at bits [23:22]
        assert_eq!(dabt_sas(0b11 << 22), 3);
        assert_eq!(dabt_access_size_bytes(3), 8);

        // SAS = 0b00 (byte)
        assert_eq!(dabt_sas(0), 0);
        assert_eq!(dabt_access_size_bytes(0), 1);
    }

    #[test]
    fn test_dabt_srt() {
        // SRT = 15 at bits [20:16]
        assert_eq!(dabt_srt(15 << 16), 15);
    }

    #[test]
    fn test_dabt_isv() {
        assert!(dabt_isv(1 << 24));
        assert!(!dabt_isv(0));
    }
}
```

The MMIO bus routing can be tested without a real vCPU:

```rust
#[test]
fn test_mmio_bus_routing() {
    let mut bus = MmioBus::new();
    bus.register(0x1000, 0x100, Box::new(EchoDevice::new()));
    bus.register(0x2000, 0x100, Box::new(CounterDevice::new()));

    // Address 0x1050 should route to echo device with offset 0x50
    let region = bus.find_region(0x1050);
    assert!(region.is_some());

    // Address 0x3000 should not match any device
    let region = bus.find_region(0x3000);
    assert!(region.is_none());
}

#[test]
fn test_echo_device_roundtrip() {
    let mut echo = EchoDevice::new();
    echo.write(0, 4, 0xDEADBEEF);
    assert_eq!(echo.read(0, 4), 0xDEADBEEF);
}

#[test]
fn test_counter_device_sequence() {
    let mut counter = CounterDevice::new();
    assert_eq!(counter.read(0, 4), 1);
    assert_eq!(counter.read(0, 4), 2);
    assert_eq!(counter.read(0, 4), 3);
    // Writes should be ignored
    counter.write(0, 4, 999);
    assert_eq!(counter.read(0, 4), 4);
}
```

The full integration test runs the guest program and verifies the final register state. This catches end-to-end bugs: incorrect syndrome decoding, wrong register mapping, bus routing errors, device logic errors, and PC advancement issues would all show up as wrong register values.

## What We Have Now

The MMIO subsystem gives us the ability to create arbitrary virtual hardware. Any Rust struct that implements the `MmioDevice` trait can be mapped into the guest's address space and will respond to loads and stores from guest code. The plumbing is:

1. Guest accesses unmapped address
2. Stage-2 fault causes data abort VM exit
3. Run loop dispatches to MMIO bus
4. Bus routes to device by address
5. Device emulates the access
6. Guest continues

This is the foundation for everything that follows. The next post uses this exact mechanism to build the first real device: a PL011 UART that gives the VM a serial console. Same trait, same bus, same dispatch path - just a device implementation that talks to the host's terminal instead of storing numbers in a struct field.
