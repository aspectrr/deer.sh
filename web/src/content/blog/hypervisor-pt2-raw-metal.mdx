## The Project Setup

Every hypervisor starts the same way: an empty directory and a question about how the hardware actually works. Ours starts with `cargo new`.

```bash
cargo new apple-hypervisor
cd apple-hypervisor
```

The `Cargo.toml` is deliberately minimal. We are not pulling in a hypervisor crate - the entire point is to write the bindings ourselves. The only dependencies are for ergonomics: error handling and logging.

```toml
[package]
name = "apple-hypervisor"
version = "0.1.0"
edition = "2024"

[dependencies]
anyhow = "1"
thiserror = "2"
log = "0.4"
env_logger = "0.11"
```

No `hvf` crate, no `hypervisor` crate, no `applevisor`. Just us, the C headers, and `extern "C"`.

### The Build Script

Rust needs to know to link against Hypervisor.framework. That is the job of `build.rs`, placed in the project root:

```rust
fn main() {
    println!("cargo:rustc-link-lib=framework=Hypervisor");
}
```

One line. The `framework=` prefix tells the linker to look for a macOS framework rather than a regular library. At link time, this becomes `-framework Hypervisor` in the linker invocation. If you have ever written Objective-C code that links against a framework, this is the Rust equivalent.

### Entitlements

Apple's Hypervisor.framework requires an entitlement to use. Without it, every call returns `HV_DENIED` (permission denied). The entitlement is declared in a property list file:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>com.apple.security.hypervisor</key>
    <true/>
</dict>
</plist>
```

Save this as `entitlements.plist` in the project root.

### The Build Script

We need a shell script that builds the binary and then codesigns it with the entitlement. Without the codesigning step, the binary will not be able to create VMs:

```bash
#!/bin/bash
set -euo pipefail

cargo build --release

codesign --entitlements entitlements.plist \
         --force \
         -s - \
         target/release/apple-hypervisor
```

Save this as `build.sh` and make it executable:

```bash
chmod +x build.sh
```

The `-s -` flag means "ad-hoc sign" - no Apple Developer certificate required. This is sufficient for local development. The `--force` flag overwrites any existing signature, which is necessary because `cargo build` produces an unsigned binary and we need to replace it.

Your project directory should now look like this:

```
apple-hypervisor/
  build.rs
  build.sh
  Cargo.toml
  entitlements.plist
  src/
    main.rs
```

## Understanding the C API

Hypervisor.framework exposes a C API with roughly 30 functions. Apple documents them in their developer documentation, and the actual type definitions live in the framework's header files under `/Library/Developer/CommandLineTools/SDKs/MacOSX.sdk/System/Library/Frameworks/Hypervisor.framework/Headers/`.

The API is refreshingly small. There is no object model, no inheritance, no COM interfaces. It is flat C: you call functions, pass in pointers, and check return codes. The types are mostly typedefs over integers, and the structs are simple data containers with no methods.

To create FFI bindings, we need to translate every type from C to Rust. The rules are straightforward:

- C `int32_t` becomes Rust `i32`
- C `uint64_t` becomes Rust `u64`
- C `uint32_t` becomes Rust `u32`
- C `uint16_t` becomes Rust `u16`
- C `size_t` becomes Rust `usize`
- C `void*` becomes Rust `*const std::ffi::c_void` or `*mut std::ffi::c_void`
- C `bool` becomes Rust `bool`
- C structs with `#[repr(C)]` to ensure identical memory layout

The `#[repr(C)]` attribute is critical. Without it, the Rust compiler is free to reorder struct fields, add padding, or change alignment. With it, Rust uses the same layout rules as the C compiler, which means our struct and the C struct will have identical memory representations. If you forget `#[repr(C)]` on a struct that crosses the FFI boundary, you will get silent memory corruption.

## The Types Module

Create `src/hv.rs`. This will contain every type, constant, and function declaration we need. We start with the basic type aliases:

```rust
// src/hv.rs

// Return type for all Hypervisor.framework functions
pub type hv_return_t = i32;

// Handle for a virtual CPU
pub type hv_vcpu_t = u64;

// Guest intermediate physical address
pub type hv_ipa_t = u64;

// Memory mapping permission flags
pub type hv_memory_flags_t = u64;
```

These are direct translations of the C typedefs. `hv_return_t` is an error code (0 for success, negative for errors). `hv_vcpu_t` is an opaque handle - the framework returns one when you create a vCPU, and you pass it to every subsequent vCPU operation. `hv_ipa_t` is a guest physical address. `hv_memory_flags_t` is a bitmask for read/write/execute permissions.

### Error Codes

Next, the error constants. Every framework function returns an `hv_return_t`, and we need to match on these values:

```rust
pub const HV_SUCCESS: hv_return_t = 0;
pub const HV_ERROR: hv_return_t = 0xfae94001u32 as i32;
pub const HV_BUSY: hv_return_t = 0xfae94002u32 as i32;
pub const HV_BAD_ARGUMENT: hv_return_t = 0xfae94003u32 as i32;
pub const HV_NO_RESOURCES: hv_return_t = 0xfae94005u32 as i32;
pub const HV_NO_DEVICE: hv_return_t = 0xfae94006u32 as i32;
pub const HV_DENIED: hv_return_t = 0xfae94007u32 as i32;
pub const HV_UNSUPPORTED: hv_return_t = 0xfae9400fu32 as i32;
```

The cast from `u32` to `i32` is necessary because these are large unsigned values that overflow the signed 32-bit range. Rust requires the explicit cast. In C, the header just writes `0xfae94001` and the compiler handles it implicitly.

`HV_DENIED` is the one you will see most often during development - it means the binary is missing the hypervisor entitlement.

### Register Constants

The numbering is straightforward: X0 is 0, X1 is 1, and so on. PC is 31, then the floating-point control/status registers, then CPSR. These constants get passed to `hv_vcpu_get_reg` and `hv_vcpu_set_reg`.

### System Register Constants

System registers are a different namespace. ARM64 identifies system registers by five fields: Op0, Op1, CRn, CRm, Op2. The framework encodes these into a `u16`. We do not need to understand the encoding - we just need the constants for the registers we will use:

```rust
// System registers (u16 encoding of Op0/Op1/CRn/CRm/Op2)
pub const HV_SYS_REG_SP_EL0: u16 = 0xC208;    // Stack pointer (EL0)
pub const HV_SYS_REG_SP_EL1: u16 = 0xE208;     // Stack pointer (EL1)
pub const HV_SYS_REG_SPSR_EL1: u16 = 0xE200;   // Saved program status register
pub const HV_SYS_REG_ELR_EL1: u16 = 0xE201;    // Exception link register
pub const HV_SYS_REG_SCTLR_EL1: u16 = 0xC080;  // System control register
pub const HV_SYS_REG_VBAR_EL1: u16 = 0xC600;   // Vector base address register
pub const HV_SYS_REG_MAIR_EL1: u16 = 0xC510;   // Memory attribute indirection register
pub const HV_SYS_REG_TCR_EL1: u16 = 0xC102;    // Translation control register
pub const HV_SYS_REG_TTBR0_EL1: u16 = 0xC100;  // Translation table base register 0
pub const HV_SYS_REG_TTBR1_EL1: u16 = 0xC101;  // Translation table base register 1
```

These will matter later when the guest kernel configures its page tables and exception vectors. When the guest writes to SCTLR_EL1 to enable the MMU, or sets VBAR_EL1 to point at its exception vector table, the hypervisor may need to trap and emulate those accesses. For now, we just define the constants.

### Memory Flags

Permissions for guest memory mappings are a bitmask:

```rust
pub const HV_MEMORY_READ: hv_memory_flags_t = 1 << 0;
pub const HV_MEMORY_WRITE: hv_memory_flags_t = 1 << 1;
pub const HV_MEMORY_EXEC: hv_memory_flags_t = 1 << 2;
```

When we map memory into the guest's physical address space with `hv_vm_map`, we pass a combination of these flags. Guest RAM gets `READ | WRITE | EXEC`. A read-only firmware region might get just `READ | EXEC`. A write-only device register might get just `WRITE`. The stage-2 page tables enforce these permissions in hardware.

### Exit Reasons

When `hv_vcpu_run()` returns, the exit reason tells us why the guest stopped:

```rust
pub const HV_EXIT_REASON_CANCELED: u32 = 0;
pub const HV_EXIT_REASON_EXCEPTION: u32 = 1;
pub const HV_EXIT_REASON_VTIMER_ACTIVATED: u32 = 2;
pub const HV_EXIT_REASON_UNKNOWN: u32 = 3;
```

`EXCEPTION` is the most common - it covers HVC calls, MMIO traps, system register traps, and WFI. We distinguish between these by examining the syndrome value in the exit struct. `VTIMER_ACTIVATED` fires when the guest's virtual timer expires. `CANCELED` occurs when another thread calls `hv_vcpu_force_exit()` on our vCPU. `UNKNOWN` should not happen in practice.

### The Exit Struct

This is the most important data structure in the entire API. After every VM exit, we read this struct to find out what happened:

```rust
#[repr(C)]
pub struct hv_vcpu_exit_exception_t {
    pub syndrome: u64,
    pub virtual_address: u64,
    pub physical_address: u64,
}

#[repr(C)]
pub struct hv_vcpu_exit_t {
    pub reason: u32,
    pub exception: hv_vcpu_exit_exception_t,
}
```

The `reason` field is one of the exit reason constants above. When it is `HV_EXIT_REASON_EXCEPTION`, the `exception` field is populated:

- `syndrome` contains the ARM ESR_EL2 value. The top 6 bits (EC field) tell us the exception class: 0x16 for HVC, 0x24 for data abort, 0x18 for system register trap, etc. The remaining bits are instruction-specific.
- `virtual_address` is the guest virtual address that caused the fault (for data aborts).
- `physical_address` is the guest IPA that caused the fault (for stage-2 faults, i.e., MMIO).

The `#[repr(C)]` on both structs is essential. The framework writes directly into this memory after a VM exit. If our layout does not match the C layout byte-for-byte, we will read garbage.

## The Extern Block

Now for the function declarations. These are `extern "C"` functions - Rust's way of declaring symbols that exist in an external C library. We do not provide implementations; the linker resolves them against Hypervisor.framework at link time.

```rust
extern "C" {
    // VM lifecycle
    pub fn hv_vm_create(config: *const std::ffi::c_void) -> hv_return_t;
    pub fn hv_vm_destroy() -> hv_return_t;

    // Memory mapping
    pub fn hv_vm_map(
        addr: *const u8,
        ipa: hv_ipa_t,
        size: usize,
        flags: hv_memory_flags_t,
    ) -> hv_return_t;
    pub fn hv_vm_unmap(ipa: hv_ipa_t, size: usize) -> hv_return_t;

    // vCPU lifecycle
    pub fn hv_vcpu_create(
        vcpu: *mut hv_vcpu_t,
        exit: *mut *const hv_vcpu_exit_t,
        config: *const std::ffi::c_void,
    ) -> hv_return_t;
    pub fn hv_vcpu_destroy(vcpu: hv_vcpu_t) -> hv_return_t;

    // vCPU execution
    pub fn hv_vcpu_run(vcpu: hv_vcpu_t) -> hv_return_t;

    // General-purpose registers
    pub fn hv_vcpu_get_reg(
        vcpu: hv_vcpu_t,
        reg: u32,
        value: *mut u64,
    ) -> hv_return_t;
    pub fn hv_vcpu_set_reg(
        vcpu: hv_vcpu_t,
        reg: u32,
        value: u64,
    ) -> hv_return_t;

    // System registers
    pub fn hv_vcpu_get_sys_reg(
        vcpu: hv_vcpu_t,
        reg: u16,
        value: *mut u64,
    ) -> hv_return_t;
    pub fn hv_vcpu_set_sys_reg(
        vcpu: hv_vcpu_t,
        reg: u16,
        value: u64,
    ) -> hv_return_t;

    // Timer and interrupts
    pub fn hv_vcpu_set_vtimer_mask(
        vcpu: hv_vcpu_t,
        masked: bool,
    ) -> hv_return_t;
    pub fn hv_vcpu_set_pending_interrupt(
        vcpu: hv_vcpu_t,
        interrupt_type: u32,
        pending: bool,
    ) -> hv_return_t;
}
```

A few things to notice:

**`hv_vm_create` and `hv_vm_destroy` take no VM handle.** There is a global, process-wide VM. You create it once, and all subsequent operations (mapping memory, creating vCPUs) operate on that single VM. This is a framework limitation - one VM per process. If you need multiple VMs, you need multiple processes.

**`hv_vcpu_create` has an out-parameter for the exit struct.** It allocates the `hv_vcpu_exit_t` internally and gives you a pointer to it. You do not own this memory - the framework manages it. After each call to `hv_vcpu_run`, the framework updates this struct in-place. You read it, handle the exit, and call `hv_vcpu_run` again.

**`hv_vm_map` takes a host pointer.** The `addr` parameter is a pointer to memory in your process's address space. The framework maps this memory into the guest's IPA space at the address you specify. The memory must be page-aligned (4096 bytes on ARM64) and the size must be a multiple of the page size. If you pass unaligned values, you get `HV_BAD_ARGUMENT`.

**Both config parameters are nullable.** We pass `std::ptr::null()` for both. The framework supports optional configuration objects for VMs and vCPUs, but we do not need them for basic operation.

## Proof of Life

With all the types and functions declared, we can write a minimal program that proves the bindings work. This is the "hello world" of hypervisor development: create a VM, check that it succeeded, destroy it.

First, make `hv.rs` visible to the rest of the project. Update `src/main.rs`:

```rust
mod hv;

use hv::*;

fn main() {
    env_logger::init();

    println!("Creating VM...");

    let ret = unsafe { hv_vm_create(std::ptr::null()) };
    if ret != HV_SUCCESS {
        eprintln!(
            "hv_vm_create failed: 0x{:08x}",
            ret as u32
        );
        if ret == HV_DENIED {
            eprintln!(
                "Permission denied. Did you codesign with the \
                 hypervisor entitlement?"
            );
            eprintln!(
                "Run: codesign --entitlements entitlements.plist \
                 --force -s - target/release/apple-hypervisor"
            );
        }
        std::process::exit(1);
    }

    println!("Hypervisor.framework is alive!");
    println!("VM created successfully.");

    let ret = unsafe { hv_vm_destroy() };
    if ret != HV_SUCCESS {
        eprintln!(
            "hv_vm_destroy failed: 0x{:08x}",
            ret as u32
        );
        std::process::exit(1);
    }

    println!("VM destroyed. Clean exit.");
}
```

Build and run:

```bash
./build.sh && ./target/release/apple-hypervisor
```

If everything is set up correctly, you should see:

```terminal
Creating VM...
Hypervisor.framework is alive!
VM created successfully.
VM destroyed. Clean exit.
```

If you see `Permission denied`, the binary was not codesigned with the entitlement. If you see `No device`, your Mac does not support Hypervisor.framework (all Apple Silicon Macs do, but very old Intel Macs may not).

This is not much to look at, but it represents something significant. Your userspace process just created a hardware-backed virtual machine context in the CPU. The EL2 state has been configured, stage-2 page tables have been allocated (empty, for now), and the kernel is ready to create vCPUs on demand. We have not run any guest code yet, but the infrastructure is in place.

### Common Errors

**`HV_DENIED` (0xfae94007):** Missing entitlement. The binary must be codesigned with `com.apple.security.hypervisor`. This is the single most common error. Note that `cargo build` produces a _new_ binary each time, so you need to codesign after every build. The `build.sh` script handles this.

**`HV_NO_DEVICE` (0xfae94006):** The hardware does not support Hypervisor.framework, or the feature is disabled. On Apple Silicon, this should not happen unless something is very wrong with your system.

**`HV_BUSY` (0xfae94002):** A VM already exists in this process. You called `hv_vm_create` twice without calling `hv_vm_destroy` in between. Remember: one VM per process.

**Link error: `framework not found Hypervisor`:** Your `build.rs` is missing or incorrect. Make sure it contains the `println!("cargo:rustc-link-lib=framework=Hypervisor");` line.

**Segfault on `hv_vcpu_run`:** You created a vCPU but did not set up any guest memory or set the PC to a valid address. The CPU tried to fetch an instruction from an unmapped IPA and the framework did not handle it gracefully. We will fix this in the next post.

## Testing

Testing hypervisor code is tricky because every test binary needs to be codesigned. The standard `cargo test` workflow does not codesign test binaries, so integration tests that call `hv_vm_create` will fail with `HV_DENIED`.

For unit tests that do not touch the hypervisor API, standard `cargo test` works fine. We can verify our struct layouts:

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use std::mem;

    #[test]
    fn test_exit_exception_size() {
        // 3 u64 fields = 24 bytes
        assert_eq!(
            mem::size_of::<hv_vcpu_exit_exception_t>(),
            24
        );
    }

    #[test]
    fn test_exit_struct_size() {
        // u32 reason (4 bytes) + 4 bytes padding + exception (24 bytes)
        // = 32 bytes total due to alignment
        assert_eq!(
            mem::size_of::<hv_vcpu_exit_t>(),
            32
        );
    }

    #[test]
    fn test_exit_struct_alignment() {
        // Must be 8-byte aligned because it contains u64 fields
        assert_eq!(
            mem::align_of::<hv_vcpu_exit_t>(),
            8
        );
    }

    #[test]
    fn test_error_codes_are_negative() {
        // All error codes should be negative when interpreted as i32
        assert!(HV_ERROR < 0);
        assert!(HV_BUSY < 0);
        assert!(HV_BAD_ARGUMENT < 0);
        assert!(HV_NO_RESOURCES < 0);
        assert!(HV_NO_DEVICE < 0);
        assert!(HV_DENIED < 0);
        assert!(HV_UNSUPPORTED < 0);
    }

    #[test]
    fn test_success_is_zero() {
        assert_eq!(HV_SUCCESS, 0);
    }

    #[test]
    fn test_register_constants_sequential() {
        assert_eq!(HV_REG_X0, 0);
        assert_eq!(HV_REG_X30, 30);
        assert_eq!(HV_REG_PC, 31);
    }

    #[test]
    fn test_memory_flags_are_powers_of_two() {
        assert_eq!(HV_MEMORY_READ, 1);
        assert_eq!(HV_MEMORY_WRITE, 2);
        assert_eq!(HV_MEMORY_EXEC, 4);
    }
}
```

These tests verify that our Rust types match the expected C layout. The `hv_vcpu_exit_t` struct has 32 bytes total: 4 bytes for the `u32` reason, 4 bytes of padding (because the next field is a struct containing `u64`s, which requires 8-byte alignment), and 24 bytes for the exception struct. If these sizes ever change, the tests will catch it before we get mysterious data corruption at runtime.

For integration tests that actually create VMs, you have two options:

**Option 1: Manual codesign.** Build the test binary with `cargo test --no-run`, find the binary in `target/debug/deps/`, codesign it, then run it directly. This is tedious but works.

**Option 2: Integration test binary.** Write a separate binary in `examples/` or `tests/` that does the VM creation, build it with `cargo build --example test_vm`, codesign it, and run it. This is what we will do for our "proof of life" tests.

For now, the struct size tests are sufficient to verify our bindings are correct. The `main.rs` proof-of-life program serves as our integration test.

## What We Have

Let us take stock. In a single file (`src/hv.rs`) plus a minimal `main.rs`, we have:

- 8 error code constants
- 35 register constants (X0-X30, PC, FPCR, FPSR, CPSR)
- 10 system register constants
- 3 memory flag constants
- 4 exit reason constants
- 2 `#[repr(C)]` structs matching the framework's exit information layout
- 14 `extern "C"` function declarations covering VM lifecycle, memory mapping, vCPU management, register access, and interrupt control

This is the complete set of bindings we need to build a functional hypervisor. Apple's API is small by design - the framework handles the low-level EL2 management, and the rest is up to us.

But calling these raw functions is painful and dangerous. Every call is `unsafe`. Every return value must be manually checked. Every resource must be manually cleaned up. Forget to call `hv_vm_destroy` and you cannot create another VM in the same process. Forget to check a return value and you silently corrupt state.

In the next post, we build safe Rust abstractions on top of these bindings: RAII structs that clean up automatically, typed register access that catches errors at compile time, and a proper error type that makes failures actionable. Then we will create our first vCPU, load two instructions into guest memory, run them, and read the result back.
