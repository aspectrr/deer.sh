## The Problem with Instruction Arrays

Up to this point in the series, every guest program we have run has been hand-assembled. We wrote ARM64 instructions as `u32` literals, stuffed them into a byte array, copied that array into guest memory, and pointed the program counter at the beginning. This worked fine for "store 42 to X0 and halt" and even for a small UART test that printed a character. But it does not scale.

Consider what we would need to do to run anything non-trivial. A bare-metal Rust program that prints "Hello, world!" compiles to hundreds of instructions. A Linux kernel is millions. You cannot write millions of `u32` values by hand and maintain your sanity. Even if you could, you would lose all the benefits of a real compiler: optimizations, proper register allocation, linking, constant pools, debug info. Writing machine code by hand is a party trick, not an engineering practice.

We need to load real compiled binaries into our VM. This means understanding binary formats - specifically ELF and flat images - and writing loaders that parse them, extract the loadable segments, and place the bytes at the correct guest physical addresses. Once we have a loader, we can write guest programs in Rust (or C, or anything that targets aarch64), compile them normally, and run them in our hypervisor.

This is a turning point in the series. After this post, we stop thinking about individual instructions and start thinking about programs.

## Binary Formats

Two binary formats matter for our hypervisor: ELF and flat (raw) images. They serve different purposes and we will support both.

### ELF: Executable and Linkable Format

ELF is the standard binary format on Linux, BSD, and most Unix-like systems. Every binary you have ever run on a Linux box - `/bin/ls`, the kernel itself (when compiled as `vmlinux`), shared libraries - is an ELF file. When you compile a bare-metal Rust program targeting `aarch64-unknown-none`, the output is also ELF.

An ELF file is structured as follows:

- **ELF Header**: magic number (`\x7fELF`), architecture (aarch64), entry point address, offsets to program headers and section headers.
- **Program Headers**: an array of entries that describe **segments** - contiguous regions of the file that should be loaded into memory. Each program header specifies the file offset, virtual address, physical address, file size, and memory size of one segment.
- **Section Headers**: an array of entries that describe **sections** - `.text`, `.data`, `.rodata`, `.bss`, debug info, symbol tables. Sections are a finer-grained view used by linkers and debuggers, but the OS loader only looks at program headers.
- **The actual data**: the bytes of the code and data segments.

For loading a binary into a VM, we only care about the program headers. Specifically, we look for headers with type `PT_LOAD` - these are the segments that need to be loaded into memory. A typical bare-metal binary has two or three `PT_LOAD` segments:

1. `.text` and `.rodata` - code and read-only data, loaded at the program's base address.
2. `.data` - initialized read-write data, loaded at a higher address.
3. `.bss` - zero-initialized data, which has a memory size larger than its file size (the extra bytes are zeroed).

The difference between `p_filesz` and `p_memsz` matters. `p_filesz` tells us how many bytes to read from the file. `p_memsz` tells us how much memory the segment occupies. When `p_memsz > p_filesz`, the remaining bytes must be zeroed - this is how `.bss` works without wasting space in the binary file.

The entry point in the ELF header tells us where to set the program counter after loading. For a bare-metal program, this is typically the `_start` symbol.

We use the `goblin` crate to parse ELF files. Goblin is a pure-Rust binary parsing library - it does not execute anything, it just reads headers and gives us the data we need. Add it to your `Cargo.toml`:

```toml
[dependencies]
goblin = "0.9"
```

### Flat/Raw Images

A flat image is exactly what it sounds like: raw bytes, no headers, no metadata. You load the entire file into memory at a predetermined address and set the program counter to that address (or some known offset from it).

The most important flat image for us is the Linux kernel `Image` file. When you build a Linux kernel for ARM64, the output in `arch/arm64/boot/Image` is a flat binary with a small 64-byte header. It is not ELF - it is designed to be loaded at a fixed address and jumped to directly. The header contains a magic number, the image size, and flags, but the loading process is trivial: read the file, copy it to the load address, done.

Flat images are simpler than ELF but less flexible. There is no segment information, no entry point metadata, no BSS handling. The loader has to know where to put the image and where execution starts. For Linux, these are specified by the ARM64 boot protocol.

## The Loader Module

Let us build `src/loader.rs`. We will define two functions: `load_elf` for ELF binaries and `load_flat` for raw images. Both return a `LoadResult` containing the entry point and a list of segments to write into guest memory.

### Data Structures

```rust
pub struct LoadedSegment {
    pub guest_addr: u64,
    pub data: Vec<u8>,
}

pub struct LoadResult {
    pub entry_point: u64,
    pub segments: Vec<LoadedSegment>,
}
```

A `LoadedSegment` pairs a guest physical address with the bytes that go there. The VMM iterates over these segments and writes each one into guest memory. A `LoadResult` bundles the entry point (where to set `PC`) with all the segments.

This design keeps the loader decoupled from the memory subsystem. The loader does not need to know how `GuestMemory` works - it just produces addresses and byte vectors. The caller handles the actual writes.

### ELF Loader

```rust
use anyhow::{Context, Result};
use goblin::elf::{self, program_header};

pub fn load_elf(data: &[u8]) -> Result<LoadResult> {
    let elf = elf::Elf::parse(data).context("failed to parse ELF binary")?;

    // Verify it's an AArch64 binary
    if elf.header.e_machine != goblin::elf::header::EM_AARCH64 {
        anyhow::bail!(
            "expected AArch64 ELF binary, got machine type {}",
            elf.header.e_machine
        );
    }

    let mut segments = Vec::new();

    for phdr in &elf.program_headers {
        if phdr.p_type != program_header::PT_LOAD {
            continue;
        }

        let offset = phdr.p_offset as usize;
        let filesz = phdr.p_filesz as usize;
        let memsz = phdr.p_memsz as usize;

        // Allocate the full memory region (zeroed)
        let mut segment_data = vec![0u8; memsz];

        // Copy the file data into the beginning
        if filesz > 0 {
            segment_data[..filesz]
                .copy_from_slice(&data[offset..offset + filesz]);
        }
        // Remaining bytes (filesz..memsz) are already zero - this handles .bss

        segments.push(LoadedSegment {
            guest_addr: phdr.p_vaddr,
            data: segment_data,
        });
    }

    if segments.is_empty() {
        anyhow::bail!("ELF binary has no PT_LOAD segments");
    }

    Ok(LoadResult {
        entry_point: elf.entry,
        segments,
    })
}
```

Walk through what this does:

1. Parse the ELF binary using goblin. If the data is not valid ELF, this fails immediately with a descriptive error.
2. Check the machine type. We only handle AArch64. If someone passes in an x86 binary, we bail out rather than producing nonsensical results.
3. Iterate over program headers, filtering for `PT_LOAD` segments. Each PT_LOAD segment needs to be placed in guest memory.
4. For each segment, allocate a zeroed byte vector of size `p_memsz`. Copy `p_filesz` bytes from the file data into the beginning. The tail (if `p_memsz > p_filesz`) remains zero, correctly initializing BSS.
5. Use `p_vaddr` as the guest address. For position-dependent bare-metal binaries, the virtual address in the ELF is the physical address where the segment should be loaded.
6. Return the entry point from the ELF header along with all loaded segments.

Note the use of `p_vaddr` rather than `p_paddr`. For bare-metal programs linked to run at a specific address, these are typically the same. The ELF spec says `p_paddr` is the physical address and `p_vaddr` is the virtual address, but in practice for our use case they are identical because the guest does not have an MMU enabled at boot time.

### Flat Image Loader

```rust
pub fn load_flat(data: &[u8], load_addr: u64) -> LoadResult {
    LoadResult {
        entry_point: load_addr,
        segments: vec![LoadedSegment {
            guest_addr: load_addr,
            data: data.to_vec(),
        }],
    }
}
```

This is almost comically simple, which is the entire point of flat images. No parsing, no headers, no segment iteration. The caller provides the raw bytes and the address. The loader copies the data into a single segment at that address and sets the entry point to the same address.

For the Linux kernel `Image` format, the entry point is actually at a small offset from the load address (the 64-byte header contains a branch instruction that jumps past itself), but the convention is to jump to the base of the image and let that branch instruction do its job. We will handle the Linux-specific details when we get to kernel booting in a later post.

### Integrating with GuestMemory

We need a method on `GuestMemory` to write arbitrary bytes at a guest physical address. If you followed the memory management post, you already have something like this:

```rust
impl GuestMemory {
    pub fn write_at(&mut self, guest_addr: u64, data: &[u8]) {
        let offset = (guest_addr - self.base_addr) as usize;
        let slice = &mut self.host_mapping[offset..offset + data.len()];
        slice.copy_from_slice(data);
    }
}
```

The `write_at` method calculates the offset within the host-side memory mapping, then copies the data in. This is a straight `memcpy` - the bytes go directly into the memory region that is mapped into the guest via `hv_vm_map`. No page table manipulation, no system calls, just a pointer write. After this call, the guest CPU can read those bytes by accessing the corresponding guest physical address.

## Writing a Bare-Metal Rust Guest

Now that we can load ELF binaries, let us write a real guest program. We will create a separate `guest/` crate in the project that compiles to a bare-metal AArch64 ELF binary. This guest will print a message to the UART and halt.

### Project Structure

Create the guest crate:

```bash
mkdir -p guest/.cargo
```

### Cargo.toml

```toml
[package]
name = "guest"
version = "0.1.0"
edition = "2021"

[profile.dev]
panic = "abort"

[profile.release]
panic = "abort"
```

Both profiles set `panic = "abort"` because we have no unwinding runtime. In a `no_std` bare-metal environment, there is no stack unwinding, no catch_unwind, no panic handler that can recover. Abort is the only sane option.

### Build Configuration

Create `guest/.cargo/config.toml`:

```toml
[build]
target = "aarch64-unknown-none"

[target.aarch64-unknown-none]
rustflags = ["-C", "link-arg=-Tguest/link.ld"]
```

The `aarch64-unknown-none` target produces bare-metal AArch64 binaries. No standard library, no OS, no runtime. Just your code and whatever runtime support you provide.

The `rustflags` line tells the linker to use our custom linker script. Without this, the linker would use its default layout, which places code at address 0 or some other address that does not match our guest memory map.

### Linker Script

Create `guest/link.ld`:

```
ENTRY(_start)
SECTIONS {
    . = 0x80000000;
    .text : { *(.text._start) *(.text*) }
    .rodata : { *(.rodata*) }
    .data : { *(.data*) }
    .bss : { *(.bss*) *(COMMON) }
    . = ALIGN(4096);
    _stack_top = . + 0x4000;
}
```

This linker script does several important things:

- `ENTRY(_start)` - sets the ELF entry point to `_start`. The ELF header's `e_entry` field will contain the address of `_start`.
- `. = 0x80000000` - places the beginning of the binary at guest physical address `0x80000000`. This must match where we load the binary in guest memory. The address `0x80000000` (2GB) is a common convention for ARM64 guest kernels.
- `.text` section comes first, with `*(.text._start)` before other text sections. This ensures `_start` is at the very beginning of the binary.
- `.rodata`, `.data`, `.bss` follow in the standard order.
- `_stack_top` is defined 16KB (0x4000 bytes) past the end of BSS, page-aligned. The guest will use this as its initial stack pointer.

The ordering of `.text._start` matters. Rust does not guarantee function ordering within a section, so by placing `_start` in its own subsection (via `#[link_section]` or the `#[no_mangle]` + symbol name convention), we ensure it lands at the entry point.

### Guest Source Code

Create `guest/src/main.rs`:

```rust
#![no_std]
#![no_main]

use core::panic::PanicInfo;
use core::ptr;

const UART_BASE: *mut u8 = 0x0900_0000 as *mut u8;

fn uart_putc(c: u8) {
    unsafe {
        ptr::write_volatile(UART_BASE, c);
    }
}

fn uart_puts(s: &str) {
    for b in s.bytes() {
        uart_putc(b);
    }
}

#[no_mangle]
pub extern "C" fn _start() -> ! {
    // Set up stack pointer
    unsafe {
        core::arch::asm!(
            "ldr x0, =_stack_top",
            "mov sp, x0",
        );
    }

    uart_puts("Hello from bare-metal Rust!\n");

    // Halt via HVC
    unsafe {
        core::arch::asm!("hvc #0");
    }

    loop {}
}

#[panic_handler]
fn panic(_info: &PanicInfo) -> ! {
    uart_puts("PANIC!\n");
    loop {}
}
```

Let us walk through every piece of this:

**`#![no_std]`** - No standard library. We do not have an allocator, file system, or OS. Only `core` is available: basic types, traits, intrinsics, and inline assembly.

**`#![no_main]`** - No Rust runtime entry point. Normally, Rust inserts a shim that calls `main()` after setting up the runtime (panic hooks, thread locals, etc.). We do not have a runtime, so we skip all of that and define our own `_start`.

**`UART_BASE`** - The PL011 UART's data register is mapped at `0x09000000` in our VM's memory map. Writing a byte to this address sends it to the UART, which our VMM captures and prints to the host terminal. This address must match the MMIO region we configured in the VMM's exit handler.

**`uart_putc`** - Writes a single byte to the UART using `write_volatile`. The volatile is essential: without it, the compiler might optimize away repeated writes to the same address, or reorder them with other memory accesses. MMIO registers are not normal memory - every write has a side effect.

**`uart_puts`** - Iterates over the bytes of a string and sends each one to the UART. Simple, no buffering, no newline conversion.

**`_start`** - The entry point. Marked `#[no_mangle]` so the linker can find it by name, and `extern "C"` for the C calling convention (though it does not matter here since nobody calls this function - the CPU jumps to it directly).

The first thing `_start` does is set up the stack pointer. ARM64 uses a full-descending stack: SP points to the last used slot and grows downward. We load the address of `_stack_top` (defined in the linker script) into X0, then move it to SP. Without this, we have no stack and cannot call functions, use local variables, or do much of anything.

After the stack is set up, we print our message. Then we execute `hvc #0` to trap into the hypervisor. Our VMM's exit handler will see an HVC exception, and we can use that as a clean shutdown signal.

The `loop {}` after the HVC is a safety net. If the VMM does not halt execution on HVC, the CPU has somewhere safe to spin rather than running off into uninitialized memory.

**`panic_handler`** - Required by `#![no_std]`. If Rust panics (array bounds check, unwrap on None, explicit panic!), it calls this handler. We print "PANIC!" and spin forever. In a real system you might print the panic message, but `PanicInfo` formatting requires an allocator, which we do not have.

### A Note on the Stack Setup

The inline assembly for stack setup is admittedly awkward. The `ldr x0, =_stack_top` pseudo-instruction generates a PC-relative load from a literal pool, which the assembler creates automatically. This works but generates a data access during the instruction stream, which can be surprising.

An alternative approach is to define a small assembly entry point in a separate `.S` file that sets up the stack and then calls a Rust function. This is cleaner but introduces another file and build complexity. For our purposes, the inline assembly approach works fine.

## Loading Into the VMM

With the loader module and the guest binary ready, we update the VMM's main function to load and run the guest:

```rust
use std::fs;

mod loader;
use loader::{load_elf, load_flat};

fn main() -> anyhow::Result<()> {
    // Read the guest binary
    let binary = fs::read("guest/target/aarch64-unknown-none/release/guest")
        .context("failed to read guest binary - did you build it?")?;

    // Parse as ELF
    let loaded = load_elf(&binary)?;

    println!(
        "Loaded {} segments, entry point: {:#x}",
        loaded.segments.len(),
        loaded.entry_point
    );

    // Create VM and memory
    let vm = Vm::new()?;
    let mut memory = GuestMemory::new(64 * 1024 * 1024)?; // 64MB

    // Write segments into guest memory
    for segment in &loaded.segments {
        println!(
            "  Loading segment: {:#x} ({} bytes)",
            segment.guest_addr,
            segment.data.len()
        );
        memory.write_at(segment.guest_addr, &segment.data);
    }

    // Map all guest memory into the VM
    memory.map_all(&vm)?;

    // Create and configure vCPU
    let mut vcpu = Vcpu::new()?;
    vcpu.set_reg(HV_REG_PC, loaded.entry_point)?;
    vcpu.set_reg(HV_REG_CPSR, 0x3c4)?; // EL1h, all DAIF bits masked

    // Run the VM exit/entry loop
    loop {
        let exit = vcpu.run()?;
        match exit.reason {
            ExitReason::Exception => {
                let syndrome = exit.syndrome();
                let ec = (syndrome >> 26) & 0x3f;

                match ec {
                    // HVC from AArch64 - guest wants to halt
                    0x16 => {
                        println!("Guest halted via HVC");
                        break;
                    }
                    // Data abort - MMIO
                    0x24 => {
                        handle_mmio(&mut vcpu, &exit)?;
                    }
                    _ => {
                        println!("Unhandled exception class: {:#x}", ec);
                        break;
                    }
                }
            }
            _ => {
                println!("Unexpected exit reason: {:?}", exit.reason);
                break;
            }
        }
    }

    Ok(())
}
```

The structure is familiar from previous posts. The difference is where the code comes from: instead of a hand-crafted byte array, we read a real ELF binary from disk, parse its program headers, and load each segment into the correct guest address. The entry point comes from the ELF header rather than being hardcoded.

The CPSR value `0x3c4` deserves explanation. In binary, this is:

```
0011 1100 0100
```

Breaking it down:

- Bits [3:0] = `0100` - exception level 1, using SP_EL1 (the "h" in EL1h)
- Bit [6] = `1` - FIQ masked
- Bit [7] = `1` - IRQ masked
- Bit [8] = `1` - SError masked
- Bit [9] = `1` - Debug masked

We start with all interrupts masked because we do not have an interrupt controller set up yet. The guest can unmask them later if needed (and will need to once we add GIC and timer support).

## Building and Running

Build the guest first, then the VMM:

```bash
# Build the guest (from project root)
cd guest && cargo build --release

# Verify the binary is AArch64 ELF
file target/aarch64-unknown-none/release/guest
# output: ELF 64-bit LSB executable, ARM aarch64, version 1 (SYSV), statically linked, ...

# Build the VMM
cd ..
cargo build --release

# Run it
./target/release/apple-hypervisor
```

Expected output:

```terminal
Loaded 1 segments, entry point: 0x80000000
  Loading segment: 0x80000000 (16512 bytes)
Hello from bare-metal Rust!
Guest halted via HVC
```

That "Hello from bare-metal Rust!" came from a real Rust program running inside our hypervisor. Not hand-assembled instruction bytes. Not a toy. A real `no_std` binary, compiled by rustc, linked by the system linker, loaded by our ELF parser, executed on a virtual CPU that we created. The chain from source code to execution is complete.

If you do not see the output, check these common issues:

- **Linker script not found**: make sure `guest/link.ld` exists and the path in `.cargo/config.toml` is correct relative to where you run `cargo build`.
- **Wrong UART address**: the UART_BASE in the guest must match the MMIO address range your VMM handles.
- **Memory too small**: 64MB is plenty for this guest, but if you change the linker script base address, make sure it falls within the guest memory region.
- **Segment outside memory**: if the ELF has a segment at an address that is outside the mapped guest memory, `write_at` will panic. Check your linker script.

## Testing

The loader module is pure logic with no hardware dependencies, so it is straightforward to unit test.

### Unit Tests

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_load_flat() {
        let data = vec![0xAA, 0xBB, 0xCC, 0xDD];
        let result = load_flat(&data, 0x80000000);

        assert_eq!(result.entry_point, 0x80000000);
        assert_eq!(result.segments.len(), 1);
        assert_eq!(result.segments[0].guest_addr, 0x80000000);
        assert_eq!(result.segments[0].data, data);
    }

    #[test]
    fn test_load_flat_preserves_load_addr() {
        let data = vec![0x00; 256];
        let result = load_flat(&data, 0x40000000);

        assert_eq!(result.entry_point, 0x40000000);
        assert_eq!(result.segments[0].guest_addr, 0x40000000);
    }

    #[test]
    fn test_load_elf_valid_binary() {
        // Build the guest binary first: cd guest && cargo build --release
        let binary = std::fs::read(
            "guest/target/aarch64-unknown-none/release/guest"
        );

        // Skip test if guest binary hasn't been built
        let binary = match binary {
            Ok(b) => b,
            Err(_) => {
                eprintln!("Skipping: guest binary not built");
                return;
            }
        };

        let result = load_elf(&binary).expect("failed to load ELF");

        // Entry point should be at our linker script base
        assert_eq!(result.entry_point, 0x80000000);

        // Should have at least one loadable segment
        assert!(!result.segments.is_empty());

        // First segment should be at the base address
        assert_eq!(result.segments[0].guest_addr, 0x80000000);

        // Segment should contain actual data
        assert!(!result.segments[0].data.is_empty());
    }

    #[test]
    fn test_load_elf_rejects_garbage() {
        let garbage = vec![0x00, 0x01, 0x02, 0x03];
        let result = load_elf(&garbage);
        assert!(result.is_err());
    }

    #[test]
    fn test_load_elf_bss_zeroed() {
        // Construct a minimal ELF with p_memsz > p_filesz
        // to verify BSS zeroing behavior.
        //
        // In practice we test this with the real guest binary:
        // if it has a .bss section, p_memsz will exceed p_filesz
        // and the extra bytes should be zero.
        let binary = match std::fs::read(
            "guest/target/aarch64-unknown-none/release/guest"
        ) {
            Ok(b) => b,
            Err(_) => return, // Skip if not built
        };

        let result = load_elf(&binary).expect("failed to load ELF");

        for segment in &result.segments {
            // Verify segment data is not empty
            assert!(!segment.data.is_empty());
            // The segment was allocated with vec![0u8; memsz],
            // so any bytes beyond filesz are guaranteed zero.
            // We trust the implementation here - a more thorough
            // test would compare against the raw ELF data.
        }
    }
}
```

### Integration Test

The integration test builds the guest, runs it through the VMM, and checks the UART output:

```rust
#[test]
fn test_guest_hello_world() {
    // This test requires the guest binary to be pre-built.
    // Run: cd guest && cargo build --release
    let binary = match std::fs::read(
        "guest/target/aarch64-unknown-none/release/guest"
    ) {
        Ok(b) => b,
        Err(_) => {
            eprintln!("Skipping integration test: guest binary not built");
            return;
        }
    };

    let loaded = load_elf(&binary).expect("ELF load failed");
    let vm = Vm::new().expect("VM creation failed");
    let mut memory = GuestMemory::new(64 * 1024 * 1024).unwrap();

    for seg in &loaded.segments {
        memory.write_at(seg.guest_addr, &seg.data);
    }
    memory.map_all(&vm).unwrap();

    let mut vcpu = Vcpu::new().unwrap();
    vcpu.set_reg(HV_REG_PC, loaded.entry_point).unwrap();
    vcpu.set_reg(HV_REG_CPSR, 0x3c4).unwrap();

    let mut uart_output = Vec::new();

    loop {
        let exit = vcpu.run().unwrap();
        match exit.reason {
            ExitReason::Exception => {
                let ec = (exit.syndrome() >> 26) & 0x3f;
                match ec {
                    0x16 => break, // HVC - halt
                    0x24 => {
                        // MMIO - capture UART writes
                        if let Some(byte) = handle_mmio_capture(&mut vcpu, &exit) {
                            uart_output.push(byte);
                        }
                    }
                    _ => panic!("unexpected exception: {:#x}", ec),
                }
            }
            _ => panic!("unexpected exit: {:?}", exit.reason),
        }
    }

    let output = String::from_utf8(uart_output).unwrap();
    assert_eq!(output, "Hello from bare-metal Rust!\n");
}
```

The test captures every byte written to the UART address and accumulates them into a string. After the guest halts via HVC, it asserts that the string matches exactly. This is an end-to-end test of the entire pipeline: ELF parsing, memory loading, CPU configuration, exit handling, and MMIO emulation.

## What Changed

Before this post, our hypervisor could run instruction arrays. Now it can run compiled binaries. The difference seems small but it is the difference between a demo and a tool. We can now:

- Write guest programs in Rust with full compiler support
- Use any language that targets `aarch64-unknown-none` (C, C++, Zig, etc.)
- Load the Linux kernel (as a flat image, once we add DTB support)
- Add structured testing of the VMM with real compiled test binaries

The `guest/` crate also gives us a development workflow: change the guest code, `cargo build --release`, run the VMM. The iteration cycle is fast and the feedback is immediate - you see the UART output right in your terminal.

## What is Next

We can load binaries and run bare-metal Rust programs. But Linux is not a bare-metal Rust program. Before we can boot a kernel, we need to solve a fundamental problem: the kernel has no idea what hardware it is running on. On x86, there are standardized discovery mechanisms (ACPI, PCI). On ARM, there is nothing - the bootloader must hand the kernel a data structure describing every piece of hardware in the system. That data structure is called a **Device Tree**, and building one is the subject of the next post.
