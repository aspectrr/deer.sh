## The Last Mile

The kernel boots. It initializes memory, calibrates timers, brings up the scheduler, and then panics: "VFS: Unable to mount root fs on unknown-block(0,0)." The kernel did everything it knows how to do and is now waiting for a filesystem to mount as root. Without one, there is no `/sbin/init` to execute, no userspace to start, nothing to do except print a stack trace and halt.

There are two ways to give a VM a root filesystem. The first is to emulate a block device - virtio-blk or an IDE controller - and attach a disk image containing a filesystem. This is what QEMU and Firecracker do. It is the production approach. It is also hundreds of lines of device emulation code: virtio transport, descriptor rings, interrupt signaling, block I/O threading.

The second way is an initramfs. This is a compressed cpio archive that gets baked into the kernel or loaded alongside it. The kernel unpacks it into a tmpfs in memory and uses that as the root filesystem. No block device emulation required. The kernel does all the work - it already has cpio parsing and tmpfs built in. All we have to do is put the archive in guest memory and tell the kernel where it is via the device tree.

We use the initramfs approach. It lets us get to an interactive shell without implementing any new device emulation. Everything we need is already built.

This post covers three things: building an initramfs with BusyBox, loading it into the VM, and wiring up keyboard input so the shell is interactive. By the end, you will have a complete virtual machine - a real Linux system running inside a hypervisor you wrote from scratch, where you can type commands and see output.

## Building an Initramfs

An initramfs is a gzip-compressed cpio archive containing a minimal filesystem hierarchy. When the kernel boots with an initramfs, it unpacks the archive into a tmpfs mounted at `/`, then executes `/init` as PID 1. That init process becomes the ancestor of every other process in the system.

For a minimal interactive system, we need exactly one binary: BusyBox. BusyBox is a single executable that implements hundreds of standard Unix utilities - `sh`, `ls`, `cat`, `echo`, `mount`, `mkdir`, and many more. When invoked under different names (via symlinks), it behaves as the corresponding utility. A statically linked BusyBox binary and a handful of symlinks give you a functional Unix environment in under 2 MB.

### Getting a Static BusyBox

You need an ARM64 (aarch64) statically linked BusyBox binary. There are several ways to get one:

**Option 1: Alpine Linux's package** - Alpine builds static BusyBox for aarch64. Download the `busybox-static` package from their package repository and extract the binary.

**Option 2: Cross-compile from source** - This gives you the most control:

```bash
# On a Linux host with cross-compilation tools
wget https://busybox.net/downloads/busybox-1.36.1.tar.bz2
tar xf busybox-1.36.1.tar.bz2
cd busybox-1.36.1

make ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- defconfig
# Enable static linking
sed -i 's/# CONFIG_STATIC is not set/CONFIG_STATIC=y/' .config
make ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- -j$(nproc)

# Result: busybox (statically linked aarch64 binary)
file busybox
# busybox: ELF 64-bit LSB executable, ARM aarch64, statically linked...
```

**Option 3: Pre-built binaries** - Various projects distribute pre-built static BusyBox binaries for aarch64. Verify them with `file` to confirm the architecture.

### Creating the Filesystem Structure

With the BusyBox binary in hand, build the initramfs directory tree:

```bash
# Create the directory structure
mkdir -p initramfs/{bin,sbin,etc,proc,sys,dev,tmp,usr/bin,usr/sbin}

# Copy BusyBox
cp busybox-aarch64 initramfs/bin/busybox
chmod +x initramfs/bin/busybox

# Create symlinks for common commands
# BusyBox checks argv[0] to determine which utility to run
cd initramfs/bin
for cmd in sh ash ls cat echo grep mount umount mkdir rm cp mv \
           ps kill sleep date uname hostname dmesg vi head tail \
           wc sort uniq tr sed awk cut id whoami pwd df du free \
           ifconfig ping wget tar gzip gunzip; do
    ln -s busybox "$cmd"
done
cd ../sbin
for cmd in init halt poweroff reboot; do
    ln -s ../bin/busybox "$cmd"
done
cd ../..
```

### The Init Script

The init script is the first userspace program the kernel executes. It needs to set up the essential virtual filesystems - `/proc`, `/sys`, and `/dev` - and then start a shell:

```bash
cat > initramfs/init << 'INIT'
#!/bin/sh

# Mount essential virtual filesystems
mount -t proc proc /proc
mount -t sysfs sysfs /sys
mount -t devtmpfs devtmpfs /dev 2>/dev/null

# Create essential device nodes if devtmpfs is not available
if [ ! -c /dev/console ]; then
    mknod /dev/console c 5 1
    mknod /dev/null c 1 3
    mknod /dev/zero c 1 5
    mknod /dev/tty c 5 0
fi

# Set hostname
hostname hypervisor-vm

# Print a banner
echo ""
echo "================================================"
echo "  Welcome to the hypervisor VM"
echo "  Built from scratch on Apple Silicon"
echo "================================================"
echo ""
echo "Type 'help' for BusyBox command list"
echo ""

# Start an interactive shell
# The -l flag makes it a login shell
exec /bin/sh -l
INIT

chmod +x initramfs/init
```

A few notes on this init script. The `mount -t proc proc /proc` command mounts the proc filesystem, which gives tools like `ps` and `cat /proc/cpuinfo` access to kernel data structures. The `mount -t sysfs sysfs /sys` command mounts sysfs, which exposes device and driver information. The `mount -t devtmpfs devtmpfs /dev` command asks the kernel to populate `/dev` with device nodes automatically. The `2>/dev/null` suppresses an error message if the kernel was not built with devtmpfs support.

The `exec` on the last line replaces the init process with the shell. This means the shell becomes PID 1. If the shell exits, the kernel panics because PID 1 died. For a production init system you would use something like `runit` or `systemd` that restarts services, but for an interactive hypervisor demo, a bare shell is perfect.

### Packing the Archive

The cpio format used by the kernel is "newc" (new ASCII). The archive is created by piping a file listing into `cpio` and then compressing with gzip:

```bash
cd initramfs
find . -print0 | cpio --null -H newc -o | gzip -9 > ../initramfs.cpio.gz
cd ..

# Check the result
ls -lh initramfs.cpio.gz
# Should be 1-2 MB for a BusyBox-only initramfs
```

The `-print0` and `--null` flags handle filenames with spaces correctly (not that we have any, but it is good practice). The `-H newc` flag selects the new ASCII cpio format. The `-o` flag means "create archive" (output mode). The `gzip -9` gives maximum compression.

The resulting file is what we load into the VM. The kernel knows how to decompress gzip and parse cpio - both are built into the kernel, not provided by userspace.

## Loading the Initramfs

Loading the initramfs into the VM is straightforward: read the file, write it into guest memory at an address that does not overlap the kernel, and tell the kernel where it is via the device tree's `/chosen` node.

### Memory Layout

The guest memory layout with an initramfs looks like this:

```
Guest Physical Memory Layout:

0x40000000  +---------------------------+
            | Kernel Image              |
            | (~30 MB for defconfig)    |
0x42000000  +---------------------------+  (approx kernel end)
            |                           |
            | (unused space)            |
            |                           |
0x44000000  +---------------------------+  initrd_addr (page-aligned)
            | initramfs.cpio.gz         |
            | (~1-2 MB)                 |
0x44200000  +---------------------------+  initrd_end (approx)
            |                           |
            | (free memory for kernel   |
            |  heap, page tables, etc.) |
            |                           |
0x7FE00000  +---------------------------+  dtb_addr (2MB-aligned)
            | Device Tree Blob          |
            | (~4 KB)                   |
0x80000000  +---------------------------+  end of 1 GB RAM
```

The kernel goes at the base of RAM. The initramfs goes after the kernel, aligned to a page boundary. The DTB goes at the top of RAM. Everything in between is free for the kernel to use as heap and page cache.

### The Loading Code

```rust
use anyhow::{bail, Result};
use std::path::Path;

/// Load the kernel and optional initramfs into guest memory.
/// Returns (kernel_entry, dtb_addr).
pub fn load_guest(
    memory: &GuestMemory,
    mem_base: u64,
    mem_size: u64,
    kernel_path: &Path,
    initrd_path: Option<&Path>,
    cmdline: &str,
    uart_base: u64,
) -> Result<(u64, u64)> {
    // Load kernel
    let kernel_data = std::fs::read(kernel_path)?;
    let kernel_result = load_linux_image(&kernel_data)?;

    // Write kernel to guest memory
    for segment in &kernel_result.segments {
        let offset = segment.guest_addr - mem_base;
        memory.write_at(offset, &segment.data);
    }
    log::info!(
        "Kernel loaded: {} bytes at 0x{:x}",
        kernel_data.len(),
        kernel_result.entry_point
    );

    // Load initramfs if provided
    let (initrd_start, initrd_end) = if let Some(path) = initrd_path {
        let initrd_data = std::fs::read(path)?;
        if initrd_data.is_empty() {
            bail!("Initramfs file is empty");
        }

        // Place initramfs after the kernel, page-aligned
        let kernel_end = kernel_result.entry_point + kernel_data.len() as u64;
        let initrd_addr = (kernel_end + 0xFFF) & !0xFFF; // 4K-aligned
        let initrd_end_addr = initrd_addr + initrd_data.len() as u64;

        // Sanity check: initramfs must fit in guest memory
        if initrd_end_addr >= mem_base + mem_size {
            bail!(
                "Initramfs does not fit in guest memory: end 0x{:x} >= 0x{:x}",
                initrd_end_addr,
                mem_base + mem_size,
            );
        }

        let offset = initrd_addr - mem_base;
        memory.write_at(offset, &initrd_data);
        log::info!(
            "Initramfs loaded: {} bytes at 0x{:x}-0x{:x}",
            initrd_data.len(),
            initrd_addr,
            initrd_end_addr,
        );

        (Some(initrd_addr), Some(initrd_end_addr))
    } else {
        (None, None)
    };

    // Generate device tree
    let dtb = create_fdt(
        mem_base,
        mem_size,
        1, // CPU count
        cmdline,
        uart_base,
        initrd_start,
        initrd_end,
    )?;

    // Place DTB at top of RAM, 2MB-aligned
    let dtb_addr = (mem_base + mem_size - 0x20_0000) & !0x1F_FFFF;
    let dtb_offset = dtb_addr - mem_base;
    memory.write_at(dtb_offset, &dtb);
    log::info!("DTB placed at 0x{:x} ({} bytes)", dtb_addr, dtb.len());

    Ok((kernel_result.entry_point, dtb_addr))
}
```

### Device Tree Updates for Initramfs

The kernel needs to know the initramfs's location in memory. This is communicated through the `/chosen` node in the device tree, using two properties: `linux,initrd-start` and `linux,initrd-end`. Both are 64-bit physical addresses.

Here is the updated device tree generation function, showing the `/chosen` node:

```rust
use vm_fdt::FdtWriter;

pub fn create_fdt(
    mem_base: u64,
    mem_size: u64,
    num_cpus: u32,
    cmdline: &str,
    uart_base: u64,
    initrd_start: Option<u64>,
    initrd_end: Option<u64>,
) -> Result<Vec<u8>> {
    let mut fdt = FdtWriter::new()?;

    // Root node
    let root = fdt.begin_node("")?;
    fdt.property_string("compatible", "linux,dummy-virt")?;
    fdt.property_u32("#address-cells", 2)?;
    fdt.property_u32("#size-cells", 2)?;

    // /chosen node - kernel parameters and initrd location
    let chosen = fdt.begin_node("chosen")?;
    fdt.property_string("bootargs", cmdline)?;
    fdt.property_string("stdout-path", "/pl011@9000000")?;

    if let (Some(start), Some(end)) = (initrd_start, initrd_end) {
        fdt.property_u64("linux,initrd-start", start)?;
        fdt.property_u64("linux,initrd-end", end)?;
        log::debug!(
            "DTB /chosen: initrd at 0x{:x}-0x{:x} ({} bytes)",
            start, end, end - start
        );
    }

    fdt.end_node(chosen)?;

    // /memory node
    let memory = fdt.begin_node(&format!("memory@{:x}", mem_base))?;
    fdt.property_string("device_type", "memory")?;
    fdt.property_array_u64("reg", &[mem_base, mem_size])?;
    fdt.end_node(memory)?;

    // /cpus node
    let cpus = fdt.begin_node("cpus")?;
    fdt.property_u32("#address-cells", 1)?;
    fdt.property_u32("#size-cells", 0)?;

    for i in 0..num_cpus {
        let cpu = fdt.begin_node(&format!("cpu@{}", i))?;
        fdt.property_string("compatible", "arm,arm-v8")?;
        fdt.property_string("device_type", "cpu")?;
        fdt.property_u32("reg", i)?;
        fdt.property_string("enable-method", "psci")?;
        fdt.end_node(cpu)?;
    }

    fdt.end_node(cpus)?;

    // /psci node - tells Linux how to call PSCI
    let psci = fdt.begin_node("psci")?;
    fdt.property_string("compatible", "arm,psci-0.2")?;
    fdt.property_string("method", "hvc")?;
    fdt.end_node(psci)?;

    // /timer node - ARM generic timer
    let timer = fdt.begin_node("timer")?;
    fdt.property_string("compatible", "arm,armv8-timer")?;
    fdt.property_null("always-on")?;
    // Interrupt specifiers: type (1=PPI), number, flags
    // Secure phys timer PPI 13, Non-secure phys timer PPI 14,
    // Virtual timer PPI 11, Hypervisor timer PPI 10
    fdt.property_array_u32("interrupts", &[
        1, 13, 0xf04,  // Secure physical timer
        1, 14, 0xf04,  // Non-secure physical timer
        1, 11, 0xf04,  // Virtual timer
        1, 10, 0xf04,  // Hypervisor timer
    ])?;
    fdt.end_node(timer)?;

    // /pl011 UART node
    let uart = fdt.begin_node(&format!("pl011@{:x}", uart_base))?;
    fdt.property_string("compatible", "arm,pl011\0arm,primecell")?;
    fdt.property_array_u64("reg", &[uart_base, 0x1000])?;
    fdt.property_u32("clock-names-placeholder", 0)?;
    fdt.end_node(uart)?;

    fdt.end_node(root)?;

    Ok(fdt.finish()?)
}
```

The two initrd properties are the only difference from the previous post's device tree. The kernel's early boot code reads these properties in `early_init_dt_check_for_initrd()`, which runs before the memory allocator is fully initialized. It reserves the initramfs memory region so nothing overwrites it, then later unpacks the cpio archive into the root tmpfs.

## PL011 RX Support - Reading Input

Up to this point, our PL011 UART has been transmit-only. The guest writes a byte to the Data Register at offset `0x000`, our callback prints it to the host terminal, and that is it. For an interactive shell, we need the reverse direction: the guest needs to read characters that the user types on the host keyboard.

The PL011 specification defines the receive path through two registers:

**DR (Data Register, offset 0x000)**: On read, returns the oldest byte from the receive FIFO. On write, transmits a byte. The same register serves both directions.

**FR (Flag Register, offset 0x018)**: Contains status bits. Bit 4 is RXFE (Receive FIFO Empty): 1 when the FIFO is empty, 0 when data is available. Bit 5 is TXFF (Transmit FIFO Full): always 0 in our emulation since we flush immediately.

The guest's serial driver checks FR bit 4 before reading DR. If RXFE is set, there is nothing to read. If RXFE is clear, the driver reads DR to get the next byte. This polling loop is how PL011 serial input works without interrupts.

Here is the updated PL011 implementation:

```rust
// src/devices/pl011.rs

use crate::mmio::MmioDevice;
use std::collections::VecDeque;
use std::sync::{Arc, Mutex};

/// PL011 UART emulation with transmit and receive support.
///
/// TX: guest writes to DR, callback is invoked with the byte.
/// RX: host pushes bytes into rx_buffer, guest reads them from DR.
pub struct Pl011 {
    /// Callback invoked when the guest transmits a byte
    tx_callback: Box<dyn FnMut(u8) + Send>,

    /// Shared receive buffer. The stdin reader thread pushes bytes in,
    /// the guest pops bytes out via DR reads.
    rx_buffer: Arc<Mutex<VecDeque<u8>>>,

    /// Interrupt mask register (IMSC, offset 0x038)
    /// We track this for completeness but do not generate interrupts.
    imsc: u32,

    /// Line control register (LCR_H, offset 0x02C)
    lcr: u32,

    /// Control register (CR, offset 0x030)
    cr: u32,
}

// PL011 register offsets
const PL011_DR: u64 = 0x000;     // Data Register
const PL011_FR: u64 = 0x018;     // Flag Register
const PL011_IBRD: u64 = 0x024;   // Integer Baud Rate Register
const PL011_FBRD: u64 = 0x028;   // Fractional Baud Rate Register
const PL011_LCR_H: u64 = 0x02C;  // Line Control Register
const PL011_CR: u64 = 0x030;     // Control Register
const PL011_IMSC: u64 = 0x038;   // Interrupt Mask Set/Clear Register
const PL011_ICR: u64 = 0x044;    // Interrupt Clear Register

// Flag register bits
const FR_RXFE: u32 = 1 << 4;     // Receive FIFO Empty
const FR_TXFF: u32 = 1 << 5;     // Transmit FIFO Full

impl Pl011 {
    pub fn new(
        tx_callback: impl FnMut(u8) + Send + 'static,
        rx_buffer: Arc<Mutex<VecDeque<u8>>>,
    ) -> Self {
        Pl011 {
            tx_callback: Box::new(tx_callback),
            rx_buffer,
            imsc: 0,
            lcr: 0,
            cr: 0x0300, // TX and RX enabled by default
        }
    }
}

impl MmioDevice for Pl011 {
    fn read(&mut self, offset: u64, _size: u32) -> u64 {
        match offset {
            PL011_DR => {
                // Read one byte from the receive buffer.
                // If the buffer is empty, return 0 (no data).
                let byte = self.rx_buffer
                    .lock()
                    .unwrap()
                    .pop_front()
                    .unwrap_or(0);
                byte as u64
            }

            PL011_FR => {
                // Flag register.
                // RXFE (bit 4): 1 if receive buffer is empty, 0 if data available
                // TXFF (bit 5): always 0 (transmit never full - we flush immediately)
                let mut flags: u32 = 0;
                if self.rx_buffer.lock().unwrap().is_empty() {
                    flags |= FR_RXFE;
                }
                // TXFF is 0 (not full), so we do not set bit 5
                flags as u64
            }

            PL011_LCR_H => self.lcr as u64,
            PL011_CR => self.cr as u64,
            PL011_IMSC => self.imsc as u64,

            // Identification registers - the kernel reads these to confirm
            // this is a real PL011. The values come from the ARM PL011 TRM.
            0xFE0 => 0x11,  // UARTPeriphID0
            0xFE4 => 0x10,  // UARTPeriphID1
            0xFE8 => 0x14,  // UARTPeriphID2 (revision 1, designer 0x41 = ARM)
            0xFEC => 0x00,  // UARTPeriphID3
            0xFF0 => 0x0D,  // UARTCellID0
            0xFF4 => 0xF0,  // UARTCellID1
            0xFF8 => 0x05,  // UARTCellID2
            0xFFC => 0xB1,  // UARTCellID3

            _ => {
                log::trace!("PL011: unhandled read at offset 0x{:03x}", offset);
                0
            }
        }
    }

    fn write(&mut self, offset: u64, _size: u32, value: u64) {
        match offset {
            PL011_DR => {
                // Transmit a byte
                (self.tx_callback)(value as u8);
            }

            PL011_LCR_H => {
                self.lcr = value as u32;
            }

            PL011_CR => {
                self.cr = value as u32;
            }

            PL011_IMSC => {
                self.imsc = value as u32;
            }

            PL011_ICR => {
                // Interrupt clear - acknowledge but do nothing
                // since we do not generate interrupts
            }

            PL011_IBRD | PL011_FBRD => {
                // Baud rate registers - we ignore these since our virtual
                // UART has no concept of baud rate. Characters are
                // transferred instantaneously.
            }

            _ => {
                log::trace!("PL011: unhandled write at offset 0x{:03x} = 0x{:x}", offset, value);
            }
        }
    }
}
```

The identification registers at the end of the address space (offsets `0xFE0` through `0xFFC`) are important. The Linux PL011 driver reads these during probe to confirm it has found a real PL011 UART. If the values do not match, the driver rejects the device and falls back to earlycon only. With the correct PrimeCell ID values, the full `amba-pl011` driver binds to our device and provides proper tty support.

The `Arc<Mutex<VecDeque<u8>>>` pattern for the receive buffer is the simplest correct approach. The stdin reader thread pushes bytes in, the vCPU thread pops bytes out. The mutex is held only briefly for each operation - push or pop - so contention is minimal. A lock-free ring buffer would be faster but unnecessary for a serial console running at human typing speed.

## Stdin Reader Thread

The host side of keyboard input is a background thread that reads from stdin and feeds bytes into the shared receive buffer:

```rust
use std::collections::VecDeque;
use std::io::Read;
use std::sync::{Arc, Mutex};

/// Start a background thread that reads bytes from host stdin
/// and pushes them into the shared RX buffer.
///
/// The thread runs until stdin is closed (EOF) or an error occurs.
/// It reads one byte at a time to minimize latency.
fn start_stdin_reader(rx_buffer: Arc<Mutex<VecDeque<u8>>>) {
    std::thread::spawn(move || {
        let stdin = std::io::stdin();
        let mut handle = stdin.lock();
        let mut buf = [0u8; 1];

        loop {
            match handle.read_exact(&mut buf) {
                Ok(()) => {
                    rx_buffer.lock().unwrap().push_back(buf[0]);
                }
                Err(_) => {
                    // stdin closed or error - exit the thread
                    break;
                }
            }
        }
    });
}
```

Reading one byte at a time is deliberate. With buffered reads you might read multiple bytes that then sit in a host-side buffer while the guest is not polling the UART. Single-byte reads with `read_exact` mean each byte goes straight into the shared buffer where the guest can find it on the next FR register poll.

This approach has a latency property worth understanding. The guest polls the UART flag register to check for input. Between polls, keystrokes accumulate in the `VecDeque`. When the guest reads FR and sees RXFE=0, it reads DR to get the first byte, then checks FR again. If more bytes are waiting, it reads them too. The guest's poll frequency depends on the serial driver - for a raw terminal, Linux polls quite aggressively.

## Raw Terminal Mode

By default, your terminal operates in "cooked" mode: input is line-buffered (nothing reaches stdin until you press Enter), characters are echoed back to the screen, and special keys like Ctrl+C generate signals. This is terrible for a VM console. You want every keystroke delivered immediately, no local echo (the guest's shell handles echo), and Ctrl+C sent to the guest rather than killing the host process.

The solution is raw terminal mode, configured via the POSIX termios interface:

```rust
use std::os::unix::io::AsRawFd;

/// Switch the host terminal to raw mode.
/// Returns the original termios settings so they can be restored later.
fn enable_raw_mode() -> libc::termios {
    let fd = std::io::stdin().as_raw_fd();
    let mut orig = unsafe { std::mem::zeroed::<libc::termios>() };
    unsafe {
        libc::tcgetattr(fd, &mut orig);
    }

    let mut raw = orig;

    // Input flags: disable special handling
    raw.c_iflag &= !(
        libc::IXON     // Disable software flow control (Ctrl+S/Ctrl+Q)
        | libc::ICRNL  // Do not translate CR to NL
        | libc::BRKINT // Do not generate SIGINT on break
        | libc::INPCK  // Disable parity checking
        | libc::ISTRIP // Do not strip 8th bit
    );

    // Output flags: disable post-processing
    raw.c_oflag &= !libc::OPOST; // Do not translate NL to CR+NL on output

    // Local flags: disable echo and canonical mode
    raw.c_lflag &= !(
        libc::ECHO    // Do not echo input
        | libc::ICANON // Disable canonical mode (line buffering)
        | libc::ISIG   // Do not generate signals for Ctrl+C, Ctrl+Z, etc.
        | libc::IEXTEN // Disable implementation-defined input processing
    );

    // Control characters
    raw.c_cc[libc::VMIN] = 1;  // read() returns after 1 byte
    raw.c_cc[libc::VTIME] = 0; // No timeout

    unsafe {
        libc::tcsetattr(fd, libc::TCSANOW, &raw);
    }

    orig
}

/// Restore the terminal to its original settings.
fn restore_terminal(orig: &libc::termios) {
    let fd = std::io::stdin().as_raw_fd();
    unsafe {
        libc::tcsetattr(fd, libc::TCSANOW, orig);
    }
}
```

Each flag deserves a brief explanation:

`ECHO` controls whether typed characters are displayed. In raw mode we turn it off because the guest's shell will echo characters itself. If we left echo on, every character would appear twice.

`ICANON` controls canonical mode - line buffering. With it on, `read()` blocks until the user presses Enter. With it off, `read()` returns as soon as any byte is available. We want immediate delivery.

`ISIG` controls signal generation. With it on, Ctrl+C sends SIGINT to the host process (killing the hypervisor). With it off, Ctrl+C is delivered as the byte 0x03 to stdin, which our reader thread pushes into the guest's RX buffer. The guest's shell handles it normally.

`OPOST` controls output post-processing. The main effect is translating NL (0x0A) to CR+NL (0x0D 0x0A) on output. We turn this off because the guest manages its own output encoding.

`VMIN = 1, VTIME = 0` means `read()` returns after at least 1 byte, with no timeout. This is the standard configuration for raw mode.

Terminal restoration is critical. If the hypervisor exits (or crashes) without restoring the terminal, the user's shell session is left in raw mode - no echo, no line editing, no Ctrl+C. The fix is `reset` or `stty sane`, but it is much better to restore automatically. We use a scope guard pattern:

```rust
fn main() -> Result<()> {
    let orig_termios = enable_raw_mode();

    // Ensure terminal is restored on exit, even if we panic
    let _guard = scopeguard::guard(orig_termios, |orig| {
        restore_terminal(&orig);
    });

    // ... run the VM ...
}
```

If you do not want to pull in the `scopeguard` crate, a manual approach works too - just make sure every exit path calls `restore_terminal`.

## The Full Boot

Here is the complete main function that orchestrates everything:

```rust
fn main() -> Result<()> {
    env_logger::init();
    let args = Args::parse();

    // Step 1: Enable raw terminal mode
    let orig_termios = enable_raw_mode();
    let _guard = scopeguard::guard(orig_termios, |orig| {
        restore_terminal(&orig);
        println!(); // Ensure prompt starts on a new line
    });

    // Step 2: Create VM
    let vm = vm::Vm::new()?;
    log::info!("VM created");

    // Step 3: Allocate guest memory
    let mem_size = args.memory * 1024 * 1024;
    let mem_base: u64 = 0x4000_0000;
    let memory = memory::GuestMemory::new(mem_size)?;
    vm.map_memory(mem_base, memory.as_ptr(), mem_size)?;
    log::info!("Guest memory: {} MB at 0x{:x}", args.memory, mem_base);

    // Step 4: Set up shared RX buffer for UART input
    let rx_buffer = Arc::new(Mutex::new(VecDeque::<u8>::new()));

    // Step 5: Create UART with TX callback and shared RX buffer
    let uart = Pl011::new(
        |byte| {
            let ch = byte as char;
            print!("{}", ch);
            use std::io::Write;
            std::io::stdout().flush().ok();
        },
        Arc::clone(&rx_buffer),
    );

    // Step 6: Set up MMIO bus
    let uart_base: u64 = 0x0900_0000;
    let mut mmio_bus = mmio::MmioBus::new();
    mmio_bus.register(uart_base, 0x1000, Box::new(uart));

    // Step 7: Load kernel and initramfs
    let kernel_path = std::path::Path::new(&args.kernel);
    let initrd_path = args.initrd.as_ref().map(|p| std::path::Path::new(p.as_str()));

    let (entry_point, dtb_addr) = load_guest(
        &memory,
        mem_base,
        mem_size,
        kernel_path,
        initrd_path,
        &args.cmdline,
        uart_base,
    )?;

    // Step 8: Start stdin reader thread
    start_stdin_reader(Arc::clone(&rx_buffer));

    // Step 9: Create vCPU and set initial state
    let mut vcpu = vm.create_vcpu()?;
    vcpu.set_reg(HV_REG_PC, entry_point)?;
    vcpu.set_reg(HV_REG_X0, dtb_addr)?;
    vcpu.set_reg(HV_REG_X1, 0)?;
    vcpu.set_reg(HV_REG_X2, 0)?;
    vcpu.set_reg(HV_REG_X3, 0)?;
    vcpu.set_reg(HV_REG_CPSR, 0x3C5)?;

    log::info!("vCPU ready: PC=0x{:x}, x0(DTB)=0x{:x}", entry_point, dtb_addr);

    // Step 10: Run the exit loop
    loop {
        vcpu.run()?;
        let exit = vcpu.exit_info();

        match exit.reason {
            HV_EXIT_REASON_EXCEPTION => {
                let syndrome = exit.exception.syndrome;
                let ec_val = ec(syndrome);
                let iss_val = iss(syndrome);

                match ec_val {
                    EC_HVC64 | EC_SMC64 => {
                        let func_id = vcpu.get_reg(HV_REG_X0)? as u32;
                        let is_psci = matches!(func_id >> 24, 0x84 | 0xC4);

                        if is_psci {
                            let (ret, action) = handle_psci(func_id);
                            vcpu.set_reg(HV_REG_X0, ret as u64)?;
                            vcpu.advance_pc(4)?;
                            if matches!(action, PsciAction::Shutdown) {
                                break;
                            }
                        } else {
                            vcpu.set_reg(HV_REG_X0, u64::MAX)?;
                            vcpu.advance_pc(4)?;
                        }
                    }

                    EC_DABT_LOWER => {
                        let ipa = exit.exception.physical_address;
                        mmio::handle_mmio_exit(&mut vcpu, &mut mmio_bus, iss_val, ipa)?;
                        vcpu.advance_pc(4)?;
                    }

                    EC_SYSREG => {
                        handle_sysreg_trap(&mut vcpu, iss_val)?;
                    }

                    EC_WFI => {
                        vcpu.advance_pc(4)?;
                    }

                    _ => {
                        let pc = vcpu.get_reg(HV_REG_PC)?;
                        log::warn!("Unhandled EC=0x{:02x} at PC=0x{:x}", ec_val, pc);
                        vcpu.advance_pc(4)?;
                    }
                }
            }

            HV_EXIT_REASON_VTIMER_ACTIVATED => {
                unsafe {
                    hv_vcpu_set_vtimer_mask(vcpu.handle(), true);
                    hv_vcpu_set_pending_interrupt(
                        vcpu.handle(),
                        HV_INTERRUPT_TYPE_IRQ,
                        true,
                    );
                }
            }

            _ => {
                log::error!("Unknown exit reason: {:?}", exit.reason);
                break;
            }
        }

        // Unmask vtimer after every exit
        unsafe {
            hv_vcpu_set_vtimer_mask(vcpu.handle(), false);
        }
    }

    // Step 11: Terminal is restored by the scope guard
    Ok(())
}
```

Every character you type travels this path: host keyboard to stdin, stdin reader thread pushes into `VecDeque`, guest polls UART FR register (MMIO data abort, handled by VMM), guest reads UART DR register (another data abort), byte delivered to guest's serial driver, shell processes it. Every character the guest prints travels the reverse path: shell writes to stdout, serial driver writes to UART DR (MMIO data abort), VMM calls our TX callback, `print!()` writes to host stdout.

The entire I/O path is MMIO data aborts. Every single character is a VM exit. This is why real hypervisors use virtio - a virtio-console batches hundreds of bytes per exit using shared-memory descriptor rings, reducing the exit rate by orders of magnitude. But for a serial console at human typing speed, one-exit-per-byte works fine.

## Architecture Recap

Let's step back and look at the complete system we built across ten posts:

```
Host Process (macOS)
  |
  +-- hv.rs:        Raw FFI to Hypervisor.framework
  +-- vm.rs:        Safe VM wrapper (RAII create/destroy)
  +-- vcpu.rs:      Safe vCPU wrapper + register access
  +-- memory.rs:    Guest memory (mmap + hv_vm_map)
  +-- mmio.rs:      MMIO bus + MmioDevice trait
  +-- loader.rs:    ARM64 Image loading + validation
  +-- fdt.rs:       Device tree generation (vm-fdt)
  +-- devices/
  |     +-- pl011.rs: UART serial console (TX + RX)
  +-- arch/
        +-- syndrome.rs: ESR_EL2 decoding (EC, ISS, IL)
        +-- psci.rs:     Power management (version, shutdown)
        +-- sysreg.rs:   System register trap stubs
```

Each layer has a distinct responsibility:

**FFI Layer** (`hv.rs`): Raw C bindings translated from Apple's Hypervisor.h header. `extern "C"` declarations, `#[repr(C)]` structs, type aliases. This is the boundary between Rust and the operating system. Every hypervisor call passes through here.

**Safe Abstractions** (`vm.rs`, `vcpu.rs`, `memory.rs`): RAII wrappers that ensure cleanup. `Vm::new()` calls `hv_vm_create()`, and `Drop` calls `hv_vm_destroy()`. `Vcpu` provides typed register access - `get_reg(HV_REG_X0)` instead of raw FFI calls. `GuestMemory` wraps `mmap` and ensures the memory is unmapped from the VM before being freed. These prevent resource leaks and use-after-free bugs at the type level.

**VMM Logic** (the exit loop in `main.rs`): The heart of the hypervisor. A loop that calls `hv_vcpu_run()`, inspects the exit reason, dispatches to handlers, and repeats. The structure never changes - what changes is how many exit types we handle and how sophisticated each handler is.

**Syndrome Decoding** (`syndrome.rs`): Pure bit manipulation functions that extract the Exception Class, Instruction Specific Syndrome, and Instruction Length from the ESR_EL2 value. Three small functions that every handler depends on.

**Device Emulation** (`mmio.rs`, `pl011.rs`): The MMIO bus routes data abort exits to virtual devices based on address. The `MmioDevice` trait defines the interface - `read(offset, size) -> u64` and `write(offset, size, value)`. The PL011 UART implements this trait with transmit (write to host stdout) and receive (read from shared buffer) paths.

**Guest Support** (`loader.rs`, `fdt.rs`): Infrastructure for getting a guest operating system running. The loader validates and places the kernel Image. The FDT generator builds a device tree describing the virtual machine's hardware - memory, CPUs, UART, timer, PSCI. The guest discovers its environment by parsing this tree.

**Architecture Handlers** (`psci.rs`, `sysreg.rs`): ARM-specific protocol implementations. PSCI handles power management calls (version query, shutdown, CPU control). The system register handler stubs out trapped register accesses with safe default values.

## What is Not Covered

This hypervisor boots Linux and gives you an interactive shell. That is a remarkable amount of functionality from under 2000 lines of Rust. But a production hypervisor has features we deliberately skipped:

**GIC (Generic Interrupt Controller) emulation**: We injected timer interrupts directly via `hv_vcpu_set_pending_interrupt`, bypassing any interrupt controller emulation. A real system needs a GIC to route interrupts from devices to CPUs, to handle interrupt priorities, and to support multiple interrupt sources. Without a GIC, you cannot have device interrupts - only the timer works.

**Multiple vCPUs**: We used a single vCPU throughout. Adding more requires implementing PSCI `CPU_ON` to start secondary vCPUs, managing per-CPU state, and handling inter-processor interrupts (IPIs) through the GIC. Each vCPU runs on its own host thread.

**virtio devices**: The standard for paravirtualized I/O in modern hypervisors. virtio-blk for disk, virtio-net for networking, virtio-console for serial, virtio-vsock for host-guest communication. Each is an MMIO (or PCI) device with a shared-memory ring buffer for high-throughput data transfer. Adding virtio-net would give the guest network access; adding virtio-blk would let it mount a real filesystem.

**Persistent storage**: Our initramfs lives in RAM. Everything written to the filesystem is lost when the VM shuts down. A real VM needs a block device backed by a disk image file. This requires either virtio-blk emulation or a simulated NVMe controller.

**Graphics and display**: We have serial console only. A graphical display requires a framebuffer device or virtio-gpu, plus a way to render the framebuffer on the host (a window, a VNC server, or similar).

**Live migration**: Saving the complete VM state (CPU registers, memory contents, device state) to a file and restoring it on the same or different host. This is how cloud providers do zero-downtime maintenance - they migrate VMs between physical machines while they are running.

**Security**: Our hypervisor trusts the guest completely. A production hypervisor needs to validate MMIO access sizes, handle malicious guests that try to break out of the sandbox, limit resource consumption, and isolate VMs from each other.

Each of these is a substantial project in its own right. Implementing virtio-blk alone, with proper descriptor ring handling, interrupt coalescing, and asynchronous I/O, is easily 1000+ lines. A full GICv3 emulation is similar. But the architecture we built supports all of them - they are all MMIO devices on the bus, or extensions to the exit loop handlers.

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_pl011_rx_empty() {
        let rx = Arc::new(Mutex::new(VecDeque::new()));
        let mut uart = Pl011::new(|_| {}, rx);

        // FR register should show RXFE (bit 4) when buffer is empty
        let fr = uart.read(PL011_FR, 4);
        assert_eq!(fr & (FR_RXFE as u64), FR_RXFE as u64);

        // DR should return 0 when buffer is empty
        let dr = uart.read(PL011_DR, 4);
        assert_eq!(dr, 0);
    }

    #[test]
    fn test_pl011_rx_with_data() {
        let rx = Arc::new(Mutex::new(VecDeque::new()));
        rx.lock().unwrap().push_back(b'A');
        rx.lock().unwrap().push_back(b'B');

        let mut uart = Pl011::new(|_| {}, Arc::clone(&rx));

        // FR should show RXFE=0 (data available)
        let fr = uart.read(PL011_FR, 4);
        assert_eq!(fr & (FR_RXFE as u64), 0);

        // Read first byte
        assert_eq!(uart.read(PL011_DR, 4), b'A' as u64);

        // Read second byte
        assert_eq!(uart.read(PL011_DR, 4), b'B' as u64);

        // Buffer is now empty, FR should show RXFE=1
        let fr = uart.read(PL011_FR, 4);
        assert_eq!(fr & (FR_RXFE as u64), FR_RXFE as u64);
    }

    #[test]
    fn test_pl011_tx() {
        let rx = Arc::new(Mutex::new(VecDeque::new()));
        let mut transmitted = Vec::new();
        let tx_buf = Arc::new(Mutex::new(Vec::<u8>::new()));
        let tx_ref = Arc::clone(&tx_buf);

        let mut uart = Pl011::new(
            move |byte| { tx_ref.lock().unwrap().push(byte); },
            rx,
        );

        uart.write(PL011_DR, 4, b'X' as u64);
        uart.write(PL011_DR, 4, b'Y' as u64);

        let sent = tx_buf.lock().unwrap();
        assert_eq!(&*sent, &[b'X', b'Y']);

## Closing

You built a hypervisor from scratch. Not "from scratch" with air quotes meaning you imported some framework and configured it. Literally from `extern "C" { fn hv_vm_create(...) -> hv_return_t; }` all the way up to an interactive Linux shell.

Every VM exit decoded. Every MMIO access routed. Every byte of the device tree generated. Every PSCI call answered. Every system register trap handled. Every character of UART I/O passing through your code, one data abort at a time.

The code is deliberately minimal. A production hypervisor - QEMU, Firecracker, crosvm, Cloud Hypervisor - has tens of thousands of lines handling edge cases, performance optimizations, device quirks, and guest compatibility issues. But the core loop is the same one you wrote: create VM, map memory, load kernel, run vCPU, handle exits. That is what every hypervisor does, on every architecture, on every platform. The details differ. The loop does not.
