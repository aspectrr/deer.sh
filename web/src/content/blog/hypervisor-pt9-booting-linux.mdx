## The Moment of Truth

Over the past eight posts we have built every piece of a hypervisor from scratch. Raw FFI bindings to Apple's Hypervisor.framework. Safe RAII wrappers around VMs and vCPUs. An exit loop that decodes ARM64 exception syndromes. MMIO routing that dispatches data aborts to virtual devices. A PL011 UART that can print characters. A binary loader that puts code into guest memory. A device tree generator that describes the virtual machine's hardware to the guest.

Each of those pieces was tested in isolation, against small hand-crafted guest programs that executed a few instructions and halted. Now it is time to point the whole thing at something real: a Linux kernel. The ARM64 Image binary, millions of instructions, expecting a fully specified machine to be waiting for it - timers, power management, a serial console, and a coherent device tree describing all of it.

This post adds the three missing pieces that Linux requires: virtual timer handling, PSCI (Power State Coordination Interface) for power management, and system register trap handling. Then we wire everything together, load a kernel Image, and watch Linux boot.

## Obtaining an ARM64 Linux Kernel

You need an ARM64 kernel Image file. There are two paths: download a pre-built one or cross-compile from source.

### Pre-built

The fastest option. Arch Linux ARM, Debian, and Alpine all distribute pre-built ARM64 kernels. Look for a file called `Image` or `vmlinuz` (the latter is usually gzip-compressed - you want the uncompressed flat binary). Debian's `linux-image-arm64` package contains one. Extract it, and you have your kernel.

### Cross-compile from source

This is the approach that gives you full control over kernel configuration, which matters when you need specific drivers enabled:

```bash
# Install cross compiler on macOS
brew install aarch64-elf-gcc  # or use aarch64-linux-gnu-gcc from a Linux host

# Clone Linux at a known stable version
git clone --depth 1 --branch v6.6 https://github.com/torvalds/linux.git
cd linux

# Start with the default arm64 config
make ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- defconfig

# Enable the PL011 UART driver and early console support
echo "CONFIG_SERIAL_AMBA_PL011=y" >> .config
echo "CONFIG_SERIAL_AMBA_PL011_CONSOLE=y" >> .config

# Build the Image
make ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- -j$(nproc) Image

# Result is at arch/arm64/boot/Image
ls -lh arch/arm64/boot/Image
```

The two config lines are critical. `CONFIG_SERIAL_AMBA_PL011` builds support for the PL011 UART - the exact device we emulate in our MMIO bus. `CONFIG_SERIAL_AMBA_PL011_CONSOLE` tells the kernel it can use that UART as a console device, which means kernel messages will flow through it. Without these, the kernel boots silently and you have no way to know what is happening.

The default arm64 config (`defconfig`) includes most of what we need: the ARM generic timer driver, PSCI support, device tree parsing, earlycon (early console before the full driver model initializes). A custom minimal config would boot faster but risks missing something. Start with defconfig and trim later.

## Linux ARM64 Boot Protocol

Loading a Linux ARM64 kernel is not the same as loading an ELF binary or a flat binary at an arbitrary address. The kernel Image has a specific format and a specific set of expectations about the machine state when it gains control.

### The Image Header

The ARM64 Linux Image is a flat binary with a 64-byte header at the start. The header contains metadata the bootloader needs to place the image correctly:

```
Offset  Size  Field
------  ----  -----
0x00    4     Branch instruction (jumps past the header)
0x04    4     Reserved
0x08    8     text_offset (preferred offset from a 2MB-aligned base)
0x10    8     image_size (total size of the Image including BSS)
0x18    8     Flags
0x20    8     Reserved
0x28    8     Reserved
0x30    8     Reserved
0x38    4     Magic: 0x644d5241 ("ARM\x64" in little-endian)
0x3c    4     PE header offset (for UEFI boot, we ignore this)
```

The first instruction at offset 0x00 is a branch that skips over the rest of the header, so if you jump to the start of the Image it just works - execution lands past the header and into the real kernel code. The magic value at offset 0x38 is `0x644d5241`, which reads as "ARM\x64" in ASCII. If this does not match, you do not have a valid ARM64 kernel Image.

The `text_offset` field tells you where the kernel prefers to be loaded relative to a 2MB-aligned boundary. In practice, most modern kernels set this to 0 and are relocatable, but the specification says to respect it. For simplicity - and because it works with every kernel version we care about - we load at `0x80000000`, which is 2MB-aligned.

### Loading the Image

Here is the loader that reads the Image file, validates the header, and produces a load result:

```rust
// src/loader.rs

use anyhow::{bail, Result};

pub struct LoadedSegment {
    pub guest_addr: u64,
    pub data: Vec<u8>,
}

pub struct LoadResult {
    pub entry_point: u64,
    pub segments: Vec<LoadedSegment>,
}

const ARM64_IMAGE_MAGIC: u32 = 0x644d5241;
const KERNEL_LOAD_ADDR: u64 = 0x8000_0000;

pub fn load_linux_image(data: &[u8]) -> Result<LoadResult> {
    if data.len() < 64 {
        bail!("Image too small for ARM64 header ({} bytes)", data.len());
    }

    // Validate the magic at offset 56 (0x38)
    let magic = u32::from_le_bytes(data[56..60].try_into()?);
    if magic != ARM64_IMAGE_MAGIC {
        bail!(
            "Not an ARM64 Image: expected magic 0x{:08x}, got 0x{:08x}",
            ARM64_IMAGE_MAGIC,
            magic
        );
    }

    // Read image_size from offset 16 (0x10)
    let image_size = u64::from_le_bytes(data[16..24].try_into()?);
    log::info!(
        "ARM64 Image: {} bytes on disk, image_size={} bytes (includes BSS)",
        data.len(),
        image_size
    );

    Ok(LoadResult {
        entry_point: KERNEL_LOAD_ADDR,
        segments: vec![LoadedSegment {
            guest_addr: KERNEL_LOAD_ADDR,
            data: data.to_vec(),
        }],
    })
}
```

The loader is deliberately simple. We do not parse the text_offset field or attempt to decode PE headers. The kernel goes at `0x80000000`, which is the conventional load address for ARM64 Linux. The entry point is the start of the Image - recall that the first instruction is a branch past the header, so jumping to the start is correct.

### Boot Protocol Requirements

When the kernel gains control, it expects the following machine state:

- **x0**: Physical address of the device tree blob (DTB)
- **x1, x2, x3**: Zero
- **PC**: Set to the Image load address
- **PSTATE/CPSR**: EL1h with DAIF masked (0x3C5)
- **MMU**: Off (stage-1 disabled in EL1; stage-2 is managed by the hypervisor transparently)
- **Caches**: May be on or off
- **D-cache**: Clean relative to the kernel Image

The kernel uses x0 to find the device tree. Everything it needs to know about the machine - memory layout, UART address, timer interrupts, CPU topology - comes from that DTB. If x0 does not point at a valid flattened device tree, the kernel will fault or hang almost immediately.

## VTimer Handling

The ARM Generic Timer is the heartbeat of any ARM64 system. The kernel uses it for process scheduling, timeout handling, jiffies, and every other time-sensitive operation. Without a working timer, the kernel cannot schedule, which means it hangs right after calibrating the delay loop.

Apple's Hypervisor.framework virtualizes the timer for us. The guest reads and writes the timer's system registers directly - `CNTV_CTL_EL0`, `CNTV_TVAL_EL0`, `CNTVCT_EL0` - without trapping. The framework intercepts the timer hardware and notifies us when the guest's virtual timer fires by returning from `hv_vcpu_run()` with `HV_EXIT_REASON_VTIMER_ACTIVATED`.

When the timer fires, we need to do two things: mask the timer so it does not fire again immediately, and inject the timer interrupt into the guest so the kernel's interrupt handler runs.

```rust
// In the exit loop
HV_EXIT_REASON_VTIMER_ACTIVATED => {
    // Mask the vtimer. This prevents the timer from immediately
    // re-firing when we re-enter the guest. The guest kernel will
    // re-arm the timer itself after handling the interrupt.
    unsafe {
        hv_vcpu_set_vtimer_mask(vcpu.handle(), true);
    }

    // Inject the timer interrupt into the guest.
    // The ARM generic timer's virtual timer interrupt is PPI 27
    // (Private Peripheral Interrupt 27), which corresponds to
    // interrupt ID 27. We inject it as an IRQ (type 0).
    // FIQ would be type 1.
    unsafe {
        hv_vcpu_set_pending_interrupt(
            vcpu.handle(),
            HV_INTERRUPT_TYPE_IRQ,
            true,
        );
    }
}
```

The masking is important. The vtimer fires because the guest programmed it to fire at a specific counter value, and that counter value has been reached. If we re-enter the guest without masking, the timer condition is still true and `hv_vcpu_run()` returns immediately - an infinite exit loop that burns CPU doing nothing. By masking, we tell the framework to suppress the timer exit until we unmask it.

The interrupt injection is equally important. The guest kernel set up the timer and is expecting an interrupt to arrive. `hv_vcpu_set_pending_interrupt` tells the framework to assert the IRQ line when the vCPU next enters the guest. The kernel's interrupt handler will run, acknowledge the timer, schedule the next tick, and re-arm the timer.

After the guest handles the timer interrupt, we need to unmask the vtimer so it can fire again for the next tick. The right place to do this is at the bottom of the exit loop, after handling whatever exit occurred:

```rust
// At the bottom of the exit loop, after the match on exit reason:
// Unmask the vtimer so it can fire again next time the guest programs it.
// This is safe to call unconditionally - if the timer is not masked,
// unmasking is a no-op.
unsafe {
    hv_vcpu_set_vtimer_mask(vcpu.handle(), false);
}
```

This unmask-after-every-exit pattern is simple and correct. The timer stays masked only during the brief window while we are handling the timer exit. As soon as we finish and loop back to `hv_vcpu_run()`, the timer can fire again.

The alternative - unmasking only after timer exits - is subtly wrong. If the guest handles the timer interrupt and re-arms the timer during execution that was triggered by a different exit type (say, an MMIO access), the timer might have fired while we were busy handling the MMIO exit. By unmasking unconditionally, we catch that case.

## PSCI Implementation

PSCI - Power State Coordination Interface - is the standard ARM mechanism for managing CPU power states. It defines a calling convention where the guest issues an HVC or SMC instruction with a specific function ID in x0, and the hypervisor responds with a result in x0.

Linux calls PSCI during boot for several reasons: to query what PSCI version the firmware supports, to bring up secondary CPUs, and to shut down or reboot the machine. Our single-vCPU hypervisor needs to handle a minimal set of PSCI functions. Without this handling, the kernel either hangs at boot (waiting for a PSCI version response) or panics when it tries to shut down.

Here is the PSCI module:

```rust
// src/arch/psci.rs

/// PSCI function IDs (from ARM DEN 0022D - Power State Coordination Interface)
pub const PSCI_VERSION: u32 = 0x8400_0000;
pub const PSCI_CPU_SUSPEND_64: u32 = 0xC400_0001;
pub const PSCI_CPU_OFF: u32 = 0x8400_0002;
pub const PSCI_CPU_ON_64: u32 = 0xC400_0003;
pub const PSCI_SYSTEM_OFF: u32 = 0x8400_0008;
pub const PSCI_SYSTEM_RESET: u32 = 0x8400_0009;
pub const PSCI_FEATURES: u32 = 0x8400_000A;

/// PSCI return codes
pub const PSCI_SUCCESS: i64 = 0;
pub const PSCI_NOT_SUPPORTED: i64 = -1;

/// What the VMM should do after handling a PSCI call
pub enum PsciAction {
    /// Continue running the guest
    Continue,
    /// Guest requested shutdown
    Shutdown,
}

/// Handle a PSCI function call.
/// Returns (return_value_for_x0, action_for_vmm).
pub fn handle_psci(func_id: u32) -> (i64, PsciAction) {
    match func_id {
        PSCI_VERSION => {
            // Report PSCI 0.2 (the version Linux requires for HVC-based PSCI).
            // Version is encoded as major in bits [31:16], minor in bits [15:0].
            // 0.2 = 0x00000002.
            log::info!("PSCI: VERSION query, returning 0.2");
            (0x0000_0002, PsciAction::Continue)
        }

        PSCI_FEATURES => {
            // PSCI_FEATURES queries whether a specific function is supported.
            // We could check the argument and return PSCI_SUCCESS for functions
            // we implement, but returning NOT_SUPPORTED for everything is safe.
            // The kernel falls back gracefully.
            log::debug!("PSCI: FEATURES query, returning NOT_SUPPORTED");
            (PSCI_NOT_SUPPORTED, PsciAction::Continue)
        }

        PSCI_CPU_ON_64 => {
            // CPU_ON is how the guest brings up secondary CPUs.
            // We only have one vCPU, so this always fails.
            log::info!("PSCI: CPU_ON requested, returning NOT_SUPPORTED (single vCPU)");
            (PSCI_NOT_SUPPORTED, PsciAction::Continue)
        }

        PSCI_CPU_OFF => {
            // CPU_OFF is how a CPU takes itself offline.
            // On a single-vCPU VM this should not happen during normal
            // operation, but if it does, treat it as shutdown.
            log::info!("PSCI: CPU_OFF requested, treating as shutdown");
            (PSCI_SUCCESS, PsciAction::Shutdown)
        }

        PSCI_CPU_SUSPEND_64 => {
            // CPU_SUSPEND puts the CPU into a low-power state.
            // Return success and continue - the guest will wake up
            // immediately as if the suspend was instantaneous.
            log::debug!("PSCI: CPU_SUSPEND, returning immediately");
            (PSCI_SUCCESS, PsciAction::Continue)
        }

        PSCI_SYSTEM_OFF => {
            log::info!("PSCI: SYSTEM_OFF - shutting down VM");
            (PSCI_SUCCESS, PsciAction::Shutdown)
        }

        PSCI_SYSTEM_RESET => {
            // A proper implementation would reset the VM state and re-enter
            // the kernel from the top. For now, just shut down.
            log::info!("PSCI: SYSTEM_RESET - treating as shutdown");
            (PSCI_SUCCESS, PsciAction::Shutdown)
        }

        _ => {
            log::warn!("PSCI: unknown function 0x{:08x}, returning NOT_SUPPORTED", func_id);
            (PSCI_NOT_SUPPORTED, PsciAction::Continue)
        }
    }
}
```

The encoding of PSCI function IDs follows a pattern. The high byte `0x84` indicates a 32-bit fast call in the standard SMC/HVC calling convention. `0xC4` indicates a 64-bit fast call. The kernel issues 64-bit calls for functions that take 64-bit arguments (like CPU_ON, which takes a 64-bit target CPU MPIDR and a 64-bit entry point address).

The version number is the most important response. Linux probes for PSCI early in boot via `psci_dt_init()`, which reads the PSCI method (HVC or SMC) and version from the device tree. If the version call returns something less than 0.2, the kernel disables PSCI support, which means it cannot shut down the machine cleanly. Returning 0.2 satisfies the kernel's requirements.

### Integrating PSCI into the Exit Loop

PSCI calls arrive as HVC exceptions (EC = 0x16). The function ID is in x0. We check whether the function ID falls in the PSCI range and, if so, dispatch to our PSCI handler:

```rust
EC_HVC64 => {
    let func_id = vcpu.get_reg(HV_REG_X0)? as u32;

    // Check if this is a PSCI call.
    // PSCI function IDs have 0x84 or 0xC4 in the high byte.
    let is_psci = matches!(func_id >> 24, 0x84 | 0xC4);

    if is_psci {
        let (ret, action) = handle_psci(func_id);
        vcpu.set_reg(HV_REG_X0, ret as u64)?;
        vcpu.advance_pc(4)?;

        match action {
            PsciAction::Continue => {}
            PsciAction::Shutdown => {
                log::info!("Guest requested shutdown via PSCI");
                break;
            }
        }
    } else {
        // Non-PSCI hypercall. Log and return NOT_SUPPORTED.
        let imm = iss(syndrome) & 0xFFFF;
        log::debug!("HVC #{}: x0=0x{:x}", imm, func_id);
        vcpu.set_reg(HV_REG_X0, u64::MAX)?;
        vcpu.advance_pc(4)?;
    }
}
```

The `is_psci` check is a quick heuristic. In a real hypervisor you would dispatch based on the SMCCC (SMC Calling Convention) owner field, but checking the top byte is sufficient for Linux.

## System Register Trap Handling

During boot, the Linux kernel accesses dozens of system registers. Most of these are handled directly by the hardware - the kernel reads and writes them without trapping. But some system registers are configured by the hypervisor to trap on access, and others trap because the hardware does not implement them in a virtualized context.

When a system register access traps, we get a VM exit with EC = 0x18 (MSR/MRS trap). The ISS field encodes which register was accessed, the direction (read or write), and the destination/source general-purpose register. The encoding is dense:

```
ISS bits for EC=0x18 (system register access):
[24]    cv   - Condition code valid (only for AArch32, always 1 for AArch64)
[23:20] cond - Condition code (only for AArch32)
[21:20] Op0  - Op0 field (encoded as actual_Op0 - 2, so value 1 means Op0=3)
[19:17] Op1  - Op1 field
[16:13] CRn  - CRn field
[12:9]  CRm  - CRm field
[8:6]   Op2  - Op2 field
[4:1]   Rt   - Target/source general-purpose register
[0]     Direction - 0 = write (MSR), 1 = read (MRS)
```

Wait - that encoding has a subtlety. The ISS for MSR/MRS traps uses bit 0 for direction, but the convention is inverted from what you might expect: 0 means the guest was _writing_ to the system register (MSR), and 1 means the guest was _reading_ from it (MRS). Some references describe it the other way. The ARM Architecture Reference Manual (DDI 0487) is the authoritative source - check the EC=0x18 ISS encoding table.

Here is the system register trap handler:

```rust
// src/arch/sysreg.rs

use anyhow::Result;

/// Decode and handle a system register access trap (EC = 0x18).
///
/// Most trapped system register accesses can be safely handled by returning
/// 0 for reads and ignoring writes. This is not correct for all registers,
/// but it gets Linux through boot. A production hypervisor would need to
/// emulate specific registers properly.
pub fn handle_sysreg_trap(vcpu: &mut Vcpu, iss: u32) -> Result<()> {
    let direction = iss & 1;         // 0 = write (MSR), 1 = read (MRS)
    let rt = ((iss >> 1) & 0xF) as u32;  // Target register
    let op2 = (iss >> 6) & 0x7;
    let crm = (iss >> 9) & 0xF;
    let crn = (iss >> 13) & 0xF;
    let op1 = (iss >> 17) & 0x7;
    let op0 = ((iss >> 20) & 0x3) + 2;  // Actual Op0 = encoded + 2

    let is_read = direction == 1;

    log::trace!(
        "sysreg trap: {} x{}, S{}_{}_C{}_C{}_{}",
        if is_read { "MRS" } else { "MSR" },
        rt,
        op0, op1, crn, crm, op2,
    );

    if is_read {
        // For most trapped system registers, returning 0 is safe.
        // The guest gets a zero value and continues.
        //
        // Notable registers that trap during Linux boot:
        // - ICC_* (GIC system registers): returning 0 tells the kernel
        //   that interrupts are not pending
        // - ID_AA64* (CPU feature registers): returning 0 disables
        //   optional features the guest checks for
        // - AIDR_EL1 (implementation defined): 0 is fine
        //
        // A real hypervisor would maintain shadow values for many of
        // these and emulate them faithfully.
        let reg = HV_REG_X0 + rt;
        vcpu.set_reg(reg, 0)?;
    }
    // For writes (MSR): silently discard the value.
    // The guest thinks the write succeeded.

    vcpu.advance_pc(4)?;
    Ok(())
}
```

This is a blunt approach and it works surprisingly well. The Linux kernel queries dozens of feature registers during early boot - `ID_AA64PFR0_EL1`, `ID_AA64ISAR0_EL1`, `ID_AA64MMFR0_EL1`, and many more. These tell the kernel what CPU features are available. Returning 0 means "no optional features" - the kernel uses baseline code paths for everything. Not optimal for performance, but correct.

The GIC (Generic Interrupt Controller) system registers - `ICC_SRE_EL1`, `ICC_PMR_EL1`, `ICC_IAR1_EL1`, and friends - also trap here. Without a full GIC emulation (which we do not have), returning 0 for reads and ignoring writes is the best we can do. The kernel will not be able to use interrupts through the GIC interface, but our direct timer interrupt injection via `hv_vcpu_set_pending_interrupt` bypasses the GIC entirely. This works for a single-vCPU setup.

### A More Targeted Approach

If you want to log exactly which registers are being accessed rather than silently returning 0 for everything, you can decode the `(Op0, Op1, CRn, CRm, Op2)` tuple into known register names:

```rust
fn sysreg_name(op0: u32, op1: u32, crn: u32, crm: u32, op2: u32) -> &'static str {
    match (op0, op1, crn, crm, op2) {
        (3, 0, 0, 1, 0) => "ID_AA64PFR0_EL1",
        (3, 0, 0, 1, 1) => "ID_AA64PFR1_EL1",
        (3, 0, 0, 6, 0) => "ID_AA64ISAR0_EL1",
        (3, 0, 0, 6, 1) => "ID_AA64ISAR1_EL1",
        (3, 0, 0, 7, 0) => "ID_AA64MMFR0_EL1",
        (3, 0, 0, 7, 1) => "ID_AA64MMFR1_EL1",
        (3, 0, 12, 12, 5) => "ICC_SRE_EL1",
        (3, 0, 12, 8, 0) => "ICC_IAR1_EL1",
        (3, 0, 12, 12, 0) => "ICC_PMR_EL1 / ICC_CTLR_EL1",
        _ => "unknown",
    }
}
```

During development, logging every trapped register access at the `debug` level is invaluable. You can see exactly what the kernel is probing for and whether your return values are causing problems. Set `RUST_LOG=trace` and watch the stream of system register accesses scroll by. There will be hundreds of them in the first few milliseconds of boot.

## The Complete Boot Sequence

With timer handling, PSCI, and system register traps in place, we can now assemble the complete boot flow. Here is the full `main.rs` that boots a Linux kernel:

```rust
use anyhow::Result;
use std::path::PathBuf;

mod arch;
mod devices;
mod fdt;
mod hv;
mod loader;
mod memory;
mod mmio;
mod vcpu;
mod vm;

use arch::psci::{handle_psci, PsciAction};
use arch::syndrome::*;
use arch::sysreg::handle_sysreg_trap;

const GUEST_MEM_SIZE: u64 = 512 * 1024 * 1024; // 512 MB
const GUEST_MEM_BASE: u64 = 0x4000_0000;        // 1 GB
const KERNEL_LOAD_ADDR: u64 = 0x8000_0000;      // 2 GB (must be 2MB-aligned)
const UART_BASE: u64 = 0x0900_0000;
const UART_SIZE: u64 = 0x1000;

fn main() -> Result<()> {
    env_logger::init();

    // Parse arguments
    let kernel_path = std::env::args()
        .nth(1)
        .map(PathBuf::from)
        .expect("Usage: apple-hypervisor <kernel-image>");

    // Read the kernel Image
    let kernel_data = std::fs::read(&kernel_path)?;
    log::info!("Loaded kernel: {} bytes from {:?}", kernel_data.len(), kernel_path);

    // Load and validate the kernel Image
    let load_result = loader::load_linux_image(&kernel_data)?;
    log::info!("Kernel entry point: 0x{:016x}", load_result.entry_point);

    // Create the VM
    let vm = vm::Vm::new()?;

    // Allocate and map guest memory
    let memory = memory::GuestMemory::new(GUEST_MEM_SIZE)?;
    vm.map_memory(GUEST_MEM_BASE, memory.as_ptr(), GUEST_MEM_SIZE)?;

    // Write the kernel into guest memory
    for segment in &load_result.segments {
        let offset = segment.guest_addr - GUEST_MEM_BASE;
        memory.write_at(offset, &segment.data);
    }
    log::info!("Kernel loaded at guest address 0x{:016x}", KERNEL_LOAD_ADDR);

    // Generate the device tree
    let dtb = fdt::create_fdt(
        GUEST_MEM_BASE,
        GUEST_MEM_SIZE,
        1, // number of CPUs
        "console=ttyAMA0 earlycon",
        UART_BASE,
        None, // no initrd
        None,
    )?;

    // Place the DTB at the top of RAM, 2MB-aligned, leaving space
    let dtb_addr = (GUEST_MEM_BASE + GUEST_MEM_SIZE - 0x20_0000) & !0x1F_FFFF;
    let dtb_offset = dtb_addr - GUEST_MEM_BASE;
    memory.write_at(dtb_offset, &dtb);
    log::info!("DTB placed at guest address 0x{:016x} ({} bytes)", dtb_addr, dtb.len());

    // Set up the MMIO bus with our PL011 UART
    let mut mmio_bus = mmio::MmioBus::new();
    let uart = devices::pl011::Pl011::new(|byte| {
        print!("{}", byte as char);
        use std::io::Write;
        std::io::stdout().flush().ok();
    });
    mmio_bus.register(UART_BASE, UART_SIZE, Box::new(uart));

    // Create vCPU
    let mut vcpu = vm.create_vcpu()?;

    // Set initial register state per ARM64 Linux boot protocol
    vcpu.set_reg(HV_REG_PC, load_result.entry_point)?;
    vcpu.set_reg(HV_REG_X0, dtb_addr)?;  // x0 = DTB address
    vcpu.set_reg(HV_REG_X1, 0)?;          // x1 = 0
    vcpu.set_reg(HV_REG_X2, 0)?;          // x2 = 0
    vcpu.set_reg(HV_REG_X3, 0)?;          // x3 = 0
    vcpu.set_reg(HV_REG_CPSR, 0x3C5)?;    // EL1h, DAIF masked

    log::info!("Starting VM: PC=0x{:016x}, x0(DTB)=0x{:016x}", load_result.entry_point, dtb_addr);

    // The exit loop
    let mut exit_count: u64 = 0;
    loop {
        vcpu.run()?;
        exit_count += 1;

        let exit = vcpu.exit_info();

        match exit.reason {
            HV_EXIT_REASON_EXCEPTION => {
                let syndrome = exit.exception.syndrome;
                let ec_val = ec(syndrome);
                let iss_val = iss(syndrome);

                match ec_val {
                    EC_HVC64 => {
                        let func_id = vcpu.get_reg(HV_REG_X0)? as u32;
                        let is_psci = matches!(func_id >> 24, 0x84 | 0xC4);

                        if is_psci {
                            let (ret, action) = handle_psci(func_id);
                            vcpu.set_reg(HV_REG_X0, ret as u64)?;
                            vcpu.advance_pc(4)?;
                            match action {
                                PsciAction::Continue => {}
                                PsciAction::Shutdown => {
                                    println!("\n[VMM] Guest shutdown via PSCI after {} exits", exit_count);
                                    return Ok(());
                                }
                            }
                        } else {
                            log::debug!("HVC: non-PSCI call, func=0x{:08x}", func_id);
                            vcpu.set_reg(HV_REG_X0, u64::MAX)?;
                            vcpu.advance_pc(4)?;
                        }
                    }

                    EC_SMC64 => {
                        // Linux may try PSCI over SMC before falling back to HVC.
                        // Handle the same way.
                        let func_id = vcpu.get_reg(HV_REG_X0)? as u32;
                        let is_psci = matches!(func_id >> 24, 0x84 | 0xC4);

                        if is_psci {
                            let (ret, action) = handle_psci(func_id);
                            vcpu.set_reg(HV_REG_X0, ret as u64)?;
                            vcpu.advance_pc(4)?;
                            match action {
                                PsciAction::Continue => {}
                                PsciAction::Shutdown => {
                                    println!("\n[VMM] Guest shutdown via PSCI (SMC) after {} exits", exit_count);
                                    return Ok(());
                                }
                            }
                        } else {
                            vcpu.set_reg(HV_REG_X0, u64::MAX)?;
                            vcpu.advance_pc(4)?;
                        }
                    }

                    EC_DABT_LOWER => {
                        // Data abort from guest - this is MMIO.
                        let ipa = exit.exception.physical_address;
                        mmio::handle_mmio_exit(&mut vcpu, &mut mmio_bus, iss_val, ipa)?;
                        vcpu.advance_pc(4)?;
                    }

                    EC_SYSREG => {
                        handle_sysreg_trap(&mut vcpu, iss_val)?;
                    }

                    EC_WFI => {
                        // Guest is idle. In a proper VMM we would block until
                        // an interrupt is pending. For now, just continue.
                        vcpu.advance_pc(4)?;
                    }

                    EC_BRK64 => {
                        let pc = vcpu.get_reg(HV_REG_PC)?;
                        println!("\n[VMM] BRK at PC=0x{:016x} after {} exits", pc, exit_count);
                        break;
                    }

                    _ => {
                        let pc = vcpu.get_reg(HV_REG_PC)?;
                        log::error!(
                            "Unhandled exception: EC=0x{:02x}, ISS=0x{:08x}, PC=0x{:016x}",
                            ec_val, iss_val, pc
                        );
                        // Skip the instruction and hope for the best.
                        // In production, you would inject a fault into the guest.
                        vcpu.advance_pc(4)?;
                    }
                }
            }

            HV_EXIT_REASON_VTIMER_ACTIVATED => {
                // Guest's virtual timer expired
                unsafe {
                    hv_vcpu_set_vtimer_mask(vcpu.handle(), true);
                    hv_vcpu_set_pending_interrupt(vcpu.handle(), HV_INTERRUPT_TYPE_IRQ, true);
                }
            }

            reason => {
                log::error!("Unknown exit reason: {:?}", reason);
                break;
            }
        }

        // Unmask the vtimer after every exit so it can fire again
        unsafe {
            hv_vcpu_set_vtimer_mask(vcpu.handle(), false);
        }
    }

    Ok(())
}
```

Let's walk through the key decisions.

**Memory layout**: Guest RAM starts at `0x40000000` (1 GB) and is 512 MB. The kernel loads at `0x80000000` (2 GB), which is within the mapped range since `0x40000000 + 0x20000000 = 0x60000000` ... actually, this is a problem. `0x80000000` is not within `[0x40000000, 0x60000000)`. We need to either start guest RAM lower or make the kernel load address relative to the start of RAM.

The standard approach is to set `GUEST_MEM_BASE = 0x40000000` and load the kernel at `GUEST_MEM_BASE + some_offset`. Or, more commonly for ARM64 VMs, start RAM at `0x40000000` and set the kernel load address to `0x40000000` itself - the kernel will relocate internally. But the simplest approach that matches the conventional ARM64 virt machine layout: start RAM at `0x40000000` with enough size that the kernel load address `0x80000000` falls within it. That means at least 1 GB + 1 byte of RAM. Alternatively:

```rust
const GUEST_MEM_BASE: u64 = 0x4000_0000;
const GUEST_MEM_SIZE: u64 = 1024 * 1024 * 1024; // 1 GB
const KERNEL_LOAD_ADDR: u64 = GUEST_MEM_BASE;    // Load kernel at start of RAM
```

Loading the kernel at the start of RAM works fine. Modern ARM64 Linux kernels are relocatable and do not care about the specific address as long as it is 2 MB-aligned. The DTB address in x0 just needs to point somewhere in mapped memory that does not overlap the kernel.

**DTB placement**: We put the DTB at the top of RAM, 2 MB-aligned, with plenty of room below. The kernel copies the DTB early in boot, so even if the DTB overlaps with the kernel's BSS expansion, the data is safe. Placing it at the top avoids any overlap with the kernel Image at the bottom.

**UART address**: `0x09000000` matches the ARM "virt" machine's standard PL011 address. This matters because the `earlycon` kernel parameter looks for a PL011 at the address specified in the device tree. If the DTB says the UART is at `0x09000000`, and we register our MMIO device at `0x09000000`, and we _don't_ map that address range in stage-2 so it generates data aborts, then everything works.

**Exit counting**: The `exit_count` variable is purely for curiosity - when the VM shuts down, you can see how many exits occurred. A typical Linux boot to kernel panic (no rootfs) generates several hundred thousand exits.

## Debugging Tips

Not everything will work on the first try. Here are the failure modes you are most likely to encounter and how to diagnose them.

**No output at all**: The kernel is not finding the UART, or it is not using earlycon. Check three things: (1) the DTB has a `stdout-path` pointing to the UART node, (2) the UART node's `reg` property matches the MMIO address you registered, (3) the kernel command line includes `earlycon`. Also verify that the UART's MMIO range is NOT mapped in stage-2 memory - it needs to generate data aborts to be intercepted. If you accidentally mapped `0x09000000` as regular RAM, the guest's writes go to RAM instead of trapping.

**Kernel panics immediately (within the first few instructions)**: The DTB is probably invalid or x0 does not point to it. Run `dtc` (device tree compiler) on your generated DTB to validate it: `dtc -I dtb -O dts your.dtb`. If dtc cannot parse it, the kernel cannot either. Also verify that x0 contains the DTB's guest physical address, not a host virtual address.

**Hang after "Calibrating delay loop"**: The timer is not working. The kernel is waiting for a timer interrupt that never arrives. Check that your vtimer exit handler both masks the timer AND injects the interrupt. Check that you unmask the timer after handling the exit. Add a log line inside the vtimer handler and verify it is being called.

**Crash with "unhandled exception class"**: A system register access or other trap that you do not handle. Log the EC and ISS values. Look up the EC in the ARM Architecture Reference Manual to identify the trap type. Most likely it is a system register access (EC=0x18) that you need to add to your handler. Check the `(Op0, Op1, CRn, CRm, Op2)` encoding to identify the specific register.

**"Synchronous External Abort"**: The guest accessed a memory address that faults in the host. This usually means a bug in your memory mapping - the kernel is accessing a guest physical address that you did not map and that is not in an MMIO range. Check the faulting address in the exception information and verify your memory map covers it.

**Extremely slow boot**: Every WFI instruction causes a VM exit. If the kernel is executing WFI in a loop (which it does when idle), and your WFI handler immediately returns, you get a tight loop of VM exits. Each exit costs microseconds, but millions of them add up. This is normal for a minimal hypervisor without proper idle handling. The fix is to block in the WFI handler until an interrupt is pending, but that requires tracking pending interrupts properly.

## Testing

### Unit Tests

The PSCI handler is pure logic, perfect for unit testing:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_psci_version() {
        let (ret, action) = handle_psci(PSCI_VERSION);
        assert_eq!(ret, 0x00000002);
        assert!(matches!(action, PsciAction::Continue));
    }

    #[test]
    fn test_psci_cpu_on_not_supported() {
        let (ret, action) = handle_psci(PSCI_CPU_ON_64);
        assert_eq!(ret, PSCI_NOT_SUPPORTED);
        assert!(matches!(action, PsciAction::Continue));
    }

    #[test]
    fn test_psci_system_off() {
        let (_, action) = handle_psci(PSCI_SYSTEM_OFF);
        assert!(matches!(action, PsciAction::Shutdown));
    }

    #[test]
    fn test_psci_system_reset() {
        let (_, action) = handle_psci(PSCI_SYSTEM_RESET);
        assert!(matches!(action, PsciAction::Shutdown));
    }

    #[test]
    fn test_psci_unknown_function() {
        let (ret, action) = handle_psci(0x84FF_FFFF);
        assert_eq!(ret, PSCI_NOT_SUPPORTED);
        assert!(matches!(action, PsciAction::Continue));
    }
}
```

### Integration Test

The real integration test boots a kernel and checks for expected output. This requires having a kernel Image available, so it is typically run as an ignored test that you invoke explicitly:

```rust
#[test]
#[ignore = "requires kernel Image at test-data/Image"]
fn test_boot_to_panic() {
    // Boot the kernel, capture UART output, verify "Booting Linux" appears
    let output = boot_kernel("test-data/Image", None, Duration::from_secs(10));
    assert!(output.contains("Booting Linux"));
    assert!(output.contains("earlycon: pl011"));
    // The kernel should panic because there is no rootfs
    assert!(output.contains("VFS: Unable to mount root fs"));
}
```
