## Why Device Trees Exist

On x86, hardware discovery is mostly automatic. The BIOS or UEFI firmware populates ACPI tables describing the system. PCI buses can be enumerated by probing configuration space. USB devices announce themselves. The kernel boots, scans these standard interfaces, and builds a model of the hardware. You do not have to tell the kernel what is there - it can find out on its own.

ARM has none of this. The ARM architecture does not mandate any particular bus topology, memory map, interrupt routing, or device placement. One ARM board might have a UART at `0x09000000`. Another might have it at `0x20201000`. A third might have no UART at all, just a SPI-attached serial adapter at some arbitrary chip-select address. The kernel cannot probe random addresses hoping to find devices - writing to an unmapped address on ARM does not return `0xFFFFFFFF` like a PCI bus miss. It causes a data abort, which crashes the kernel.

The solution is to tell the kernel what hardware exists before it starts looking. That is what a **device tree** does. It is a structured data format that describes the hardware layout of a machine: how much RAM, where the UART is, how many CPUs, where the interrupt controller is, what the clock frequencies are, how the timer works. The bootloader constructs a device tree, places it in memory, and passes its address to the kernel as a boot parameter.

For our hypervisor, this means we need to generate a device tree that accurately describes the virtual hardware we are emulating. If we tell the kernel there is a PL011 UART at `0x09000000`, it will configure a PL011 driver to access that address. When it reads and writes that address, our MMIO handler will intercept the accesses and emulate the UART. The device tree is the contract between the VMM and the guest kernel - it says "here is what you have to work with."

## The FDT Binary Format

Device trees come in two forms:

**Device Tree Source (DTS)** is the human-readable text format. You have probably seen files like `arch/arm64/boot/dts/broadcom/bcm2711.dts` in the kernel source. These are compiled by the Device Tree Compiler (`dtc`) into the binary format.

**Flattened Device Tree Blob (FDT/DTB)** is the binary format that the kernel actually consumes at boot. It starts with a magic number `0xd00dfeed`, followed by a header containing the blob size, structure offset, strings offset, and other metadata. The body is a flattened representation of the tree: `FDT_BEGIN_NODE` tokens, property tokens with name references into a string table, and `FDT_END_NODE` tokens.

We do not need to understand every byte of the binary format because we are going to use a crate that handles serialization for us. But it is worth knowing the high-level layout:

```
+---------------------------+
| FDT Header (40 bytes)     |
|  magic: 0xd00dfeed        |
|  totalsize                |
|  off_dt_struct             |
|  off_dt_strings            |
|  ...                       |
+---------------------------+
| Memory reservation block  |
+---------------------------+
| Structure block           |
|  FDT_BEGIN_NODE ""        |
|    properties...          |
|    FDT_BEGIN_NODE "memory" |
|      properties...        |
|    FDT_END_NODE           |
|    ...                    |
|  FDT_END_NODE             |
+---------------------------+
| Strings block             |
|  "compatible\0"           |
|  "reg\0"                  |
|  ...                      |
+---------------------------+
```

The tree structure is intuitive. The root node contains child nodes, each of which can contain properties (key-value pairs) and further child nodes. Properties have a name (stored as an offset into the strings block) and a value (which can be a string, a u32, a u64, a list of u32s, or raw bytes).

We use the `vm-fdt` crate to generate DTBs programmatically. Despite the "vm" prefix in its name, `vm-fdt` is purely a serialization library - it writes bytes into the FDT binary format. It does not interact with any hypervisor APIs or VM infrastructure. Add it to your `Cargo.toml`:

```toml
[dependencies]
vm-fdt = "0.3"
```

## The Required DTB Nodes

A Linux kernel on ARM64 needs several device tree nodes to boot. Without these, it will either panic immediately or hang trying to access hardware that it cannot find. Let us build the DTB generation function in `src/fdt.rs`, going through each required node and explaining why it matters.

### The Complete DTB Generator

```rust
use vm_fdt::FdtWriter;
use anyhow::Result;

pub fn create_fdt(
    memory_size: u64,
    num_cpus: u32,
    cmdline: &str,
    initrd_start: Option<u64>,
    initrd_end: Option<u64>,
) -> Result<Vec<u8>> {
    let mut fdt = FdtWriter::new()?;

    // Root node
    let root = fdt.begin_node("")?;
    fdt.property_string("compatible", "linux,dummy-virt")?;
    fdt.property_u32("#address-cells", 2)?;
    fdt.property_u32("#size-cells", 2)?;

    // /chosen - boot parameters
    let chosen = fdt.begin_node("chosen")?;
    fdt.property_string("bootargs", cmdline)?;
    fdt.property_string("stdout-path", "/pl011@9000000")?;
    if let (Some(start), Some(end)) = (initrd_start, initrd_end) {
        fdt.property_u64("linux,initrd-start", start)?;
        fdt.property_u64("linux,initrd-end", end)?;
    }
    fdt.end_node(chosen)?;

    // /memory - RAM description
    let memory = fdt.begin_node("memory@80000000")?;
    fdt.property_string("device_type", "memory")?;
    fdt.property_array_u64("reg", &[0x80000000, memory_size])?;
    fdt.end_node(memory)?;

    // /cpus - processor description
    let cpus = fdt.begin_node("cpus")?;
    fdt.property_u32("#address-cells", 1)?;
    fdt.property_u32("#size-cells", 0)?;
    for i in 0..num_cpus {
        let cpu = fdt.begin_node(&format!("cpu@{i}"))?;
        fdt.property_string("device_type", "cpu")?;
        fdt.property_string("compatible", "arm,arm-v8")?;
        fdt.property_u32("reg", i)?;
        fdt.end_node(cpu)?;
    }
    fdt.end_node(cpus)?;

    // /pl011 - UART
    let uart = fdt.begin_node("pl011@9000000")?;
    fdt.property_string("compatible", "arm,pl011\0arm,primecell")?;
    fdt.property_array_u64("reg", &[0x09000000, 0x1000])?;
    fdt.property_string("clock-names", "uartclk\0apb_pclk")?;
    fdt.end_node(uart)?;

    // /timer - ARM generic timer
    let timer = fdt.begin_node("timer")?;
    fdt.property_string("compatible", "arm,armv8-timer")?;
    fdt.property_null("always-on")?;
    fdt.end_node(timer)?;

    // /psci - power state coordination
    let psci = fdt.begin_node("psci")?;
    fdt.property_string("compatible", "arm,psci-0.2")?;
    fdt.property_string("method", "hvc")?;
    fdt.end_node(psci)?;

    fdt.end_node(root)?;

    Ok(fdt.finish()?)
}
```

That is the complete function. Now let us explain each node in detail.

### Root Node

```rust
let root = fdt.begin_node("")?;
fdt.property_string("compatible", "linux,dummy-virt")?;
fdt.property_u32("#address-cells", 2)?;
fdt.property_u32("#size-cells", 2)?;
```

The root node (empty name) is the top of the tree. Every device tree has exactly one.

`compatible` identifies the machine. Linux uses this string to look up a machine descriptor in its compiled-in platform list. `"linux,dummy-virt"` is a generic virtual machine compatible string - the kernel does not require a specific platform match for it, which works in our favor since we are not emulating any real hardware platform.

`#address-cells` and `#size-cells` specify how many 32-bit cells are used to encode addresses and sizes in child nodes' `reg` properties. With both set to 2, addresses and sizes are 64-bit values encoded as two consecutive u32s. This is standard for AArch64 systems where addresses are 64 bits wide.

### The /chosen Node

```rust
let chosen = fdt.begin_node("chosen")?;
fdt.property_string("bootargs", cmdline)?;
fdt.property_string("stdout-path", "/pl011@9000000")?;
if let (Some(start), Some(end)) = (initrd_start, initrd_end) {
    fdt.property_u64("linux,initrd-start", start)?;
    fdt.property_u64("linux,initrd-end", end)?;
}
fdt.end_node(chosen)?;
```

`/chosen` is a special node that does not describe hardware. It passes boot-time parameters from the bootloader (us) to the kernel.

`bootargs` is the kernel command line. For a minimal boot, `"console=ttyAMA0 earlycon"` tells Linux to use the ARM AMBA UART (`ttyAMA0`, which is the PL011) as its console and to enable early console output before the full driver model is initialized. The `earlycon` parameter is particularly important - without it, you get no output during early kernel initialization, which makes debugging impossible.

`stdout-path` tells the kernel which device to use for standard output. The path `/pl011@9000000` references the UART node we define later in the tree. Linux resolves this path to find the UART driver and connect it to the console subsystem. If this path does not match the UART node's name and address, the kernel will boot silently - no output at all until userspace starts, if it starts.

`linux,initrd-start` and `linux,initrd-end` specify the guest physical address range where an initial ramdisk (initramfs) has been loaded. The kernel uses these addresses to find the initramfs, mount it as the root filesystem, and execute `/init`. We make these optional because not every boot needs an initramfs - for now, we might just want to see the kernel print messages to the console.

### The /memory Node

```rust
let memory = fdt.begin_node("memory@80000000")?;
fdt.property_string("device_type", "memory")?;
fdt.property_array_u64("reg", &[0x80000000, memory_size])?;
fdt.end_node(memory)?;
```

This is how the kernel learns how much RAM it has. Without this node, the kernel has no idea how much memory is available and will panic during early boot.

The node name `memory@80000000` follows the device tree naming convention: `type@address`. The `@80000000` suffix indicates the unit address of this memory region.

`device_type` must be `"memory"` - this is how the kernel identifies memory nodes. It is one of the few properties where the kernel checks an exact string match.

`reg` contains two 64-bit values (encoded as four u32s because `#address-cells` and `#size-cells` are both 2): the base address and the size. We use `0x80000000` as the base, which is the conventional RAM base for ARM64 virtual machines (QEMU's `virt` machine uses the same address). The size comes from the `memory_size` parameter.

The base address here must match where we actually mapped guest memory in the VMM via `hv_vm_map`. If we tell the kernel it has RAM at `0x80000000` but we mapped guest memory starting at `0x40000000`, the kernel will fault on its first memory access.

### The /cpus Node

```rust
let cpus = fdt.begin_node("cpus")?;
fdt.property_u32("#address-cells", 1)?;
fdt.property_u32("#size-cells", 0)?;
for i in 0..num_cpus {
    let cpu = fdt.begin_node(&format!("cpu@{i}"))?;
    fdt.property_string("device_type", "cpu")?;
    fdt.property_string("compatible", "arm,arm-v8")?;
    fdt.property_u32("reg", i)?;
    fdt.end_node(cpu)?;
}
fdt.end_node(cpus)?;
```

The `/cpus` node describes the system's processors. Each CPU gets its own child node.

`#address-cells = 1` and `#size-cells = 0` within the `/cpus` node mean that CPU `reg` properties are single u32 values with no size component. This makes sense - a CPU has an ID, not an address range.

Each `cpu@N` node has:

- `device_type = "cpu"` - identifies this as a CPU node.
- `compatible = "arm,arm-v8"` - tells the kernel this is an ARMv8 processor. The kernel uses this to select the right CPU feature initialization code.
- `reg = i` - the CPU's hardware ID. For our single-CPU VM, this is 0. For multi-CPU VMs, each CPU would have a sequential ID, and these IDs must match the MPIDR_EL1 register values that the hypervisor configures for each vCPU.

We start with `num_cpus = 1` because Hypervisor.framework creates one vCPU per `hv_vcpu_create` call. Multi-CPU support would require creating multiple vCPUs and running them on separate host threads. That is a significant amount of complexity that we do not need for a working system.

### The /pl011 (UART) Node

```rust
let uart = fdt.begin_node("pl011@9000000")?;
fdt.property_string("compatible", "arm,pl011\0arm,primecell")?;
fdt.property_array_u64("reg", &[0x09000000, 0x1000])?;
fdt.property_string("clock-names", "uartclk\0apb_pclk")?;
fdt.end_node(uart)?;
```

This tells the kernel about our virtual UART. The `compatible` property is a null-terminated list of compatible device strings. `"arm,pl011\0arm,primecell"` says this device is compatible with both the PL011 UART driver and the more general ARM PrimeCell bus driver. The kernel tries each compatible string in order until it finds a matching driver.

Note the embedded `\0` (null byte) in the string. This is how device trees encode lists of strings - they are concatenated with null separators. The `vm-fdt` crate handles the final null terminator, but the internal separators must be explicit.

`reg` specifies the MMIO register region: base address `0x09000000` and size `0x1000` (4KB). This must match the MMIO address range that our VMM traps and routes to the UART emulation code. When the kernel's PL011 driver reads or writes within this 4KB region, the access faults to our VMM, which emulates the UART register behavior.

`clock-names` lists the clocks the PL011 driver expects. A real PL011 needs `uartclk` (the baud rate generator clock) and `apb_pclk` (the APB bus clock). Our emulated UART does not have real clocks, but the driver checks for them during probe. Providing the names without actual clock nodes causes the driver to use default/fallback clock handling, which is sufficient for a virtual UART that does not actually need clock configuration.

A real device tree would also include an `interrupts` property linking the UART to an interrupt controller. We omit this for now because we do not have full GIC support. The UART will work in polling mode - the kernel will poll the UART status registers rather than waiting for interrupts. This is less efficient but perfectly functional.

### The /timer Node

```rust
let timer = fdt.begin_node("timer")?;
fdt.property_string("compatible", "arm,armv8-timer")?;
fdt.property_null("always-on")?;
fdt.end_node(timer)?;
```

ARM64 processors have a built-in generic timer that provides high-resolution time measurement and periodic interrupts. The kernel needs this timer for scheduling, timekeeping, and delay loops. Without a timer node, the kernel cannot initialize the scheduler and will hang.

`compatible = "arm,armv8-timer"` tells the kernel to use the ARMv8 generic timer driver. This driver accesses the timer through system registers (`CNTPCT_EL0` for the counter, `CNTP_CTL_EL0` / `CNTV_CTL_EL0` for timer control), which our VMM may need to trap or expose directly depending on configuration.

`always-on` is a boolean property (no value - its presence indicates `true`). It tells the kernel the timer keeps running even when the CPU is in low-power states. For a VM, this is always true because the host keeps running.

A complete timer node would include an `interrupts` property listing the timer interrupt numbers for each timer type (secure physical, non-secure physical, virtual, hypervisor). We will add these when we integrate interrupt support. For now, the kernel will detect the timer hardware via system registers even without interrupt routing.

### The /psci Node

```rust
let psci = fdt.begin_node("psci")?;
fdt.property_string("compatible", "arm,psci-0.2")?;
fdt.property_string("method", "hvc")?;
fdt.end_node(psci)?;
```

PSCI (Power State Coordination Interface) is ARM's standard interface for power management operations: turning CPUs on/off, entering low-power states, system reset, system shutdown. The kernel uses PSCI to bring up secondary CPUs, handle reboot/poweroff commands, and manage CPU idle states.

`compatible = "arm,psci-0.2"` specifies PSCI version 0.2, which is the baseline version that supports CPU_ON, CPU_OFF, SYSTEM_RESET, and SYSTEM_OFF functions.

`method = "hvc"` tells the kernel to use the HVC instruction to invoke PSCI functions. The alternative is `"smc"` (Secure Monitor Call), which transitions to EL3 rather than EL2. Since we are a hypervisor running at EL2 (via Hypervisor.framework), HVC is the correct method. When the guest kernel executes `hvc #0` with PSCI function IDs in registers, our VMM's HVC handler can intercept these and perform the requested operations - most importantly SYSTEM_RESET and SYSTEM_OFF, which let the guest shut down cleanly.

The PSCI function IDs follow the ARM SMCCC (SMC Calling Convention): function 0x84000000 is CPU_ON, 0x84000008 is SYSTEM_OFF, 0x84000009 is SYSTEM_RESET. Our VMM will need to recognize these IDs in the HVC handler and respond appropriately.

## DTB Placement in Guest Memory

The DTB is a blob of bytes that needs to live in guest memory where the kernel can read it. There are a few constraints on where it goes:

1. It must not overlap with the kernel image itself.
2. It must not overlap with the initramfs (if present).
3. It must be within the addressable memory range.
4. The ARM64 boot protocol requires the DTB address to be 8-byte aligned.

A common convention is to place the DTB at a high address in the guest RAM, well away from the kernel load address. We will put it 2MB before the end of RAM:

```rust
let memory_size: u64 = 128 * 1024 * 1024; // 128MB
let dtb = create_fdt(
    memory_size,
    1,                          // 1 CPU
    "console=ttyAMA0 earlycon", // kernel command line
    None,                       // no initrd yet
    None,
)?;

// Place DTB near the end of RAM
let dtb_addr = 0x80000000 + memory_size - 0x200000; // 2MB before end
memory.write_at(dtb_addr, &dtb);

println!("DTB placed at {:#x} ({} bytes)", dtb_addr, dtb.len());
```

The ARM64 boot protocol specifies how the bootloader communicates the DTB location to the kernel. The registers at kernel entry must be:

```rust
vcpu.set_reg(HV_REG_X0, dtb_addr)?;  // x0 = DTB physical address
vcpu.set_reg(HV_REG_X1, 0)?;          // x1 = 0 (reserved)
vcpu.set_reg(HV_REG_X2, 0)?;          // x2 = 0 (reserved)
vcpu.set_reg(HV_REG_X3, 0)?;          // x3 = 0 (reserved)
```

X0 contains the physical address of the DTB. X1 through X3 must be zero. The kernel's entry code reads X0, validates the DTB magic number at that address, and begins parsing. If X0 does not point to a valid DTB (magic number `0xd00dfeed` in big-endian at the start), the kernel prints nothing and hangs.

This is a common source of silent boot failures. If you get no output at all from the kernel - not even "Booting Linux" - the first thing to check is whether X0 points to a valid DTB. The second thing to check is whether the DTB's `stdout-path` matches a valid UART node.

## Putting It Together

Here is the updated VMM initialization sequence that combines the ELF loader from the previous post with DTB generation:

```rust
fn main() -> anyhow::Result<()> {
    let kernel_path = std::env::args()
        .nth(1)
        .unwrap_or_else(|| "guest/target/aarch64-unknown-none/release/guest".to_string());

    let binary = std::fs::read(&kernel_path)?;
    let loaded = load_elf(&binary)?;

    let memory_size: u64 = 128 * 1024 * 1024; // 128MB
    let vm = Vm::new()?;
    let mut memory = GuestMemory::new(memory_size as usize)?;

    // Load binary segments
    for segment in &loaded.segments {
        memory.write_at(segment.guest_addr, &segment.data);
    }

    // Generate and place DTB
    let dtb = create_fdt(
        memory_size,
        1,
        "console=ttyAMA0 earlycon",
        None,
        None,
    )?;
    let dtb_addr = 0x80000000 + memory_size - 0x200000;
    memory.write_at(dtb_addr, &dtb);

    // Map guest memory
    memory.map_all(&vm)?;

    // Configure vCPU per ARM64 boot protocol
    let mut vcpu = Vcpu::new()?;
    vcpu.set_reg(HV_REG_PC, loaded.entry_point)?;
    vcpu.set_reg(HV_REG_CPSR, 0x3c4)?;
    vcpu.set_reg(HV_REG_X0, dtb_addr)?;
    vcpu.set_reg(HV_REG_X1, 0)?;
    vcpu.set_reg(HV_REG_X2, 0)?;
    vcpu.set_reg(HV_REG_X3, 0)?;

    println!("Entry: {:#x}, DTB: {:#x}", loaded.entry_point, dtb_addr);

    // Run loop...
    loop {
        let exit = vcpu.run()?;
        // ... exit handling as before ...
        # break; // placeholder
    }

    Ok(())
}
```

For our bare-metal Rust guest from the previous post, the DTB is technically unnecessary - the guest does not parse device trees, it just writes to a hardcoded UART address. But the infrastructure is now in place for Linux, which absolutely requires it.

## Testing the DTB with a Bare-Metal Guest

Let us write a guest that actually reads the DTB to verify it is being passed correctly. This guest will:

1. Read the DTB pointer from X0 (set by the VMM at boot).
2. Read the magic number at that address.
3. Verify it matches `0xd00dfeed` (big-endian).
4. Print a confirmation to the UART.

```rust
#![no_std]
#![no_main]

use core::panic::PanicInfo;
use core::ptr;

const UART_BASE: *mut u8 = 0x0900_0000 as *mut u8;

fn uart_putc(c: u8) {
    unsafe { ptr::write_volatile(UART_BASE, c); }
}

fn uart_puts(s: &str) {
    for b in s.bytes() {
        uart_putc(b);
    }
}

fn uart_put_hex(val: u32) {
    let hex_chars = b"0123456789abcdef";
    uart_puts("0x");
    for i in (0..8).rev() {
        let nibble = ((val >> (i * 4)) & 0xf) as usize;
        uart_putc(hex_chars[nibble]);
    }
}

#[no_mangle]
pub extern "C" fn _start() -> ! {
    // x0 contains the DTB address, passed by the VMM.
    // We need to capture it before any other code clobbers x0.
    let dtb_addr: u64;
    unsafe {
        core::arch::asm!(
            "mov {}, x0",
            out(reg) dtb_addr,
            // x0 is an input from the boot protocol
        );
    }

    // Set up stack
    unsafe {
        core::arch::asm!(
            "ldr x1, =_stack_top",
            "mov sp, x1",
        );
    }

    uart_puts("DTB address: ");
    uart_put_hex(dtb_addr as u32);
    uart_puts("\n");

    // Read the first 4 bytes at the DTB address (magic number)
    let magic = unsafe {
        ptr::read_volatile(dtb_addr as *const u32)
    };

    // FDT magic is 0xd00dfeed in big-endian.
    // ARM is little-endian by default, so we need to byte-swap.
    let magic_be = magic.to_be();

    uart_puts("DTB magic: ");
    uart_put_hex(magic_be);
    uart_puts("\n");

    if magic_be == 0xd00dfeed {
        uart_puts("DTB found!\n");
    } else {
        uart_puts("DTB NOT found - wrong magic\n");
    }

    // Halt
    unsafe { core::arch::asm!("hvc #0"); }
    loop {}
}

#[panic_handler]
fn panic(_info: &PanicInfo) -> ! {
    uart_puts("PANIC!\n");
    loop {}
}
```

There is a subtlety here. The ARM64 boot protocol says X0 contains the DTB address at the moment the kernel starts executing. But our `_start` function's prologue (the stack setup) might clobber X0 before we can read it. So we capture X0 into a local variable as the very first operation, using inline assembly that explicitly moves X0 into a compiler-managed register.

The DTB magic number check is straightforward. The FDT specification stores the magic as `0xd00dfeed` in big-endian byte order. ARM64 processors default to little-endian, so when we read the 4-byte value from memory, the bytes are swapped. We use `to_be()` to convert from the native byte order to big-endian for comparison. If the magic matches, we have a valid DTB at the address the VMM told us about.

Running this guest should produce:

```terminal
DTB address: 0x87e00000
DTB magic: 0xd00dfeed
DTB found!
Guest halted via HVC
```

The exact DTB address depends on your memory size. With 128MB of RAM starting at `0x80000000`, the DTB at `memory_end - 2MB` is `0x80000000 + 0x08000000 - 0x200000 = 0x87E00000`.

## Unit Tests

The DTB generator is pure computation - it takes parameters and produces bytes. This makes it easy to test without any VM infrastructure.

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_dtb_magic() {
        let dtb = create_fdt(
            64 * 1024 * 1024,
            1,
            "console=ttyAMA0",
            None,
            None,
        ).expect("DTB creation failed");

        // FDT magic number in big-endian
        assert_eq!(dtb[0], 0xd0);
        assert_eq!(dtb[1], 0x0d);
        assert_eq!(dtb[2], 0xfe);
        assert_eq!(dtb[3], 0xed);
    }

    #[test]
    fn test_dtb_nonzero_size() {
        let dtb = create_fdt(
            64 * 1024 * 1024,
            1,
            "console=ttyAMA0",
            None,
            None,
        ).expect("DTB creation failed");

        // A minimal DTB should be at least a few hundred bytes
        assert!(dtb.len() > 100);
        // But should not be unreasonably large
        assert!(dtb.len() < 16384);
    }

    #[test]
    fn test_dtb_varies_with_memory_size() {
        let dtb_small = create_fdt(
            64 * 1024 * 1024,
            1,
            "console=ttyAMA0",
            None,
            None,
        ).unwrap();

        let dtb_large = create_fdt(
            1024 * 1024 * 1024,
            1,
            "console=ttyAMA0",
            None,
            None,
        ).unwrap();

        // Different memory sizes should produce different DTBs
        // (the reg property in /memory has a different value)
        assert_ne!(dtb_small, dtb_large);
    }

    #[test]
    fn test_dtb_with_initrd() {
        let dtb = create_fdt(
            128 * 1024 * 1024,
            1,
            "console=ttyAMA0",
            Some(0x84000000),
            Some(0x85000000),
        ).expect("DTB creation failed");

        // DTB with initrd should be larger than without
        let dtb_no_initrd = create_fdt(
            128 * 1024 * 1024,
            1,
            "console=ttyAMA0",
            None,
            None,
        ).unwrap();

        assert!(dtb.len() > dtb_no_initrd.len());
    }

    #[test]
    fn test_dtb_multiple_cpus() {
        let dtb_1cpu = create_fdt(
            64 * 1024 * 1024,
            1,
            "console=ttyAMA0",
            None,
            None,
        ).unwrap();

        let dtb_4cpu = create_fdt(
            64 * 1024 * 1024,

            "console=ttyAMA0",
            None,
            None,
        ).unwrap();

        // More CPUs means more nodes, so larger DTB
        assert!(dtb_4cpu.len() > dtb_1cpu.len());
    }

    #[test]
    fn test_dtb_cmdline_embedded() {
```

These tests verify the structural properties of the generated DTB without needing to fully parse the binary format. The magic number test confirms we produce valid FDT output. The comparison tests verify that our parameters actually affect the output. The string search tests confirm that key values (command line, device compatible strings) appear in the binary blob.

For more rigorous validation, you could use the `dtc` command-line tool to decompile the generated DTB back into DTS format:

```bash
# Save the DTB to a file (e.g., from a test or by dumping from the VMM)
# Then decompile it:
dtc -I dtb -O dts test.dtb
```

This will show you the human-readable device tree source corresponding to your generated binary. It is an excellent debugging tool when things go wrong - you can see exactly what nodes and properties your DTB contains.

## Common Pitfalls

A few things that will trip you up when working with device trees:

**Mismatched addresses.** If the UART `reg` property says `0x09000000` but your VMM traps MMIO accesses at `0x10000000`, the kernel will try to access `0x09000000`, get a data abort that your VMM does not recognize as a UART access, and crash. Every address in the DTB must match the corresponding address in your MMIO emulation.

**Missing stdout-path.** The kernel uses `stdout-path` to find the early console device. Without it, you get zero output during boot. The kernel is running, it is parsing the DTB, it is initializing subsystems - but it has nowhere to print. Add `stdout-path` pointing to your UART and add `earlycon` to bootargs.

**Wrong compatible strings.** The kernel matches drivers to devices via `compatible`. If you typo `"arm,pl01"` instead of `"arm,pl011"`, no driver matches, and the UART does not initialize. No errors, no warnings - the driver just never probes.

**Byte order confusion.** The FDT format stores multi-byte values in big-endian. The `vm-fdt` crate handles this for you, but if you ever need to parse DTB data manually (as our test guest does with the magic number), remember to byte-swap on little-endian ARM.

**DTB alignment.** The kernel expects the DTB to be at an 8-byte-aligned address. If you place it at an odd offset, the kernel's DTB parser will read garbage and fail to validate the magic number. Our placement calculation (`memory_end - 0x200000`) is 2MB-aligned, so this is not an issue, but watch out if you change the formula.

## What We Built

We now have a function that generates a complete Flattened Device Tree describing our virtual machine. The DTB tells the kernel:

- How much RAM it has and where it starts
- How many CPUs are available
- Where the UART is and what kind of UART it is
- That there is an ARM generic timer
- That PSCI is available via HVC calls
- What the kernel command line is
- Where the initial ramdisk is (if present)
- Where to send console output

This is the contract between our VMM and the guest kernel. When we eventually boot Linux, the kernel will parse this DTB, initialize its drivers accordingly, and expect the hardware to behave as described. Our MMIO handlers had better match.
