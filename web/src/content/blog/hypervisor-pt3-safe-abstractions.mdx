## Why Safe Wrappers Matter

The raw FFI bindings from the last post work, but they are a minefield. Every function call is `unsafe`. Every return value is an `i32` that you must remember to check. Every resource - the VM, each vCPU, mapped memory - must be manually cleaned up in the right order.

Here is what goes wrong without safe wrappers:

Forget to call `hv_vm_destroy` and the process-global VM persists. Call `hv_vm_create` again and you get `HV_BUSY`. Now your program cannot create any VMs and the only fix is to restart the process.

Forget to call `hv_vcpu_destroy` and you leak a kernel resource. The framework tracks vCPUs per-VM, and there is a hard limit. Leak enough of them and `hv_vcpu_create` starts returning `HV_NO_RESOURCES`.

Forget to check the return value of `hv_vcpu_run` and you read stale data from the exit struct, handling a phantom exit that never happened.

Pass the wrong register constant to `hv_vcpu_set_reg` - say, a system register ID where it expects a general-purpose register ID - and you get `HV_BAD_ARGUMENT` at runtime, not a compiler error.

All of these are preventable. Rust's ownership system, RAII (Resource Acquisition Is Initialization), and type system can catch every one of them at compile time. That is what we build in this post.

## Error Handling

Before we wrap anything, we need a proper error type. Matching on raw `i32` return codes is not ergonomic and it is easy to forget a variant.

Create `src/error.rs`:

```rust
use crate::hv::*;
use thiserror::Error;

#[derive(Debug, Error)]
pub enum HvError {
    #[error("hypervisor error (0x{0:08x})")]
    Generic(i32),
    #[error("hypervisor busy")]
    Busy,
    #[error("bad argument")]
    BadArgument,
    #[error("no resources")]
    NoResources,
    #[error("no device")]
    NoDevice,
    #[error("permission denied")]
    Denied,
    #[error("unsupported")]
    Unsupported,
}

pub fn check(ret: hv_return_t) -> Result<(), HvError> {
    match ret {
        HV_SUCCESS => Ok(()),
        HV_BUSY => Err(HvError::Busy),
        HV_BAD_ARGUMENT => Err(HvError::BadArgument),
        HV_NO_RESOURCES => Err(HvError::NoResources),
        HV_NO_DEVICE => Err(HvError::NoDevice),
        HV_DENIED => Err(HvError::Denied),
        HV_UNSUPPORTED => Err(HvError::Unsupported),
        other => Err(HvError::Generic(other)),
    }
}
```

The `check` function takes a raw return code and converts it to a `Result`. Every wrapper function will call this, turning the C-style "check the return code" pattern into Rust's idiomatic `?` operator. If we ever forget to check a return value, the compiler will warn us about an unused `Result`.

The `thiserror` derive gives us nice error messages for free. When an error is printed, you get `"permission denied"` instead of `"0xfae94007"`.

## The Vm Struct

The VM is the first resource to wrap. It has the simplest lifecycle: create, use, destroy. The key insight is that `hv_vm_create` and `hv_vm_destroy` manage a process-global resource - there is no handle, just a global flag. This maps naturally to a zero-sized struct with a `Drop` impl.

Create `src/vm.rs`:

```rust
use crate::error::{check, HvError};
use crate::hv;

pub struct Vm;

impl Vm {
    pub fn new() -> Result<Self, HvError> {
        check(unsafe { hv::hv_vm_create(std::ptr::null()) })?;
        Ok(Vm)
    }
}

impl Drop for Vm {
    fn drop(&mut self) {
        let ret = unsafe { hv::hv_vm_destroy() };
        if ret != hv::HV_SUCCESS {
            log::error!(
                "hv_vm_destroy failed: 0x{:08x}",
                ret as u32
            );
        }
    }
}
```

That is the entire implementation. `Vm::new()` calls `hv_vm_create` and returns an error if it fails. When the `Vm` value is dropped - whether by going out of scope, being explicitly dropped, or during stack unwinding from a panic - `hv_vm_destroy` is called automatically.

This means you cannot leak the VM. If you do `let _vm = Vm::new()?;` and then your function returns early due to an error, the `Vm` is dropped and `hv_vm_destroy` runs. You never end up in the "VM exists but nobody called destroy" state.

The struct is zero-sized (`Vm` has no fields), so it takes no memory. It exists purely as a compile-time token that proves the VM has been created and gives the type system a place to hang the `Drop` impl.

One subtlety: we cannot implement `Clone` or `Copy` for `Vm`. If we could copy the `Vm` value, dropping any copy would destroy the VM while other copies still think it exists. Rust's move semantics prevent this by default - there is exactly one `Vm` value, and it can only be moved, not copied.

## Guest Memory

Guest memory is more complex. We need to allocate a page-aligned buffer in host memory, map it into the guest's IPA space, and clean up both when we are done.

Create `src/memory.rs`:

```rust
use crate::error::{check, HvError};
use crate::hv;
use std::ptr;

const PAGE_SIZE: usize = 4096;

pub struct GuestMemory {
    ptr: *mut u8,
    size: usize,
    ipa: Option<hv::hv_ipa_t>,
}

impl GuestMemory {
    pub fn new(size: usize) -> Result<Self, HvError> {
        // Round up to page boundary
        let size = (size + PAGE_SIZE - 1) & !(PAGE_SIZE - 1);

        let ptr = unsafe {
            libc::mmap(
                ptr::null_mut(),
                size,
                libc::PROT_READ | libc::PROT_WRITE,
                libc::MAP_PRIVATE | libc::MAP_ANONYMOUS,
                -1,
                0,
            )
        };

        if ptr == libc::MAP_FAILED {
            return Err(HvError::NoResources);
        }

        Ok(GuestMemory {
            ptr: ptr as *mut u8,
            size,
            ipa: None,
        })
    }

    pub fn map(
        &mut self,
        ipa: hv::hv_ipa_t,
        flags: hv::hv_memory_flags_t,
    ) -> Result<(), HvError> {
        // Unmap previous mapping if any
        if let Some(old_ipa) = self.ipa.take() {
            check(unsafe {
                hv::hv_vm_unmap(old_ipa, self.size)
            })?;
        }

        check(unsafe {
            hv::hv_vm_map(
                self.ptr as *const u8,
                ipa,
                self.size,
                flags,
            )
        })?;

        self.ipa = Some(ipa);
        Ok(())
    }

    pub fn unmap(&mut self) -> Result<(), HvError> {
        if let Some(ipa) = self.ipa.take() {
            check(unsafe {
                hv::hv_vm_unmap(ipa, self.size)
            })?;
        }
        Ok(())
    }

    pub fn write_at(
        &mut self,
        offset: usize,
        data: &[u8],
    ) -> Result<(), HvError> {
        if offset + data.len() > self.size {
            return Err(HvError::BadArgument);
        }
        unsafe {
            ptr::copy_nonoverlapping(
                data.as_ptr(),
                self.ptr.add(offset),
                data.len(),
            );
        }
        Ok(())
    }

    pub fn read_at(
        &self,
        offset: usize,
        len: usize,
    ) -> Result<Vec<u8>, HvError> {
        if offset + len > self.size {
            return Err(HvError::BadArgument);
        }
        let mut buf = vec![0u8; len];
        unsafe {
            ptr::copy_nonoverlapping(
                self.ptr.add(offset),
                buf.as_mut_ptr(),
                len,
            );
        }
        Ok(buf)
    }

    pub fn size(&self) -> usize {
        self.size
    }

    pub fn as_ptr(&self) -> *const u8 {
        self.ptr as *const u8
    }
}

impl Drop for GuestMemory {
    fn drop(&mut self) {
        // Unmap from guest first
        if let Some(ipa) = self.ipa.take() {
            let ret = unsafe {
                hv::hv_vm_unmap(ipa, self.size)
            };
            if ret != hv::HV_SUCCESS {
                log::error!(
                    "hv_vm_unmap failed: 0x{:08x}",
                    ret as u32
                );
            }
        }

        // Free the host allocation
        let ret = unsafe {
            libc::munmap(self.ptr as *mut libc::c_void, self.size)
        };
        if ret != 0 {
            log::error!("munmap failed: {}", ret);
        }
    }
}
```

Several design decisions here:

**Page-aligned allocation via mmap.** We use `mmap` with `MAP_ANONYMOUS` to get a page-aligned allocation. The framework requires page-aligned host memory for `hv_vm_map`. We could use `aligned_alloc` from libc, but `mmap` is more natural for large allocations and gives us guaranteed alignment.

**Size rounding.** The constructor rounds the requested size up to the nearest page boundary. The framework requires the size parameter of `hv_vm_map` to be a multiple of 4096. Rather than making the caller worry about this, we handle it internally.

**Drop order matters.** The `Drop` impl first unmaps the memory from the guest (via `hv_vm_unmap`), then frees the host allocation (via `munmap`). If we did it in the other order, the framework would be holding a stale pointer to freed memory. The `Option<hv_ipa_t>` tracks whether the memory is currently mapped - if we never called `map`, or if we already called `unmap`, the Drop impl skips the `hv_vm_unmap` call.

**Bounds checking on read/write.** `write_at` and `read_at` check that the offset and length do not exceed the allocation size. Without this, we would have a buffer overflow hiding behind `unsafe`.

You need to add `libc` to your `Cargo.toml`:

```toml
[dependencies]
anyhow = "1"
thiserror = "2"
log = "0.4"
env_logger = "0.11"
libc = "0.2"
```

## The Vcpu

The vCPU wrapper is the most involved because it owns two resources: the vCPU handle and the exit struct pointer.

Create `src/vcpu.rs`:

```rust
use crate::error::{check, HvError};
use crate::hv;

pub struct Vcpu {
    handle: hv::hv_vcpu_t,
    exit: *const hv::hv_vcpu_exit_t,
}

#[derive(Debug)]
pub enum ExitInfo {
    Exception {
        syndrome: u64,
        virtual_address: u64,
        physical_address: u64,
    },
    VTimerActivated,
    Canceled,
    Unknown,
}

impl Vcpu {
    pub fn new() -> Result<Self, HvError> {
        let mut handle: hv::hv_vcpu_t = 0;
        let mut exit: *const hv::hv_vcpu_exit_t =
            std::ptr::null();

        check(unsafe {
            hv::hv_vcpu_create(
                &mut handle,
                &mut exit,
                std::ptr::null(),
            )
        })?;

        Ok(Vcpu { handle, exit })
    }

    pub fn get_reg(&self, reg: u32) -> Result<u64, HvError> {
        let mut value: u64 = 0;
        check(unsafe {
            hv::hv_vcpu_get_reg(
                self.handle,
                reg,
                &mut value,
            )
        })?;
        Ok(value)
    }

    pub fn set_reg(
        &self,
        reg: u32,
        value: u64,
    ) -> Result<(), HvError> {
        check(unsafe {
            hv::hv_vcpu_set_reg(self.handle, reg, value)
        })
    }

    pub fn get_sys_reg(
        &self,
        reg: u16,
    ) -> Result<u64, HvError> {
        let mut value: u64 = 0;
        check(unsafe {
            hv::hv_vcpu_get_sys_reg(
                self.handle,
                reg,
                &mut value,
            )
        })?;
        Ok(value)
    }

    pub fn set_sys_reg(
        &self,
        reg: u16,
        value: u64,
    ) -> Result<(), HvError> {
        check(unsafe {
            hv::hv_vcpu_set_sys_reg(
                self.handle,
                reg,
                value,
            )
        })
    }

    pub fn set_vtimer_mask(
        &self,
        masked: bool,
    ) -> Result<(), HvError> {
        check(unsafe {
            hv::hv_vcpu_set_vtimer_mask(
                self.handle,
                masked,
            )
        })
    }

    pub fn set_pending_interrupt(
        &self,
        interrupt_type: u32,
        pending: bool,
    ) -> Result<(), HvError> {
        check(unsafe {
            hv::hv_vcpu_set_pending_interrupt(
                self.handle,
                interrupt_type,
                pending,
            )
        })
    }

    pub fn run(&self) -> Result<ExitInfo, HvError> {
        check(unsafe { hv::hv_vcpu_run(self.handle) })?;

        let exit = unsafe { &*self.exit };
        let info = match exit.reason {
            hv::HV_EXIT_REASON_EXCEPTION => {
                ExitInfo::Exception {
                    syndrome: exit.exception.syndrome,
                    virtual_address: exit
                        .exception
                        .virtual_address,
                    physical_address: exit
                        .exception
                        .physical_address,
                }
            }
            hv::HV_EXIT_REASON_VTIMER_ACTIVATED => {
                ExitInfo::VTimerActivated
            }
            hv::HV_EXIT_REASON_CANCELED => ExitInfo::Canceled,
            _ => ExitInfo::Unknown,
        };

        Ok(info)
    }
}

impl Drop for Vcpu {
    fn drop(&mut self) {
        let ret =
            unsafe { hv::hv_vcpu_destroy(self.handle) };
        if ret != hv::HV_SUCCESS {
            log::error!(
                "hv_vcpu_destroy failed: 0x{:08x}",
                ret as u32
            );
        }
    }
}
```

The design mirrors the C API but makes it safe:

**Resource cleanup is automatic.** Drop calls `hv_vcpu_destroy`. You cannot leak a vCPU.

**Register access returns Results.** `get_reg` and `set_reg` return `Result<u64, HvError>` and `Result<(), HvError>` respectively. If you pass an invalid register constant, you get a `BadArgument` error instead of silent corruption.

**The run method returns typed exit info.** Instead of giving you a raw pointer to the exit struct (which could be null, could be stale, could be misinterpreted), `run()` reads the exit struct and returns an `ExitInfo` enum. Pattern matching on this enum covers all cases - the compiler warns you if you miss one.

**The exit pointer is private.** The raw `*const hv_vcpu_exit_t` is not exposed to callers. You can only access exit information through the `run()` method, which reads it safely and returns a copy. This prevents use-after-free bugs where someone holds a reference to the exit struct across multiple `run()` calls.

One thing we do not do here is make `Vcpu` `Send` or `Sync`. A vCPU is inherently tied to a single OS thread in Hypervisor.framework - you cannot call `hv_vcpu_run` from a different thread than the one that created the vCPU. Rust's default behavior is correct here: `Vcpu` contains a raw pointer (`*const hv_vcpu_exit_t`), which is neither `Send` nor `Sync`, so `Vcpu` is not either. If you try to send a `Vcpu` to another thread, the compiler stops you.

## Proof of Life: First Running VM

Now we put it all together. Our goal: create a VM, load two ARM64 instructions into guest memory, run the vCPU, and read the result back. The two instructions are:

1. `mov x0, #42` - load the immediate value 42 into register X0
2. `hvc #0` - hypervisor call, which triggers a VM exit

When the vCPU runs, it will execute `mov x0, #42` (setting X0 to 42), then execute `hvc #0` (causing a VM exit with exception class EC=0x16). We read X0 after the exit and verify it contains 42.

First, update `src/main.rs` to declare all our modules:

```rust
mod error;
mod hv;
mod memory;
mod vcpu;
mod vm;

use crate::hv::*;
use crate::memory::GuestMemory;
use crate::vcpu::Vcpu;
use crate::vm::Vm;

fn main() -> anyhow::Result<()> {
    env_logger::init();

    // Step 1: Create the VM
    println!("Creating VM...");
    let _vm = Vm::new()?;
    println!("VM created.");

    // Step 2: Allocate and map guest memory
    let guest_base: u64 = 0x8000_0000;
    let mut memory = GuestMemory::new(4096)?;

    // Step 3: Write two instructions
    //
    //   mov x0, #42     -> 0xD2800540
    //   hvc #0           -> 0xD4000002
    //
    // ARM64 instructions are 4 bytes, little-endian.
    let mov_x0_42: u32 = 0xD280_0540;
    let hvc_0: u32 = 0xD400_0002;

    memory.write_at(0, &mov_x0_42.to_le_bytes())?;
    memory.write_at(4, &hvc_0.to_le_bytes())?;

    // Step 4: Map memory into the guest
    memory.map(
        guest_base,
        HV_MEMORY_READ | HV_MEMORY_WRITE | HV_MEMORY_EXEC,
    )?;

    println!(
        "Mapped 4KB at IPA 0x{:08x}",
        guest_base
    );

    // Step 5: Create the vCPU
    let vcpu = Vcpu::new()?;

    // Step 6: Set initial register state
    //
    //   PC = guest_base (start of our code)
    //   CPSR = 0x3c4 (EL1h mode, DAIF masked)
    //
    // CPSR 0x3c4 means:
    //   bits [3:0] = 0x4 -> EL1h (EL1, using SP_EL1)
    //   bit 6 (F) = 1   -> FIQ masked
    //   bit 7 (I) = 1   -> IRQ masked
    //   bit 8 (A) = 1   -> SError masked
    //   bit 9 (D) = 1   -> Debug masked
    vcpu.set_reg(HV_REG_PC, guest_base)?;
    vcpu.set_reg(HV_REG_CPSR, 0x3c4)?;

    println!(
        "vCPU created. PC = 0x{:08x}, CPSR = 0x3c4",
        guest_base
    );

    // Step 7: Run the vCPU
    println!("Running...");
    let exit = vcpu.run()?;

    // Step 8: Examine the result
    match &exit {
        vcpu::ExitInfo::Exception {
            syndrome, ..
        } => {
            let ec = syndrome >> 26;
            println!(
                "VM exit: exception, EC = 0x{:02x}",
                ec
            );
            if ec == 0x16 {
                println!(
                    "  -> HVC instruction (expected)"
                );
            }
        }
        other => {
            println!("VM exit: {:?}", other);
        }
    }

    let x0 = vcpu.get_reg(HV_REG_X0)?;
    println!("x0 = {}", x0);

    if x0 == 42 {
        println!(
            "Success! The vCPU executed our instructions."
        );
    } else {
        println!(
            "Unexpected: x0 = {} (expected 42)",
            x0
        );
    }

    // Resources are cleaned up automatically:
    //   vcpu drops -> hv_vcpu_destroy
    //   memory drops -> hv_vm_unmap + munmap
    //   _vm drops -> hv_vm_destroy

    Ok(())
}
```

Let us walk through what happens when you run this:

**Step 1-2:** We create the VM (configuring EL2 state in the kernel) and allocate 4KB of page-aligned host memory.

**Step 3:** We write two ARM64 instructions at the beginning of the buffer. `0xD2800540` is the encoding for `MOV X0, #42`. Breaking it down: the top bits `110100101` identify it as a MOVZ (move wide with zero) instruction, the `00` means shift by 0, the immediate field contains 42, and the register field `00000` is X0. The second instruction `0xD4000002` is `HVC #0`: the encoding for a hypervisor call with immediate value 0.

**Step 4:** We map the host memory into the guest's IPA space at `0x80000000`. This populates the stage-2 page tables. The guest can now read, write, and execute memory at IPA `0x80000000` through `0x80000FFF`.

**Step 5-6:** We create a vCPU and set its initial state. PC points at our first instruction. CPSR is `0x3c4`, which means EL1h with all asynchronous exceptions masked. This is the state a guest kernel expects to start in - EL1 privilege level, using the SP_EL1 stack pointer, with interrupts disabled so it can set up its exception vectors before enabling them.

**Step 7:** We call `vcpu.run()`. This is where the magic happens. The framework saves our thread's context, loads the guest context (PC, CPSR, and all the zero-initialized registers), configures the stage-2 page tables, and performs an ERET into EL1.

The CPU is now running guest code. It fetches the instruction at IPA `0x80000000`, which the stage-2 tables translate to our mmap'd buffer. It executes `MOV X0, #42` - register X0 now contains 42. It advances PC to `0x80000004` and fetches the next instruction: `HVC #0`. This instruction traps to EL2, causing a VM exit.

The framework catches the trap, saves the guest state, restores our thread's context, and returns from `hv_vcpu_run()`. Our `run()` wrapper reads the exit struct and returns `ExitInfo::Exception` with the syndrome value.

**Step 8:** We examine the syndrome. The top 6 bits (EC field) are `0x16`, which is the exception class for HVC from AArch64. This confirms the exit was caused by our HVC instruction. We read X0 and find 42 - proof that the MOV instruction executed before the HVC.

Build and run:

```bash
./build.sh && ./target/release/apple-hypervisor
```

Expected output:

```terminal

Mapped 4KB at IPA 0x80000000
vCPU created. PC = 0x80000000, CPSR = 0x3c4
Running...
VM exit: exception, EC = 0x16
  -> HVC instruction (expected)
x0 = 42
Success! The vCPU executed our instructions.
```

That is it. Two instructions, one VM exit, and we have proven that our safe abstractions work end-to-end. The guest code ran on real hardware at full speed - no emulation, no interpretation. The only overhead was the VM entry and exit transitions.

## Testing

We can now write proper tests. Unit tests verify our abstractions behave correctly in isolation. Integration tests verify the full stack works against the real framework.

### Unit Tests

Memory read/write round-tripping does not require the hypervisor, so it works with standard `cargo test`:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_memory_write_read_roundtrip() {
        let mut mem = GuestMemory::new(4096).unwrap();
        let data = b"hello, guest memory";
        mem.write_at(0, data).unwrap();
        let readback = mem.read_at(0, data.len()).unwrap();
        assert_eq!(&readback, data);
    }

    #[test]
    fn test_memory_write_at_offset() {
        let mut mem = GuestMemory::new(4096).unwrap();
        let data = b"offset test";
        mem.write_at(100, data).unwrap();
        let readback =
            mem.read_at(100, data.len()).unwrap();
        assert_eq!(&readback, data);
    }

    #[test]
    fn test_memory_bounds_check() {
        let mut mem = GuestMemory::new(4096).unwrap();
        let result = mem.write_at(4090, &[0u8; 10]);
        assert!(result.is_err());
    }

    #[test]
    fn test_memory_size_rounds_up() {
        let mem = GuestMemory::new(1).unwrap();
        assert_eq!(mem.size(), 4096);
    }

    #[test]
    fn test_memory_size_exact_page() {
        let mem = GuestMemory::new(4096).unwrap();
        assert_eq!(mem.size(), 4096);
    }

    #[test]
    fn test_memory_size_rounds_to_next_page() {
        let mem = GuestMemory::new(4097).unwrap();
        assert_eq!(mem.size(), 8192);
    }
}
```

These tests verify that our memory abstraction handles allocation, rounding, bounds checking, and read/write correctly. They run without the hypervisor entitlement because they do not touch any `hv_*` functions.

### Integration Tests

The full VM test requires codesigning. Write it as a separate example that you build and codesign manually:

```rust
// examples/test_vm.rs

use apple_hypervisor::hv::*;
use apple_hypervisor::memory::GuestMemory;
use apple_hypervisor::vcpu::{ExitInfo, Vcpu};
use apple_hypervisor::vm::Vm;

fn main() {
    let _vm = Vm::new().expect("failed to create VM");

    let guest_base: u64 = 0x8000_0000;
    let mut memory =
        GuestMemory::new(4096).expect("alloc failed");

    // mov x0, #42
    memory
        .write_at(0, &0xD280_0540u32.to_le_bytes())
        .unwrap();
    // hvc #0
    memory
        .write_at(4, &0xD400_0002u32.to_le_bytes())
        .unwrap();

    memory
        .map(
            guest_base,
            HV_MEMORY_READ
                | HV_MEMORY_WRITE
                | HV_MEMORY_EXEC,
        )
        .expect("map failed");

    let vcpu = Vcpu::new().expect("vcpu create failed");
    vcpu.set_reg(HV_REG_PC, guest_base).unwrap();
    vcpu.set_reg(HV_REG_CPSR, 0x3c4).unwrap();

    let exit = vcpu.run().expect("vcpu run failed");

    match exit {
        ExitInfo::Exception { syndrome, .. } => {
            let ec = syndrome >> 26;
            assert_eq!(
                ec, 0x16,
                "expected HVC exception class"
            );
        }
        other => {
            panic!("unexpected exit: {:?}", other);
        }
    }

    let x0 =
        vcpu.get_reg(HV_REG_X0).expect("get_reg failed");
    assert_eq!(x0, 42, "x0 should be 42");

    println!("All assertions passed.");
}
```

```bash
./target/debug/examples/test_vm
```

This is admittedly more friction than `cargo test`, but it is inherent to the platform. Apple requires entitlements for hypervisor access, and entitlements require codesigning. There is no way around this. You could write a Cargo xtask that automates the codesign step, or add a post-build hook, but the fundamental requirement stays.

For day-to-day development, you will spend most of your time writing unit tests that do not touch the hypervisor (testing instruction decoding, device emulation logic, memory layout calculations) and only run the codesigned integration tests when you change the core VM/vCPU code.

## What We Have Now

Let us step back and look at what we have built across these first three posts.

In `src/hv.rs`: raw FFI types, constants, and function declarations. This is the foundation - it matches Apple's C headers byte-for-byte.

In `src/error.rs`: a proper error type that converts raw return codes into descriptive Rust errors.

In `src/vm.rs`: a zero-sized RAII wrapper that creates the VM on construction and destroys it on drop.

In `src/memory.rs`: a guest memory manager that handles page-aligned allocation, mapping into the guest IPA space, bounds-checked reads and writes, and automatic cleanup.

In `src/vcpu.rs`: a vCPU wrapper with typed register access, safe run semantics, and automatic resource cleanup.

In `src/main.rs`: a proof-of-life program that creates a VM, loads instructions, runs them, and reads the result.

The total code is roughly 400 lines across all files. That is the entire foundation of a hypervisor.

## What is Next

We can create VMs and run individual instructions, but we do not have a loop. The HVC instruction caused a VM exit and we just read X0 and stopped. A real hypervisor does not stop after one exit - it handles the exit and re-enters the guest.

In the next post, we build a proper VMM run loop. We will decode the ARM syndrome register to distinguish between HVC calls, data aborts (MMIO), system register traps, WFI, and other exit types. Each one gets its own handler. The loop runs until the guest does something we cannot handle or until we deliberately shut it down.

That run loop is the heartbeat of the entire hypervisor. Once it is working, every subsequent post adds a new capability to it: UART for console output, GIC for interrupts, timer for scheduling, and finally kernel loading. But the loop itself stays the same.
